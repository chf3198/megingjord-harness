# Adversarial Spoofing Detection + Crypto Promotion Strategy (#1718)

When does crypto-signing become mandatory? What fingerprinting techniques
exist for non-crypto operators? Phase 1 research per Epic #1716.

## Existing harness primitives (state at session)

`inventory/team-model-signatures.json` already contains a `cryptoKeys` array with **publicKey** entries for the copilot team's 4 roles (manager/collaborator/admin/consultant), each with `keyId` like `copilot-manager-v1` and ed25519 public keys. **Crypto infrastructure exists today** — it just isn't yet default-on.

The optional fields are already specified in `instructions/team-model-signing.instructions.md`:

- `Crypto-Algorithm: ed25519`
- `Crypto-Key-Id: <team-role-key-id>`
- `Crypto-Signature: <base64-signature>`

This means the promotion from optional to default is a flip-the-flag exercise + key bootstrap for teams beyond Copilot, NOT new infrastructure.

| Distribution mechanism | ⚠️ Existing: keys live in the repo's `inventory/` JSON. Public keys distributed via `git pull`. Acceptable for the harness's trust model. |

**Minimum infrastructure for the promotion**:

1. Per-team `ed25519` keypair (1 file per role per team).
2. Public keys committed to `inventory/team-model-signatures.json`.
3. Private keys stored at `~/.megingjord/keys/<team>-<role>.key` (mode 0600).
4. `npm run signing:bootstrap` generates + registers keys.
5. `npm run signing:rotate` rotates and appends old keyId to revocation list.

## Question 2: Fingerprinting techniques for non-crypto operators

For operators who opt out of crypto-signing via `MEGINGJORD_CRYPTO_SIGNING_DISABLED=1`, three behavioral-fingerprinting techniques can detect spoofed `Team&Model` declarations:

### Technique A — Token-distribution perplexity profiling

Each model family has a characteristic token-distribution signature. A baton artifact claiming `claude-code:opus-4-7@anthropic` should exhibit Claude-Opus-style markers:
- Sentence-length distribution (Claude tends toward medium-length structured paragraphs).
- Vocabulary-richness signature (Claude's lexical density is measurable).
- Header/list usage (Claude favors structured markdown).

**Implementation**: train a lightweight per-family classifier on existing signed artifacts; flag artifacts whose signature deviates >2σ from declared family.

**Limitation**: prompting / temperature / system prompt can shift output style; classifier needs to tolerate prompt variation. False-positive rate likely 5-15% based on similar 2024-2026 detection literature.

### Technique B — Response-time profiling

Each provider has a characteristic latency distribution. A `Crypto-Signature` absent artifact claiming `claude-code:opus-4-7@anthropic` should have been generated in a time window consistent with Anthropic's Opus API (typically 2-15s p95). An artifact generated by `ollama:gemma3:1b` would complete in <2s.

**Implementation**: record artifact-emission-timestamp; cross-check against expected provider latency band.

**Limitation**: cannot prove which model — only that the latency was inconsistent with the declaration. Useful as a heuristic, not a proof.

### Technique C — Output-pattern correlation

Same-model artifacts produced by the same operator over time accumulate a personal style signature. A signed artifact deviating dramatically from the operator's prior signed artifacts (with same `Team&Model`) suggests spoofing OR a legitimate model change. Combined with `Crypto-Signature` presence/absence, this becomes a stronger signal.

**Implementation**: per-operator-per-team-model embedding average; flag artifacts >3σ from the centroid.

**Limitation**: cold-start problem for new operators; multi-tenant accounts.

## Question 3: Trust model — when does crypto become mandatory?

Three tiers:

### Tier 1 — Crypto-required (high-trust harness operations)

- Cross-team PR merges (claude-code merging copilot or codex work)
- Epic close-outs
- Promotion-to-required gate flips
- Tier-3 anneal escalations

These are operations where spoofed signatures could cause cross-team trust damage. **Crypto-signing should be REQUIRED** for these from Phase 3.3 onward.

### Tier 2 — Crypto-default with fingerprinting fallback

- Standard baton artifacts on routine tickets
- Soak-window advisory comments
- Cross-team consult pickups

Default to crypto-signing; if operator has opted out, run fingerprinting (Technique A + Technique C) as an advisory check. False positives surface as PR comments, not gate blocks.

### Tier 3 — Optional crypto (lowest-trust)

- Single-team intra-baton on small tickets
- `lane:trivial` / docs-only work
- Local-only operators (no cross-team interaction)

Crypto-signing is optional; alias-only artifacts acceptable; no fingerprinting enforced. Operator's choice.

## Question 4: Comparison to external literature

| Approach | Strength | Limitation | Fit for the harness |
|---|---|---|---|
| **Trusted Envelope Protocol (TEP)** — cryptographic provenance for LLM context packets | Strong: cryptographic proof of origin | Heavy-weight; requires policy enforcement point | Aligned with the existing `Crypto-*` field design; partial adoption already in place |
| **MJW Media cryptographic LLM agent security layer** | Code-signing principles applied to token sequences | Conceptual; no reference implementation | The harness's existing implementation is more concrete |
| **CMU SEI chain-of-custody controls (data poisoning)** | Immutable audit trail; chain-of-custody pattern | Targets data poisoning, not artifact attribution | Useful as a model for the revocation list design |
| **Blockchain-based provenance** | Decentralized, immutable | Heavy; cost; latency | OVERKILL for the harness's single-repo trust model |
| **Behavioral fingerprinting (Technique A above)** | Lightweight; no key infrastructure required | False-positive rate; can be defeated by careful prompting | Useful as Tier 2 fallback only |

The harness's existing approach (ed25519 keys in `inventory/team-model-signatures.json` + optional `Crypto-Signature` field) is well-aligned with TEP / code-signing precedent and uses lighter-weight infrastructure than blockchain-based provenance. **No fundamental redesign needed; only promotion to default-on.**
