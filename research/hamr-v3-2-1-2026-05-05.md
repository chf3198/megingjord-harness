## 2. §R9 (NEW) · Cross-level resource-failure recovery

**Source evidence:** Wave 1 execution surfaced four operational
failure modes that v3.2 §4 (failover/redundancy map) did not cover.
Each was validated empirically; each has a corresponding pattern
that is required architectural input for Wave 2 onward.

### R9.1 · Worktree-isolation as crash-survivable execution surface

**Pattern:** all sub-agent dispatches that produce file artifacts
SHOULD run inside an isolated git worktree under
`.claude/worktrees/agent-<task-id>/`. The worktree's branch is
discoverable via `git worktree list` even after a host crash;
in-flight artifacts persist as untracked files in the worktree.

**Validation evidence:** Wave 1 #895 (judge-quorum.js) was
implemented inside an isolated worktree by a sub-agent. VS Code
crashed mid-batch. Recovery: `git worktree list` revealed the
locked branch + uncommitted files; resumption succeeded after
ESM-to-CJS conversion + readability fix. Zero work lost.

**Required architectural use in Wave 2:**

- HAMR core CF Worker (Wave 2 child 1) deployment scripts MUST
  run inside an isolated worktree when triggered from a sub-agent
  context, even if the operator launched it directly. Tear-down
  failure mid-deploy must leave the worktree in a recoverable
  state.

### R9.2 · Cwd-vs-branch pre-flight for `gh pr create`

**Pattern:** before invoking `gh pr create`, callers MUST verify
the current working directory's git branch matches the intended
PR head ref. The Bash tool's working directory persists between
calls but `gh pr create` resolves the head from local git context
— a stale cwd causes silent mis-targeting.

**Validation evidence:** Wave 1 #895 PR #899 was created with
head `feat/894-baton-signing` because `gh pr create` ran from
`/home/curtisfranks/devenv-ops` (where feat/894 was still checked
out) instead of the worktree. Closed + recreated as PR #900 with
correct head `feat/895-judge-quorum`. The hazard surfaces in any
multi-branch baton sequence.

**Required architectural use in Wave 2:**

- All `gh pr create` invocations in Wave 2's deploy + release
  scripts MUST prepend `pwd && git branch --show-current` checks,
  or use `--head <ref>` explicitly to pin the head ref against
  the cwd state.

### R9.3 · Sequential dispatch with backoff + family-fallback

**Pattern:** any free-fleet judge / inference call MUST run in a
sequential loop with ≥3 s spacing and 3-attempt exponential
backoff on `429` / `queue_exceeded`. Family-fallback (provider →
provider in different model family) MUST be available for any
gate-critical path.

**Validation evidence:** Wave 1 #893 (S5 Stage-2 quiz) showed:

- Cerebras hit `queue_exceeded` for 14/20 candidate-generation
  calls. Gemini family-fallback covered 14/14 seamlessly.
- Groq rate-limited 0/20 candidate generations after the third
  call (no spacing); 17/20 grader calls returned with 3 s
  spacing.
- Parallel `Promise.all` dispatches without spacing produced 0%
  candidate-generation success on the second-quickest tier.

**Required architectural use in Wave 2:**

- HAMR core CF Worker (Wave 2 child 1) `/healthz` endpoint MUST
  return within 5 s under all conditions; if downstream
  dependencies are rate-limited, return `{tier: "degraded",
  reason: "rate-limited"}` rather than blocking.
- Substrate-health probe (Wave 2 child 2) MUST tolerate
  per-provider rate-limits; loss of one provider does not block
  the probe.

### R9.4 · Idempotent infrastructure tear-down

**Pattern:** all infrastructure deployments (Worker, KV, R2,
domain) MUST be torn down via idempotent operations whose success
is verified by a deterministic post-condition (e.g., HTTP 404
after `wrangler delete`). A tear-down that "appears to succeed"
without verified post-condition is a leaked resource.

**Validation evidence:** Wave 1 #891 (S3 live deploy) deployed
`hamr-spike` Worker + KV namespace, ran 60-sample measurement,
tore down via `wrangler delete --force` + `wrangler kv namespace
delete`, and verified post-condition by `curl
hamr-spike.chf3198.workers.dev/healthz → HTTP 404`. Net leaked
resources: zero.

**Required architectural use in Wave 2:**

- HAMR core CF Worker (Wave 2 child 1) deployment script MUST
  emit a paired tear-down script with HTTP-404 verification.
- SLSA pipeline (Wave 2 child 6) MUST attach tear-down evidence
  to release artifacts when an attestation fails mid-deploy.

## 3. §R6 update · Replace binary gate with 3-stage gate

**Source evidence:** Wave 1 #893 S5 Stage-2 quiz measured
free-fleet 2-of-N quorum performance against a 60-Q reasoning quiz
(20-Q balanced subset run due to rate limits). Direct mean 0.55,
counter-factual mean 0.50, boundary 0%. v3.2 §R6's ≥97% Stage-2
threshold is **not achievable** with the current free-fleet quorum.

### Replaces v3.2 §R6 thresholds

| Stage | Trigger | Threshold | Judges |
|---|---|---|---|
| Stage-1 | Every bundle build | ≥99% keyword presence | Deterministic top-k extractive (no LLM) |
| **Stage-2a** (NEW) | Weekly | **≥80% on direct + counter-factual** (boundary excluded) | Free-fleet 2-of-N quorum across model families |
| **Stage-2b** (NEW) | Monthly OR rule-change PR | **≥95% including boundary cases** | Paid-tier judge OR fine-tuned model |
| **Stage-3** (NEW) | On-demand | Operator manual review | Operator authority for any rule scoring <0.50 in Stage-2b |

### Rationale

The original v3.2 §R6 ≥97% target was set from S5 #880 analytical
spike, which used a deterministic keyword-grounded grader on a
20-Q quiz where both compression methods scored 100%. Live
execution (#893) revealed that:

- Free-fleet judges (Cerebras qwen-3-235b, Groq llama-3.3-70b,
  Gemini gemini-2.5-flash) did not chain reasoning reliably for
  boundary cases — most returned "not found in bundle" rather
  than composing two facts.
- Grader strictness varies sharply by family (Groq harsh on
  phrasing; Gemini consumes tokens for thinking before output).
- Free-tier rate limits (Cerebras queue + Groq RPM) prevent
  60-Q runs from completing in one session even with backoff.

The 3-stage gate **trades guarantee strength for operational
feasibility on the free-fleet path**, while preserving a stronger
paid-tier path (Stage-2b) for high-stakes verification. Wave 4
child 7 (constitution compressor) inherits this gate as its
input contract.

### Calibration parameters (carried from #893 D4)

- **Per-family `max_tokens`**: Gemini ≥256 (candidate) / ≥48
  (grader); Groq + Cerebras ≥24 (grader) sufficient.
- **Sequential 3+ s spacing** required on free-fleet path
  (R9.3).
- **N=5 stochasticity check** on Stage-2a only (Stage-2b is
  inherently variable; Stage-1 is deterministic).
