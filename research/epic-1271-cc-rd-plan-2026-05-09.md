## §1 — Problem framing (this team's understanding)

Epic-state truthfulness drift is a **structural impedance mismatch** between three layers:

```
Layer 1 (HONEST):    GitHub label state — status:in-progress, role:manager, type:epic
Layer 2 (DRIFTING):  Manager closeout narrative — comments using "complete", "shipped", "DONE"
Layer 3 (DECORATIVE): Epic body AC checkboxes — `- [ ]` markdown that nothing reads or updates
```

The system is correctly designed to make Layer 1 authoritative — `epic-close-readiness.yml` re-opens Epics closed with open children, `closeout-lint.yml` validates handoff schema. **What's missing is reconciliation of Layer 1 against Layer 3, and lint of Layer 2 against either.** Manager (the role) is incentivized to declare progress in Layer 2 to demonstrate work, but has no obligation to update Layer 3, and Layer 1 only gates close events — not in-progress narrative.

This recurs because **honor-system rules scale linearly with discipline; discipline degrades over long sessions**; severity therefore drifts upward. The recent #1130 incident (5 of 10 ACs unmet at "complete" narrative) is the most severe instance to date but follows the #1103 pattern (`- [ ]` ACs never updated, closed with workflow warning).

Empirical literature confirms premature closure / re-opening is a persistent pattern (Eclipse project re-opened bug study; bug-lifecycle research) but treats it as a developer-defect problem, not a Manager-narrative problem. The harness has the unique opportunity to address the Manager-narrative variant because it operates AI agents that can be lint-gated.

## §2 — Gap-by-gap analysis

### Gap 1: Manager narrative ≠ governance state — no automation reconciles
**Verdict**: REAL, fixable. Recommend Manager-language linter on Manager handoff comments + Epic update comments. False-positive risk on words like "shipped" used in non-Epic-completion context. Mitigate with marker-aware regex (`\bEpic\b.*\b(complete|done|shipped)\b`).

### Gap 2: Epic ACs are decorative checklists; never updated
**Verdict**: REAL, fixable. Recommend AC-checkbox reconciliation skill that:
- Parses Epic body for `- [ ]` and `- [x]` AC lines
- Cross-references each AC against evidence catalog (closed children, file existence, sensor outputs)
- Auto-updates checkboxes via gh API on Epic edit / child close events
- Posts a "AC reconciliation report" comment on Epic with delta

### Gap 3: Time-windowed ACs structurally trap Epics
**Verdict**: REAL, partial fix. Recommend `status:awaiting-measurement` Epic state (NEW LABEL) with mandatory `re-evaluation-by:` body field naming explicit date. Workflow re-tags status:in-progress when date elapses. Acknowledged limitation: 7-day windows still real wait time, but the *meaning* of in-progress becomes unambiguous.

### Gap 4: Cross-Epic AC dependencies create deadlocks
**Verdict**: REAL, partial fix. Recommend explicit `depends-on-epic: #N` declaration in Epic body schema. Topological resolver runs at Epic close attempt; circular-dep flagged. For #1130/#1113 mutual reference: rule "AC9 satisfaction = sibling Epic at status:done OR status:awaiting-measurement; not both Epics simultaneously closing each other's ACs."

### Gap 5: EPIC_RESCOPE rule honored in spirit, not artifact
**Verdict**: REAL, highest-impact fix. Recommend formal `EPIC_RESCOPE` artifact with template:
```
EPIC_RESCOPE
deferred_acs: [AC4, AC5, AC7, AC9, AC10]
deferred_reason_per_ac:
  AC4: structural-measurement-window  # category enum
  AC5: dependent-on-producer            # category enum  
  AC7: scope-cut-to-followon            # category enum
re_evaluate_by: 2026-05-16              # for time-windowed
follow_on_tickets: [#1157, #1159, #1234]
signed_by: <consultant alias>            # NOT manager — see Gap 6
```
Closeout-schema gate parses this; rejects close-attempt without it when ACs unmet.

### Gap 6: Manager+Consultant role conflict on AC verification
**Verdict**: REAL, critical fix. The four-eyes principle (regulatory baseline under SOX, DORA, ISO 27001) applies directly. Recommend: **Manager forbidden from signing EPIC_RESCOPE or final close narrative.** Consultant runs AC reconciliation independently and signs. Closeout-lint extended: rejects `Signed-by:` on close artifacts when role matches Manager team.

### Gap 7: epic-close-readiness checks open children but not unmet ACs
**Verdict**: REAL, extension of existing gate. Recommend extending `epic-close-readiness-check.js` to also parse Epic body ACs (via Gap 2's reconciler) and treat unmet ACs as equivalent to open children for re-open decision.

## §3 — Proposed fixes (concrete mechanisms)

### F1: AC-checkbox reconciliation skill + workflow (addresses Gaps 2, 7)
- `scripts/global/epic-ac-reconciler.js` — pure-function module
  - `parseACs(body)` → `[{id, text, checked, evidence_hint}]`
  - `reconcile(acs, evidence)` → updated AC list with checked-by-evidence
  - `evidenceCatalog(epicNum)` → builds catalog from closed children, file existence, sensor outputs
- `.github/workflows/epic-ac-reconcile.yml` — triggers on Epic edit, child close, manual dispatch
- Estimated effort: 1.5–2 days. Risk: medium (evidence-catalog heuristics may misclassify).

### F2: EPIC_RESCOPE artifact + closeout-schema gate (addresses Gap 5, 6)
- Template added to `instructions/epic-governance.instructions.md`
- `closeout-lint.yml` extended to parse `EPIC_RESCOPE` markers; reject close on unmet AC without
- Force Consultant signing (not Manager) per role-baton-routing
- Estimated effort: 0.5 day. Risk: low (additive lint rule).

### F3: Manager-language linter (addresses Gap 1)
- `scripts/global/manager-narrative-lint.js` — runs on `issues.comment` events for `type:epic` issues
- Detects "complete", "shipped", "done", "DONE" near `Epic` references when Layer 1 / Layer 3 honestly says otherwise
- Posts advisory comment with delta; does NOT block (advisory-first per #1211 graduation pattern)
- Estimated effort: 0.5 day. Risk: medium (false positives on non-Epic completion phrases).

### F4: status:awaiting-measurement state (addresses Gap 3)
- New label; documented in `instructions/role-baton-routing.instructions.md` + `ticket-driven-work.instructions.md`
- Epic body schema extension: `re-evaluation-by: <ISO date>` field required when label applied
- Daily cron workflow re-tags `awaiting-measurement` → `in-progress` when date elapses
- Estimated effort: 0.5 day. Risk: low (label additive; existing workflows continue).

### F5: depends-on-epic declaration + topological resolver (addresses Gap 4)
- Epic body schema extension: `depends-on-epic: #N1, #N2` (Markdown-parsable)
- `scripts/global/epic-dependency-graph.js` — builds DAG, detects cycles
- `epic-close-readiness-check.js` extension: cycle-detection on close attempt
- Estimated effort: 1 day. Risk: low–medium (the data is sparse; circular cases rare but real).

### F6: GHS sensor for "declared-complete-but-AC-unmet" (addresses meta-gap)
- New sensor in `scripts/global/sensors/dc.js` (declared-complete) for Epic #1113's GHS aggregator
- Computes: count(epics with Manager-narrative-says-done AND ACs unmet) / total open Epics
- Feeds into Goal Health Score with weight ~0.10
- Estimated effort: 0.3 day. Risk: low (additive sensor; existing #1113 framework).

### F7: Consultant role enhancement — AC reconciliation skill (addresses Gap 6)
- New skill: `role-consultant-epic-close-audit` (separate from existing `role-consultant-critique`)
- Operationalizes the AC reconciliation Consultant must perform before close
- Templates the rubric: each AC must be checked-with-evidence OR named in EPIC_RESCOPE
- Estimated effort: 0.5 day. Risk: low (skill addition; no code change).

## §5 — Open questions (need operator input or sibling-team perspective)

1. **F3 false-positive tolerance**: how aggressive should Manager-narrative lint be? Strict = many false positives; lax = same drift continues. Suggest A/B test on this session's transcript.
2. **F6 sensor weight**: should "declared-complete-but-AC-unmet" be weighted equal to governance-audit violations, or lower? My instinct says equal-or-higher, but operator may disagree.
3. **F7 enforcement vs advisory**: should Consultant role be forbidden by gate from being same person as Manager (signer-independence), or just advisory? Trade-off: harness has 1 operator + 4 AI roles; strict separation might block all closeouts.
4. **Backwards compatibility**: existing Epics (#1130, #1113, #1103) have unmet AC checkboxes that pre-date this Epic. Grandfather (don't enforce reconciliation), or backfill (reconcile retroactively)?
5. **Cross-team AC ownership**: when AC9 says "sibling Epic's ACs satisfied" and that Epic is Codex Team's, who validates? Suggest: each team self-validates its own Epic's ACs; cross-team AC requires explicit handoff signature.
