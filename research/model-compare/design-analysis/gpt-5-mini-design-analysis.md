## Critical analysis — strengths, weaknesses, risks

Strengths of the proposed design
- Uses authoritative fetchers + prompt-builder + multi-LLM redundancy which reduces single-model hallucination risk.
- JSON Schema + validator + resubmission loop increases structural correctness of outputs and makes autoupdate safer.
- Audit trail (encrypted raw responses) preserves provenance and supports rollbacks and investigations.
- Vault-backed secret storage + lock/unlock UX provides strong separation between code and secrets, preventing accidental commits.

Primary risks and weaknesses
1. **Model hallucination / factual drift:** LLMs can invent fields or plausible-but-wrong values; aggregation reduces but does not eliminate this risk. Structural schema does not guarantee factual correctness of numeric limits or availability. Mitigation: prefer authoritative fetchers (provider docs, status pages) and use LLMs primarily for synthesis rather than primary fact extraction.
2. **Schema/format drift & breaking commits:** Automatic commits that add new fields or change schema can break downstream consumers. Mitigation: auto-commit only for non-structural changes; require PR for schema changes (policy toggle).
3. **Quota and cost exposure:** Running multiple LLMs for median consensus can consume quota/costs. Mitigation: prioritize external free models; run full multi-LLM consensus only on diffs or high-impact changes; lightweight extractors for day-to-day.
4. **Secrets handling pitfalls:** Risk of secrets leakage into logs or raw stored LLM outputs. Mitigation: never include secrets in prompts; use ephemeral validation tokens or provider test endpoints that accept masked inputs; redact before logging; use Vault.
5. **Operational complexity:** Playwright crawling + multi-LLM orchestration + validation loops increases failure modes and maintenance burden. Mitigation: incremental rollout with robust monitoring and backups.
6. **Trusting automated reviewers:** A second LLM reviewer reduces risk but could also be gamed by hallucinations in aggregated input. Human-in-the-loop for high-impact diffs remains critical.

## Specific technical recommendations (concise)
1. **Prompt-builder:** version prompts and store templates in `scripts/prompt-templates/`. Include previous matrix snapshot, watchlist, and strict JSON Schema in the prompt. Use deterministic structure: stable keys and explicit types.
2. **Data-extraction-first:** always prefer deterministic extraction (Playwright/CSS/XPath/regex) to read authoritative values (api docs, headers). Use LLMs for normalization and disambiguation only.
3. **JSON Schema + strict validator:** implement a JSON Schema (coercion off) and require complete validation before acceptance. On validation failure, re-run the resubmission flow up to 3 retries. Log validation errors into `logs/ai-matrix/`.
4. **Consensus policy:** default threshold 2/3 across at least 3 researcher LLMs; if only 2 available use unanimity. For numeric limits, require corroboration from an authoritative fetcher (e.g., header or docs). Record confidence scores per field.
5. **Commit policy:** Auto-commit allowed for simple updates (e.g., cost/timestamps/models list additions). Structural changes (new top-level keys, schemaVersion bumps) should open a PR and require human approval. This balances automation with stability. (Client preference for automation is recorded; recommend this hybrid.)
6. **Audit & retention:** encrypt raw LLM outputs to `raw/ai-matrix/` (e.g., AES-256-GCM, key in Vault) and keep 90 days by default. Store plaintext diffs in `research/ai-matrix-changelog.md` with "from the client" notes.
7. **Secrets & provisioning:** use HashiCorp Vault for runtime secret storage; implement `POST /api/fleet-resources/:id/validate` that performs server-side ephemeral validation without logging raw secret values. UI lock/unlock authenticated via GitHub OAuth or local admin tokens. Auto-lock after 10 minutes of inactivity.
8. **Fallback runner:** GitHub Actions primary; dashboard fallback limited to non-blocking updates. When running fallback, tag commits as `auto/fallback` and include reason.
9. **Testing & monitoring:** add Playwright tests for table layout (no horizontal scroll) and unlock/edit flows; add synthetic tests for updater (mocked LLM responses) and aggregator logic.
