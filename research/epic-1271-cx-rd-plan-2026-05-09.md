# Epic #1271 Codex R&D Plan: Epic-State Truthfulness

**Ticket**: #1274
**Parent Epic**: #1271
**Team**: Codex Team
**Date**: 2026-05-09

## Section 1 — Problem Framing

Epic-state truthfulness drift is not primarily a Manager writing problem. It is
an evidence-system problem: narrative, labels, AC checkboxes, child state,
time-window obligations, dependency state, and closeout authority can each say
slightly different things. Humans then read the most optimistic artifact while
governance labels keep the more truthful state.

Codex Team's model: an Epic is truthful only when all state surfaces reconcile
into one deterministic close-readiness table. Manager narrative becomes an input
to validation, not the final source of truth.

Karpathy LLM Wiki use: I searched the repo wiki before web research and found
[[epic-governance]], [[baton-protocol]], [[governance-enforcement]],
[[self-annealing]], [[ticket-audit-pattern]], and [[harness-goal-controls]].
Those pages supplied the local governance vocabulary. Websearch then filled
external patterns for tasklist automation, dependency APIs, time-windowed SLOs,
and independent oversight. I added a raw source, a wiki source page, and the new
[[epic-state-truthfulness]] concept page.

## Section 3 — Proposed Fixes

### Fix A: Epic AC Reconciler

Implementation sketch: add `scripts/global/epic-ac-reconcile.js` to parse an
Epic body, extract AC checkboxes, evidence links, child references, dependency
lines, measuring metadata, and closeout comments. Emit JSON with one row per AC:
`checked`, `evidence_refs`, `child_refs`, `dependency_refs`,
`measuring_until`, `rescope_ref`, and `truth_status`.

Effort: 1.0 day. Risk: medium parser edge cases; keep first version Markdown
line-based and conservative.

### Fix B: Epic Close-Readiness v2 Gate

Implementation sketch: extend `.github/workflows/epic-close-readiness.yml` so
an Epic cannot close when any AC is unchecked unless an accepted `EPIC_RESCOPE`
or `status:measuring` exception exists. Gate should fail if a closeout comment
contains `Epic complete`, `done`, or `complete` while reconciler status is
not `ready_to_close`.

Effort: 0.75 day. Risk: low if advisory-first rollout is used for old Epics.

### Fix C: EPIC_RESCOPE Artifact Template

Implementation sketch: add `.github/ISSUE_TEMPLATE/epic_rescope.md` or an
instruction snippet requiring old AC text, new AC text, reason, impacted child
tickets, approver, and effective date. Add a validator that recognizes exactly
one `EPIC_RESCOPE` block per changed AC.

Effort: 0.5 day. Risk: low; main risk is too much ritual, so keep the template
short.

### Fix D: Time-Window State

Implementation sketch: add `status:measuring` for Epics blocked only by an
elapsed-time AC. Require `Measure-window`, `Recheck-after`, and `Sensor`
fields in the AC evidence line. A scheduled audit comments when recheck date is
reached or overdue.

Effort: 0.75 day. Risk: medium taxonomy update; needs label-lint carve-out.

### Fix E: Cross-Epic Dependency Resolver

Implementation sketch: build on the dependency graph work already shipped.
Parse `Depends-on`, `Blocks`, `Blocked-by`, `Coupled-with`, and native
GitHub issue dependency API results. Detect cycles and block closeout if an AC
points to an unresolved or circular dependency.

Effort: 1.0 day. Risk: medium because native dependency API availability may
vary by plan/repo feature state; text fallback is required.

### Fix F: Consultant-Only Epic Closeout

Implementation sketch: update `instructions/epic-governance.instructions.md`
and gates so Manager may post progress summaries but only Consultant can post
`CONSULTANT_EPIC_CLOSEOUT`. The closeout must include reconciler JSON summary
or URL, rubric rating, and explicit role separation.

Effort: 0.5 day. Risk: low; aligns with existing baton model.

### Fix G: GHS Sensor Integration

Implementation sketch: add Governance Health Score signals:
`declared_complete_unmet_ac`, `unchecked_ac_closed_epic`,
`measuring_overdue`, `rescope_missing`, and `cross_epic_cycle`. Feed them
into #1113 self-annealing health output and daily governance audit.

Effort: 1.0 day. Risk: medium; avoid alert fatigue with severity bands.

## Section 4 — Recommended Priority Order

1. Ship Fix A + Fix B first. Goal lens: G1 Governance wins because narrative
   cannot be trusted until AC reconciliation exists.
2. Ship Fix C + Fix F next. These make exceptions and independent verification
   auditable.
3. Ship Fix D + Fix E next. These handle measuring states and dependency
   deadlocks once close-readiness has a deterministic core.
4. Ship Fix G last. GHS should consume stable signals, not invent them first.

Implementation child sketch if selected:

- Child 1: AC reconciler JSON + tests.
- Child 2: close-readiness v2 gate + advisory/backcompat mode.
- Child 3: EPIC_RESCOPE template + Consultant closeout instruction update.
- Child 4: status:measuring label/taxonomy/audit support.
- Child 5: cross-Epic dependency resolver integration.
- Child 6: GHS sensor integration.

## Section 5 — Open Questions

- Should legacy Epics be grandfathered forever, or should they receive advisory
  drift comments after a sunset date?
- Should `status:measuring` be Epic-only, or also valid for child tickets with
  measurement windows?
- Should native GitHub dependency API be authoritative when it conflicts with
  text, or should text remain canonical until API feature availability is fully
  proven for this repo?
- Should Consultant closeout be a separate issue comment, or should it update a
  structured Epic summary block in the body?
