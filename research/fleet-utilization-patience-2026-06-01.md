# Fleet utilization, patience policy, and cross-team observability

Phase-0 design synthesis for Epic #2518. Addresses 3 failure modes observed today (2026-05-31): qwen-32b timeout-then-fallback (no patience), no cross-host fanout (resource map blindness), no cross-team visibility (concurrent-contention blindness). Designs the patience policy, routing matrix, cross-team claim primitive, and busy-vs-unavailable detector.

## Problem (motivating incidents)

| Incident | Class | Root cause |
|---|---|---|
| #2515 iter-2 qwen-32b dispatch timed out at 1200s | Patience failure | Arbitrary timeout; not p99-derived. Ollama OLLAMA_NUM_PARALLEL=1 default queues + 32b on 4GB-VRAM CPU-offload host is slow by physics |

## 2026 industry context (verified sources)

### Ollama-specific behaviors that matter

- **OLLAMA_NUM_PARALLEL = 1 by default**. Multi-user agents queue sequentially regardless of GPU headroom. Today's 36gbwinresource almost certainly has the default — explaining the 32b timeout when another team is concurrent.
- **OLLAMA_MAX_LOADED_MODELS = 1 by default**. Switching models triggers slow swap.
- **OLLAMA_KEEP_ALIVE = 5min default**. After idle, model unloads → next dispatch pays cold-start cost.
- Ollama benchmark: flatlines at ~22 req/sec; **vLLM handles 3.23× more at 128 concurrent connections** ([The State of LLM Serving 2026](https://thecanteenapp.com/analysis/2026/01/03/inference-serving-landscape.html)).

### Multi-tenant serving alternatives

- **vLLM** (PagedAttention, continuous batching) — purpose-built for multi-user. 16× higher throughput vs Ollama at 128-concurrent.
- **Triton** — orchestration layer hosting TensorRT-LLM/vLLM/PyTorch backends. Multi-model serving native.
- **SGLang + RadixAttention** — workflow-level scheduling, agent-aware cache.

### Multi-agent coordination patterns (2026)

- **ACMI Protocol v1.2** (April 2026): append-only timeline substrate (Upstash Redis) for agent fleet state. Each agent sees fleet's total state.
- **Practical team size: 3-4 agents** before coordination overhead dominates. Megingjord's 4-team baton is at this ceiling.
- **Sparse communication** + **hierarchical decomposition** reduce message density.

### LLM routing + cost optimization (2026)

- **LiteLLM, Bifrost, Cloudflare AI Gateway** — proven patterns for multi-level fallback (primary → secondary → tertiary).
- Cost-aware routing **reduces cost 40-85%** while maintaining ~95% quality.
- **Health-aware routing** tracks latency + error rate to avoid overloaded endpoints.

### Observability metrics that matter

Per industry-standard SLO suite:
- **Time to First Token (TTFT)** — proxy for "model busy vs loading"
- **Tokens Per Second (TPS)** — throughput indicator
- **Queue Depth** — direct contention measure

## Design: 4 solution areas

### 1. Patience policy (per-resource latency profile)

```
inventory/fleet-latency-profile.json
{
  "36gbwinresource": {
    "qwen2.5-coder:32b": {
      "ttft_p50_s": 60, "ttft_p99_s": 180,
      "total_p50_s": 600, "total_p99_s": 1500,
      "timeout_recommendation_s": 3000  // p99 × 2 per policy
    },
    "qwen2.5-coder:7b":  { "total_p50_s": 90, "total_p99_s": 180, "timeout_recommendation_s": 360 }
  },
  "windows-laptop": { ... },
  "penguin-1": { ... }
}
```

Generated by `scripts/global/fleet-profile-bench.js` — runs canonical prompts against each host/model weekly; updates p50/p99.

Per-call timeout = `p99 × 2` (the math today's failing 1200s violated; p99 ≈ 1470s observed historically; 1500 × 2 = 3000s is the patience-correct value).

### 2. Cost-aware routing matrix

```
scripts/global/fleet-router.js → routeForTask(task_class, options) → host+model
```

Algorithm (per LiteLLM/Bifrost pattern, [MindStudio AI Router Guide](https://www.mindstudio.ai/blog/what-is-ai-model-router-optimize-cost-llm-providers)):

```
1. Filter inventory by task_class capability (search/refactor/critique/etc.)
2. Annotate each candidate: { current_load, cost_mult, ttft_p99, family }
3. Sort by score = (1 / cost_mult) × (1 / current_load) × (1 / ttft_p99_normalized)
4. Pick top; fallback chain = next 2 on the list
```

For cross-family review: must respect family-independence per Epic #2511. The router filters out same-family candidates before scoring.

### 3. Busy vs unavailable detection

The critical missing primitive — distinguishes "model loaded + serving someone else (wait)" from "model unloaded / host unreachable (try elsewhere)":

```
scripts/global/fleet-probe.js → probeHostModel(host, model) → {
  reachable: bool,        // TCP connect + /api/version returns
  loaded: bool,           // /api/ps shows model in memory
  busy: bool,             // /api/ps shows last_request < 60s
  queue_depth: int,       // estimated by /api/ps active count
  ttft_recent_ms: int,    // last completed request's TTFT
  decision: "WAIT" | "AVAILABLE" | "UNAVAILABLE" | "ROUTE_ELSEWHERE"
}
```

Decision logic:
- `reachable=false` → UNAVAILABLE (route elsewhere)
- `reachable=true, loaded=false` → AVAILABLE (cold-start cost; ok if patient)
- `reachable=true, loaded=true, busy=false` → AVAILABLE (warm)
- `reachable=true, loaded=true, busy=true, queue_depth<3` → WAIT (use patience)
- `reachable=true, loaded=true, busy=true, queue_depth≥3` → ROUTE_ELSEWHERE

### 4. Cross-team fleet claim primitive

```
HAMR Tier-2:  POST /fleet/claim/{host}/{model}
              GET  /fleet/in-flight → [{ host, model, team, ticket, started_at, eta_s }]

Tier-1 fallback (no HAMR): GitHub repository label
              fleet-holding:<host>:<model>:<team> applied to a tracking issue
```

Each team's dispatcher posts claim BEFORE calling Ollama; releases on response or timeout. Other teams query in-flight first; queue or reroute based on probe + claim age. Mirrors Epic #2451 merge-claim pattern.

Inspired by **ACMI Protocol v1.2 append-only timeline** approach — every claim/release event is durable, audit-trail-friendly.

## Privacy considerations (G4 — added iter-1)

Cross-team fleet observability surfaces metadata that's NOT user-content but IS metadata leakage:

| Surface | Mitigation |
|---|---|
| `/fleet/in-flight` listing | Expose only host/model/team/start-ts; redact body content; ticket numbers are already public |
| Latency profile data | Aggregate weekly; never log per-request signatures |
| Cross-host fanout | Tailscale-mesh-only by default; paid-provider routing opt-in per-tenant |
| Cost telemetry | Already redacted per Epic #2451 Move 2 §Privacy contract |

**Rule for sensitive PRs** (paths matching `*.env`, `*credentials*`, `*.pem`, `*.key`, `*secret*`): force fleet-only routing (no paid-provider fallback even if fleet inconclusive); apply `redact()` from Epic #2451 Move 2 before any probe-adjacent emission.

## Cost-aware fleet-first ladder (G3 — explicit iter-1)

Per Megingjord goal-lens G3 (zero cost > paid), every dispatch traverses this ladder in order:

```
1. Tier-0 (LOCAL):    operator's local machine (penguin-1 sub-2b models)
2. Tier-1 (FLEET):    Tailscale mesh — windows-laptop / 36gbwinresource (Ollama, mult=0)
3. Tier-2 (CLOUD-FREE): Copilot-Pro free-tier (gpt-5-mini, gpt-4.1, raptor-mini, mult=0)
4. Tier-3 (CLOUD-CHEAP): Copilot-Pro cheap-tier (claude-haiku-4.5, gemini-3-flash, mult≤0.33)
5. Tier-4 (CLOUD-STANDARD): Copilot-Pro standard (claude-sonnet, gpt-5.2, mult=1)
6. Tier-5 (CLOUD-PREMIUM): claude-opus-4.5, gpt-5.4 (mult=3) — ONLY with explicit operator opt-in
```

The router NEVER skips a tier without operator override. Caller can declare `max_tier` per-task; default = Tier-3. This codifies "fleet-first not just preference but contract."

## Cross-runtime portability (G5 — explicit iter-1)

The fleet contract is runtime-agnostic. All 4 baton runtimes (Claude Code, Codex, Copilot, Antigravity) consume the same scripts/global/fleet-* APIs. The probe + router + claim primitives expose the SAME interface regardless of which runtime invokes them. `deploy:apply --target all` ensures cross-runtime parity. No per-runtime hardcoded routing logic.
