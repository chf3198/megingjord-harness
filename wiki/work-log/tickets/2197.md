# #2197 — Programmatically eliminate ticket-state drift gaps

> **Source**: github:issue/2197 | **state: CLOSED** | **Labels**: type:epic, status:done, priority:P1, area:governance, resolution:released, lane:code-change
> Mirror of `gh issue view 2197` (derived; edit the GitHub item, not this page).

## Body

## Problem

Ticket governance continues to drift despite extensive instructions and multiple validators. The current system catches some violations, but key invariants are either advisory-only or only enforced on close events.

## Evidence snapshot (2026-05-26)

- Open issues carrying terminal state `status:done`: **21**
- Open issues with multiple `status:*` labels: **3**
- Representative examples include #2103, #2104, #2106 (`status:backlog` + `status:testing`).

Code-level root cause evidence:
- `scripts/global/label-rules.js` enforces closed-state requirements (`state === closed`) but has no explicit prohibition for open tickets carrying terminal statuses.
- `scripts/global/label-lint-close-protection.js` is close-time logic only; rule 1 explicitly no-ops on open issues.

## Why instruction-heavy governance is failing

Current governance relies too heavily on human/operator adherence and post-hoc comments. This creates delayed correction and recurrence under parallel work:

1. Admission gap: invalid state transitions can still be written.
2. Reconciliation lag: drift scanners report but do not always repair.
3. Contract fragmentation: some rules in docs, some in label-rules, some in workflow scripts.
4. Terminal/open mismatch not globally blocked as an invariant.

## Goal

Move from instruction-driven governance to deterministic policy-as-code with immediate enforcement plus idempotent repair.

## Acceptance Criteria

- AC1: Define canonical state machine contract for issue `state` × `status:*` × `role:*` coherence (single source of truth).
- AC2: Enforce open/terminal coherence in blocking validators and label workflows (no open + terminal labels).
- AC3: Implement deterministic reconciler that repairs drift classes automatically when safe; escalates otherwise.
- AC4: Add test fixtures for known drift classes (open+done, multi-status, non-epic backlog+role, stale terminal labels).
- AC5: Add governance SLO dashboard metrics (drift count, MTTR to correction, recurrence by class).
- AC6: File and complete a Phase-0 research ticket documenting architecture options and migration plan.

## Initial plan

1. Research and architecture ticket (Phase-0): model contract and enforcement design.
2. Phase-1 implementation children: validator hardening, workflow enforcement, reconciler, observability.
3. Replay-eval against current backlog to verify drift reduction and no false positives.
