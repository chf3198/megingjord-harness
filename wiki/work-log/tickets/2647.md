# #2647 — Hook activity tracker must not set code_touched on read-only commands

> **Source**: github:issue/2647 | **state: CLOSED** | **Labels**: type:epic, status:done, priority:P2, area:hooks, resolution:released, lane:code-change, anneal:tier-2
> Mirror of `gh issue view 2647` (derived; edit the GitHub item, not this page).

## Body

## Self-anneal origin
Surfaced 2026-06-03 (multiple times this session). The PostToolUse activity tracker (`tool_activity.py#mark_tool_activity`) flips `flags.code_touched` / `docs_touched` to true on **read-only** Bash commands (grep/sed/git-worktree/python that merely *mention* `.js`/`.md` paths), with no actual tracked-file edit.

## Two downstream symptoms (same root cause)
1. **False-blocked `gh issue close`** via `pretool_guard.py:146` (`code_touched && !merge`) — stranded legitimate Epic/child closes (#2615 re-close + #2631/#2635/#2637 closes; worked around via `gh api` + manual state correction). Incident: `touch-flags-set-by-read-only-bash-blocks-epic-close`.
2. **Persistent "Admin baton incomplete — missing: commit/push/pr_create/ci_green/merge" nag** (Stop + PostToolUse). Traced 2026-06-04: `admin_patterns.required_admin_ops(flags, repo_type)` returns the 5-op admin list **iff `flags.code_touched` is true**. A false `code_touched` therefore manufactures a phantom admin obligation on issue-only / read-only sessions, firing the nag all session and degrading G8 (alert fatigue masks a *real* admin-incomplete). Incident: `concurrent-writer-left-uncommitted-tracked-files-in-canonical-main` session also exhibited this.

## Root cause
The tracker keys on tool name + path-like strings in the command, not on whether a tracked file was actually written (no diff/stat check; conflates Edit/Write with Bash reads). `required_admin_ops` + `pretool_guard.py:146` both consume `code_touched` as ground truth, so the single false flag fans out into both symptoms.

## Proposed fixes
- `code_touched`/`docs_touched` set ONLY by Edit/Write/MultiEdit tools (which truly mutate), OR by a Bash command that produces a real `git diff --stat` against tracked files — never by path mentions in read-only commands.
- Regression tests: (a) a read-only grep/sed/python session leaves `code_touched` false; (b) `required_admin_ops` returns `[]` for that session (no phantom admin nag); (c) a legitimate close is not blocked.

## Acceptance
- AC1: a session of read-only Bash + `gh` ops does not flip `code_touched`; a legitimate close is not blocked.
- AC2: with no real tracked-file mutation, `required_admin_ops` returns `[]` and the "Admin baton incomplete" Stop/PostToolUse nag does NOT fire.
- AC3: `code_touched` flips true on a genuine Edit/Write or a real `git diff`-producing Bash command (no false-negative regression).

## Boundary / coordination
- Primary owner of the `code_touched` false-positive (this Epic).
- **Secondary/related**: even when `code_touched` is *legitimately* true, admin work completing in a **worktree** (different cwd) or via `gh api` merge does not update the main-checkout `admin_ops`, producing the same nag — that cross-cwd **state-isolation** angle is owned by **Epic #2091** (harness state isolation root-cause fix), not here. This Epic fixes only the false-`code_touched` source.
- Same "stale gate read" failure class as #2615 / #2623 — coordinate.
