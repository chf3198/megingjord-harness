# #1451 — Validate signer aliases against registry in megalint signer-fidelity

## Body

## Problem

`scripts/global/megalint/signer-fidelity.js` (shipped in #1422) rejects only one client identity (`Curtis Franks`) in `Signed-by:` / `AI-Signature:` fields. It does NOT verify that worker aliases used in artifacts match what `scripts/global/agent-signature.js` would derive from the declared `Team&Model:` + `Role:`.

## Observed evidence

**This-session systemic drift** caught during #1436 critical analysis (2026-05-12):
- Claude Code Manager artifacts signed all session as `Cole Mason` — registry-correct alias is `Orla Mason`
- Admin artifacts signed as `Mira Reyes` — should be `Orla Reyes`  
- Consultant artifacts signed as `Yara Vale` — should be `Orla Vale`
- Phantom CONSULTANT_CLOSEOUT on #1436 signed as `Cole Vale` — not derivable from ANY team×model combination in the registry

40+ artifacts across this session affected. None caught by any lint because the existing signer-fidelity validator only checks against a 1-element blacklist (`Curtis Franks`), not against the positive registry.

## Cryptographic blast radius

Post-#1298, governance artifacts can carry `Crypto-Algorithm: ed25519` + `Crypto-Key-Id: <team>-<role>-v1` + `Crypto-Signature: <b64>`. Key lookup uses `team + role + keyId`. If the textual `Signed-by:` alias is wrong but `Team&Model` + `Role` are correct, crypto verification still passes (signature is over payload, not alias). But the textual attribution becomes a **false provenance trail** — readers see "Cole Vale" and think it's a real worker.

## Goal

Extend `signer-fidelity.js` to:
1. Parse `Signed-by:` + `Team&Model:` + `Role:` from artifact body
2. Derive the expected alias via `agent-signature.js` registry rules
3. Report violation if `Signed-by` ≠ expected derived alias
