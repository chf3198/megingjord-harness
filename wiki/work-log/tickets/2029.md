## Self-anneal Tier-2 Origin

During Epic #1962 red-team execution (2026-05-21), the cross-family fleet model (qwen2.5-coder:32b at <IPV4_REDACTED>:11434) was invoked DIRECTLY via `curl POST http://.../api/generate`, bypassing the HAMR provider wrapper entirely. The fleet model then rated the work using NIST AI 100-1 / OECD AI Principles goal labels (Security/Privacy/Reliability/Explainability/Transparency/Fairness/Accountability/Robustness/Safety/Maintainability) instead of the harness's actual G1-G10 (Governance/Quality/Zero Cost/Privacy/Portability/Resilience/Throughput/Observability/Interoperability/Maintainability).

This exposes TWO compounding governance gaps:

### Gap A — HAMR wrapper lacks harness-governance injection

`scripts/global/hamr-provider-wrapper.js` injects cache headers + spillover hints + sticky-route but does NOT inject harness goals, instructions, or canonical taxonomy into the prompt context. Confirmed via grep:
```
grep -E "harness.goals|goal.context|harness.instructions" scripts/global/hamr-provider-wrapper.js
(no matches)
```

The architectural expectation per `instructions/hamr-routing.instructions.md` is that HAMR is "the cross-team cost+observability layer" — but the implementation today is observability-and-cost-only. There's no governance context injection.

### Gap B — Direct provider calls can bypass HAMR entirely

There's no harness-level enforcement that all governed provider calls route through the wrapper. The wrapper at line 78 says "Pure library — does NOT modify provider call sites." Operators (including me, in this red-team) can curl Ollama directly and the harness has no signal.

## Why this matters (goal-lens)

- **G1 Governance**: provenance assumption broken — agents review work without the goal taxonomy they're supposed to use.
- **G2 Quality**: red-team verdicts are interpretable only when the rater knows the rubric. We just shipped one that didn't.
- **G8 Observability**: HAMR-bypassed calls leave no `cache-stats.jsonl` trace, no `gen_ai.*` emission, no attribution.
- **G9 Interoperability**: cross-runtime / cross-family review is a stated capability of the harness; if the fleet model can't see harness goals, the capability is unreliable.

## Scope

Multi-phase Epic to close both gaps.

### Phase 0 (research-first)
- Audit current HAMR wrapper surface; enumerate what gets injected today (cache headers, spillover) and what should (harness goals, instructions, canonical taxonomy)
- Survey 2026 best-practice for governance-context injection patterns (Microsoft Agent Governance Toolkit Agent OS context-injection, OPA `data` shipping, OpenTelemetry baggage propagation)
- Decision: which context goes into EVERY call (priority sentence) vs. which is hook-driven (full G1-G10 definitions per `goal_lens.py` pattern)

### Phase 1 (implementation, candidate children)
1. **Inject harness priority sentence on every wrapProviderCall** — adds 1-line goal-lens prefix to system prompt; applies to ALL providers including fleet
2. **Goal-keyword expansion** — when prompt mentions a goal keyword, inject the relevant goal definition (parallels `goal_lens.py` for hooks)
3. **Provider-call attribution check** — emit `gen_ai.system` from the WRAPPER, not from the caller, preventing provider-spoofing
4. **Bypass detection** — track all outbound HTTP to known provider endpoints (Ollama, Anthropic, OpenAI, etc.); compare to wrapped-call ledger; flag bypass
5. **Fleet-route enforcement** — add a `MEGINGJORD_FLEET_DIRECT_BLOCK` env that hard-fails on direct curl to fleet endpoints
6. **Tests + stress** — replay-eval the wrapper on existing fleet-call corpus; verify governance context lands in actual prompts
