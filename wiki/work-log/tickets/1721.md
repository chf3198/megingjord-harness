## Goal

Plan the promotion of `Crypto-Algorithm`, `Crypto-Key-Id`, `Crypto-Signature` baton-artifact fields from optional to default. Key management, rotation, revocation, and the operator-side UX.

## Acceptance Criteria

- [ ] AC1a: `docs/howto/crypto-signing-promotion.md` covers current state of optional fields per `instructions/team-model-signing.instructions.md`.
- [ ] AC1b: Documents the promotion timeline: 30-day advisory soak (signed-but-warn) → 14-day required-with-waiver → required default.
- [ ] AC1c: Documents key generation per team — `ed25519` per `Team` in `inventory/team-model-signatures.json`.
- [ ] AC1d: Documents the key rotation cadence (annual + emergency rotation procedure).
- [ ] AC1e: Documents the revocation list distribution mechanism.
- [ ] AC2: Operator UX: `npm run signing:bootstrap` initializes operator-side signing key; `npm run signing:rotate` rotates.
- [ ] AC3: Cites Phase 1.2 (#1718) adversarial spoofing research output to decide which operator classes are exempt (fingerprinting fallback) vs required (crypto-signing).
- [ ] AC4: Defines the opt-out: `MEGINGJORD_CRYPTO_SIGNING_DISABLED=1` (parity with HAMR opt-out); skills falling back to alias-only artifacts must declare reason.
- [ ] AC5 (conditional on #1714): IF #1714 lands option (a) (allow batching), sibling-ticket evidence comments must also carry signatures. IF #1714 lands option (b) (no batching), this AC is moot — drop it at close time.
