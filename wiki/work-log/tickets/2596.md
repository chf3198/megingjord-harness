# #2596 — Harden ci_gate_status against empty gh pr checks --json (false block)

> Mirror of `gh issue view 2596` (derived; edit the GitHub item, not this page).

## Body

## Problem
The pre-merge gate (`pretool_guard` → `live_checks.ci_all_pass` → `ci_gate_status`) false-blocks a legitimate admin merge with "required CI checks are not fully green (live API check)" even when **all checks pass**.

Root cause: `ci_gate_status` runs `gh pr checks <pr> --json name,state,conclusion` and does `json.loads(r.stdout or "[]")`. For some all-green PRs, that `--json` invocation exits 1 with **empty stdout** (observed on #2595, 2026-06-02: plain `gh pr checks` exit 0 / 33 pass / `statusCheckRollup` all SUCCESS, but `--json` returned nothing). Empty → `classify_ci_checks([]) == "unknown"` → not green → blocked. Retrying does not help (persistent for that PR). Workaround used: the API PUT merge endpoint.

## Objective
Make `ci_gate_status` resilient to the empty `--json` quirk by falling back to gh's plain exit code, so a genuinely-green PR is not false-blocked.

## Acceptance Criteria
- AC1: When `gh pr checks --json` yields a non-empty list, behavior is unchanged (`classify_ci_checks`).
- AC2: When `--json` yields empty/unparseable stdout, `ci_gate_status` falls back to the plain `gh pr checks <pr>` exit code: 0 → `green`, 8 → `pending-only`, 1 → `failing`, else → `unknown`.
- AC3: Unit tests (mocked subprocess) cover: non-empty json (unchanged), empty-json + exit 0 → green, empty-json + exit 8 → pending-only, empty-json + exit 1 → failing, empty-json + exit 2/other → unknown.
- AC4: `classify_ci_checks` semantics unchanged. Anti-merge safety preserved: only a true green signal (json green OR exit 0) returns green.
- AC5: CHANGELOG fragment.

## Goal Lens
- G7 Throughput / G6 Resilience: removes a false-block that forces API-merge workarounds.
- G1 Governance: preserves the gate's intent (block on non-green) — only fixes the false-negative on green.
