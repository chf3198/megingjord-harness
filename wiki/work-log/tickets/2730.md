# #2730 — fix: self-anneal — per-ticket workflow violated in Phase-1 batch #2712-2719

## Body

## Problem

Phase-1 delivery for Epic #2707 (tickets #2712–#2719) was batched into a single worktree, branch, and PR (#2726) instead of following the one-ticket-one-workflow contract. Eight separable ACs were treated as inseparable.

## Root cause

Incorrect use of the multi-close batch contract. The contract permits batching when work is genuinely a single inseparable diff (e.g. one config change that simultaneously touches 3 required surfaces). Here, C1–C8 were independent deliverables — each had its own file(s), its own logic, and its own test coverage — making them fully separable.

## What should have happened

For each of #2712–#2719:
1. Create dedicated worktree: `git worktree add ~/devenv-ops-<N> feat/<N>-<slug>`
2. Implement only the AC for that ticket
3. Post MANAGER_HANDOFF on the ticket
4. Post COLLABORATOR_HANDOFF on the ticket
5. Post ADMIN_HANDOFF on the ticket
6. Open PR for that branch only, with `Refs #N` or `Closes #N`
7. Wait for CI green on that PR
8. Post CONSULTANT_CLOSEOUT on the ticket
9. Merge and close

## Acceptance criteria

- [ ] A decision rule is documented in operator memory distinguishing "inseparable single-diff batch" from "convenience batch"
- [ ] All future Phase-N work under Epic #2707 (if any) uses per-ticket workflow
- [ ] Self-anneal memory entry captures the pattern
