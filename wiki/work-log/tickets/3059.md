# #3059 — Research and plan a stuck-issue resolution-consensus protocol (cross-model + web-research, no client deferral)

> **Source**: github:issue/3059 | **state: OPEN** | **Labels**: type:research, status:backlog, priority:P1, area:governance, lane:docs-research, phase-gate:research-first
> Mirror of `gh issue view 3059` (derived; edit the GitHub item, not this page).

## Body

## Problem
When the operator hits an issue it is unsure of (a stuck PR, an ambiguous gate, a novel failure), it has been **deferring the decision to the client** instead of resolving it. The client is design/UAT only and must not be a decision substrate for dev issues (#2578; memory feedback-operator-owns-merges-and-cleanup).

## Desired protocol (to research and plan)
When the operator is unsure how to resolve an issue:
1. Run **CURRENT web-search research** so reasoning uses the latest-and-greatest expertise (not stale training).
2. Get **cross-model consensus** from an **optimal capable** model (e.g. gemini-2.5-pro / fleet qwen-32b) to **validate/verify** the recommended course of action against the harness goals.
3. **Act autonomously** on the validated course; do NOT present an (a)/(b)/(c) menu to the client.

## Scope to plan
- Trigger conditions (what counts as 'unsure' / stuck); the fleet-decision-oracle relationship (scripts/global/fleet-decision-oracle.js).
- Which model/web-search surface is 'optimal' per issue class; how to record the consensus as observable evidence (G8).
- Guardrail: how to detect-and-prevent client-deferral of dev/governance decisions.

## Gate
Research-first; needs iterative cross-model consensus at-or-above 93 before ACs/children. Refs Epic #3051. Related: #2509 (operator decision routing to fleet oracle), #2578, memory feedback-operator-owns-merges-and-cleanup.
