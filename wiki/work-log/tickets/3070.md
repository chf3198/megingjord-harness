## B. Five design considerations (client-directed)

**1. Best model suite for 36gbwinresource?** Partially. Refresh to 2026 SOTA + family diversity:
- Replace qwen2.5-coder:32b with **Qwen3-Coder-32B** (2026-best local code review, ~88% bug detection, ~20GB Q4).
- Add **GLM-4.x (Zhipu)** and **DeepSeek-R1-distill-32B** locally for jury family-diversity (both fit <=22GB Q4).
- Keep **Llama-3.3-70B** on free-cloud (NVIDIA NIM) — won't fit 36GB at Q4. Add **Mistral-Large** + **Gemini** (free-tier) for more families.
- Retire 3b/7b (granite/starcoder/qwen-7b) to "cheap pre-screen only" — too weak to be final judges.
- Restore/repair the second fleet host (<IPV4_REDACTED>) so the jury isn't single-host.

**2. Reviewer instructions/skills/tools?** Currently under-grounded. Give each reviewer: the FROZEN rubric + G1-G10 definitions + explicit scope boundary in-context, plus read-only GROUNDING (relevant source excerpts / `gh issue view` / wiki page) via the fleet-MCP read-only tools (Epic #2791) so they critique facts, not guesses. Under-specified rubric is the documented cause of inconsistent strictness + overconfidence.

**3. Not taxing the paid orchestrator team (Claude/Copilot/Codex/Antigravity)?** Offload the GENERATOR (design/code reviser) to a fleet model; the orchestrator only freezes the rubric + tallies (cheap arbitration). Peer paid teams are NEVER a review pool. Hard token ceiling + fleet/free-cloud provider allowlist. Position-swap (bias fix) only on the final round to control cost.

**4. Memory across iterations?** Reviewers are intentionally STATELESS (no in-family coaching). The fix is a Reflexion-style **critique+resolution LEDGER** (`{round, design-hash, each critique, resolution+evidence}`) passed as context so the panel evaluates the DELTA and cannot re-litigate resolved points — while the reviewer IDENTITY stays fresh per the cross-family invariant. These are compatible.

**5. Preventing out-of-scope / work-for-work's-sake?** Two documented failures:
- Reward hacking in iterative self-refinement (arXiv 2407.04549): refinement can inflate judge scores while true quality drops. Mitigation: orchestrator OWNS+FREEZES the rubric (generator can't edit it); anchor to a held-out gold criterion (e.g. "would this have prevented the original incident?").
- Verbosity bias + scope creep: dedicated **scope-guard reviewer** flags out-of-scope/over-engineering (discounted from score); length-controlled design; early stopping (scores converge in a few rounds).

## D. Recommended optimized pipeline (zero-cost, bias-hardened)
| Stage | Who | Notes |
|---|---|---|
| 0 Freeze | Orchestrator (cheap) | Immutable rubric: scope boundary + G1-G10 + held-out gold anchor. Generator/judges cannot edit. |
| 1 Generate/revise | Fleet Qwen3-Coder-32B | Offloads authoring off paid orchestrator; reads the ledger. |
| 2 Jury critique | 3-5 cross-family judges (fleet GLM+DeepSeek + free-cloud Llama-70b/Mistral) | rubric + grounding + ledger; position-swap final round; use MEDIAN not mean. |
| 3 Scope-guard | 1 dedicated judge | flags out-of-scope/over-engineering; discounted from score. |
| 4 Arbitrate/record | Orchestrator (cheap) | tally median, append to ledger, check stop criteria. |

**Stopping:** `median(capable jury) >= 92` OR a dry round (no new in-scope critique) OR max 3 rounds. **Goalpost:** median-of-capable-jury >= 92 against the round-1 frozen rubric (removes the capped-small-grader, scope-creep, and reward-hacking artifacts that made the old mean-based >93 both unreachable and gameable).
