## Local embeddings / RAG for fleet context (P1-0 child of #2802)

Parent: #2802 (P1-0 fleet context-access layer). Phase-0 source: #2792 (D14, G4). Sibling: #2847.

### Problem
#2802 AC5 (D14, G4): proprietary code + context must never leave the machine — a **local embeddings /
RAG** retrieval path. The shipped slices reuse `wiki-search` over Wiki A/B (D13 non-redundancy), which is
keyword/compiled-index retrieval, not semantic embeddings. A local-embedding RAG (on-device model, no
network) is the remaining AC5 gap.

### Objective
A local, on-device embedding + similarity-retrieval path for fleet context (G4 zero-egress), exposed as a
retrieval source the loop (#2854) / tool adapters (#2847) can query. Tier-graceful: absent a local
embedding model, fall back to the existing wiki-search retrieval (G6).

### Acceptance criteria
- AC1: local embedding + top-k similarity retrieval over a local corpus; **zero network egress** (G4) —
  asserted by test.
- AC2: tier-graceful fallback to wiki-search when no local embedding model is present (G6); declared opt-out.
- AC3: returns a manifest-compatible result the loop/adapters can consume (D15 delta-fetch friendly).
- AC4: side-effects (embed/index) injectable → tests deterministic + network-free.
- AC5: <=100 lines/file; readability <=475.
- AC6: tdd-pyramid spec (+ stress if state-mutating index writes are added).

### Lane / strategy
lane:code-change. test_strategy: tdd-pyramid. overlap_decision: complements (does not replace)
wiki-search; AC2 makes wiki-search the fallback, so no parallel-RAG redundancy (D13).
