# #2009 — Add hook timeouts + BASH_DEFAULT_TIMEOUT_MS to harness deploy template

## Body

## Origin

During Epic #1962 Phase-1 execution (2026-05-20 session), repeated Bash-tool stoppages traced to two compounding root causes:

1. PreToolUse hooks lack timeout configuration — documented as anthropics/claude-code#44435: "PreToolUse hook race causes 'Tool permission stream closed before response received' when hook lacks timeout". Both commit_ticket_gate.py and pretool_guard.py spawn gh / git subprocesses that can block the permission stream. Confirmed in live reproduction during git commit of #1974 PDCA work.

2. Bash default timeout (2 min) too short for harness-typical operations — CI polling until-loops, rebase + force-push chains, and multi-issue gh comment cascades routinely run 3-15 minutes. Default timeout sends every long bash to background mode, costing context for output round-trips.

The deployed harness template at .claude/settings.json.template does not include timeout fields on any hook entry, so every operator inherits the bug.

## Scope

Harness-wide fixes shipped via the deploy template (NOT operator-unique):

- Add "timeout": 5 to every hook entry in .claude/settings.json.template
- Add env.BASH_DEFAULT_TIMEOUT_MS: "600000" and BASH_MAX_TIMEOUT_MS: "1800000" to top-level env block
- Run npm run deploy:apply post-merge so all teams refresh their deployed settings.json
- New stress-test tests/stress-hook-timing.spec.js asserting every hook entry has a timeout field and PreToolUse hooks complete in <2s under typical load
- New doc docs/howto/hook-timeout-configuration.md explaining the configuration + claude-code#44435 reference

OUT OF SCOPE (operator-unique):

- Bash(*) permission allowlists — each operator chooses their threat model
- Deeper state_store.py improvements for active_ticket reset (covered by #1975 follow-on)
