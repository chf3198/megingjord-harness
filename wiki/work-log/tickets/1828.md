## Two real gaps in the current taxonomy

### Gap 1 — `status:backlog` semantic overload for child tickets

Today `status:backlog` means BOTH:
- "Epic itself untouched, child not yet on anyone's radar" (Epic-pending state)
- "Epic is active; this child is recognized but Manager hasn't picked it up yet" (queued state)

Same label, two different operational meanings. Dashboard filters can't distinguish. Operators reading a single ticket can't tell whether a `status:backlog` child is realistically next-up or sitting in a dormant Epic.

**Fix**: introduce `status:queued` label. State machine:
- Independent ticket: `backlog → triage → ready → in-progress → testing → review → done`
- Child of pending Epic (Epic at `backlog`): child stays at `backlog`
- Child of active Epic (Epic at `triage | in-progress | dormant | deferred`): child progresses `backlog → queued → triage → ready → in-progress → testing → review → done`

The transition `backlog → queued` happens when the child's parent Epic itself moves out of `status:backlog`. **Manager-initiated, not auto-cascaded** (per the agentic governance principle that non-reversible automated actions are themselves a failure mode; manual application preserves operator agency; the label-lint workflow advises but doesn't force).

### Gap 2 — Rule E2 doc-vs-practice drift on Epic role

Current `instructions/role-baton-routing.instructions.md` Rule E2: "Epic always carries `role:manager` — never changes."

Audit of this session's 5 Epic closes (#1716, #1736, #1771, #1745, #1798): 100% flipped `role:manager` → `role:consultant` during status:review phase preceding the CONSULTANT_EPIC_CLOSEOUT artifact, then dropped the role at terminal. Strict Rule E2 says this is invalid; every operator did it anyway because the consultant role-name is what CONSULTANT_EPIC_CLOSEOUT semantics REQUIRE.

**Fix**: Rule E2 v2 — "Epic carries `role:manager` (default, throughout backlog/triage/in-progress/dormant/deferred) OR `role:consultant` (transient, only during `status:review` phase preceding terminal close). Collaborator and Admin labels never apply to Epics — those roles act on child tickets, not the Epic itself."

This aligns with industry research findings:
- **F1 orchestrator-worker pattern** ([Augment Code 2026](https://www.augmentcode.com/guides/multi-agent-orchestration-architecture-guide)): orchestrator (Manager) owns the parent; workers (Collab/Admin) own subtasks. Workers never edit the orchestrator's task definition.
- **F8 shared-state scoping** ([Stack AI 2026](https://www.stackai.com/blog/the-2026-guide-to-agentic-workflow-architectures)): "shared state is powerful but can leak information between roles; default to keeping fields scoped to the agent that needs them."

## What this Epic does NOT include (deliberately)

- **Auto-propagation cascade** when Epic transitions. Manual operator transitions preserve agency and avoid the agentic non-reversibility risk per [Mayer Brown Feb 2026 governance research](https://www.mayerbrown.com/en/insights/publications/2026/02/governance-of-agentic-artificial-intelligence-systems). Label-lint emits ADVISORY comments suggesting child-state alignment; operator confirms.
- **Migration script for currently-open tickets**. Existing `status:backlog` on children of active Epics remains valid in the legacy interpretation. Forward-only adoption — new movements use the refined taxonomy; legacy tickets close out unmolested.
- **State-machine visualization diagram**. The text table in `instructions/epic-governance.instructions.md` is sufficient; visual is nice-to-have, not blocking.
- **GitHub Issue Fields migration** (public preview March 2026). Separate future Epic; the label model continues to work for now.

- [ ] AC6: **Label-lint enforces single-status invariant** — exactly one `status:*` label per ticket at any time. Detection: count `status:*` labels; if > 1, post ADR-010 violation comment and fail the label-lint check. Same mechanism as existing role-status combination rules. Without this AC, adding `status:queued` would create a NEW multi-status drift surface (tickets could carry backlog + queued + in-progress); AC6 closes that loophole at the same time as the gap-1 fix lands. Verified live 2026-05-17 against #1793 (carried `status:backlog` + `status:in-progress`; resolved out-of-band via Manager hygiene; AC6 enforcement makes recurrence machine-prevented).
