# #1957 — Add JSON-schema validation gate for cross-runtime config files

> **Source**: github:issue/1957 | **state: CLOSED** | **Labels**: type:task, status:done, priority:P1, area:governance, resolution:released, lane:code-change, merge-evidence-override:approved
> Mirror of `gh issue view 1957` (derived; edit the GitHub item, not this page).

## Body

Parent Epic: #1956

## Problem

`.claude/settings.json`, `.codex/settings.json`, and equivalent cross-runtime config files are authored by one team for another team's runtime. The #1917 regression shipped because the only CI gate validated presence-of-references, not whether the generated config parsed against the target extension's schema.

## Scope

Add a CI gate that validates each cross-runtime config file against the target runtime's published JSON schema before merge.

## Acceptance criteria

- [ ] CI gate (workflow under `.github/workflows/` or extension of `governance:orchestrator-parity`) validates `.claude/settings.json` against the Claude Code extension's `$schema` reference using a Draft 2020-12 validator (ajv or jsonschema CLI).
- [ ] Same gate covers `.codex/settings.json` against the Codex runtime schema if one is published; otherwise advisory.
- [ ] Gate fails on flat-hook-entry regression (the exact shape that broke #1917) — proven by a regression-test fixture committed under `tests/fixtures/`.
- [ ] Gate runs on pull_request and on push to main; failure blocks merge.
- [ ] `npm run lint` clean; ≤100 lines per file.

## Validation evidence

- Regression-test fixture replays the #1917 flat-entry shape and the gate fails with a clear error pointing at the wrapper-format requirement.
- Current canonical `.claude/settings.json` passes the gate.

## References

- Origin: #1917, fix #1952 (jsonschema Draft202012Validator already used manually — promote to CI).
- 2026 best practice: pre-merge JSON schema validation eliminates an entire class of structural regressions.
