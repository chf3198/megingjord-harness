# #1814 — Epic: Optimize cross-family review quality via multi-agent judge ensemble + adversarial critique

## Body

## Epic Summary

Optimize **substantive quality** of cross-family reviews. Today the cross-family judge runs as single-pass Qwen 7B against an operator-summarized prompt with rubber-stamp verdict. This Epic adds rubric-loaded multi-judge fan-out with an adversarial role and inter-judge variance as a confidence signal — feeding the existing `review-score-classifier.js` `agreement` field.

## Problem (verified 2026-05-17 audit)

- `scripts/global/judge-quorum.js`: 2-of-N family pair; no fan-out beyond 2; binary disagreement signal only
- `scripts/global/local-judge.js`: single phi3:mini judge; generic 0.0-1.0 scoring; does not load `inventory/rubric-g1-g9-v2.json`
- `scripts/global/consultant-second-opinion.js`: single rater pair; no variance
- `scripts/global/review-score-classifier.js`: already accepts `confidence` + `agreement` params, but no upstream multi-judge source feeds them

## Design (compatible with existing infrastructure)

| Component | Source | New |
|---|---|---|
| Multi-judge orchestrator | NEW `scripts/global/multi-judge-orchestrator.js` | ≥3 judges from ≥2 families via existing FAMILY_REGISTRY |
| Adversarial prompt library | NEW `scripts/global/multi-judge-prompts.js` | Approving + adversarial + balanced personas; G1-G9 rubric loaded |
| Variance / consensus aggregator | NEW `scripts/global/multi-judge-variance.js` | Mean, stdev, agreement coefficient → consumed by review-score-classifier |
| Goal-rubric loader | Reuse `inventory/rubric-g1-g9-v2.json` | None — read existing data |
| Wire-in | Existing `review-score-classifier.classify({confidence, agreement})` | Pure wiring, no API change |

## Acceptance Criteria

- [ ] AC1: `scripts/global/multi-judge-orchestrator.js` fans out to ≥3 judges across ≥2 families with dispatcher injection (testable, mirrors `judge-quorum.js` pattern).
- [ ] AC2: `scripts/global/multi-judge-prompts.js` provides `approving`, `adversarial`, `balanced` persona templates that LOAD `inventory/rubric-g1-g9-v2.json` G1-G9 boxes into the judge prompt.
- [ ] AC3: `scripts/global/multi-judge-variance.js` computes mean, stdev, and agreement-coefficient (1 - stdev/possible_range) across N judges; emits structured `{mean, stdev, agreement, judges, adversarial_dissent}`.
- [ ] AC4: Variance output integrates with `review-score-classifier.classify()` via existing `confidence` + `agreement` parameters (no API change to classifier).
- [ ] AC5: ≥15 tests covering: fan-out with mock dispatcher, persona prompt rubric loading, variance math (0/low/high stdev cases), adversarial-dissent flag, classifier integration.
- [ ] AC6: G3 preserved — orchestrator uses fleet judges first (qwen2.5-coder family on 36gbwinresource); paid escalation only on high variance, governed by `MEGINGJORD_MULTI_JUDGE_PAID_ESCALATION` env. Default: disabled.
- [ ] AC7: G5 preserved — opt-out via `MEGINGJORD_MULTI_JUDGE_DISABLED=1`; single-judge fallback documented in module header.
