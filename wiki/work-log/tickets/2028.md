# #2028 — OTel gen_ai.* content-semantic validation (red-team Attack #2 mitigation)

## Body

Follow-on from red-team analysis on Epic #1962 (comment 4509891153) — Attack #2 (Semantic Convention Evasion).

## Origin

Red-team proposed: "Obfuscated code or unconventional logging practices to evade detection by OpenTelemetry gen_ai.* semantic conventions." OpenTelemetry validates the gen_ai.* SCHEMA but NOT the CONTENT semantics — an agent could emit `gen_ai.system="anthropic"` while actually calling a different provider, defeating attribution.

## Scope

Add content-semantic validation alongside the schema check. Validate `gen_ai.system` against the declared provider in the wrapper invocation, `gen_ai.request.model` against the registry, and `gen_ai.operation.name` against the allowed enum.

## Acceptance criteria

- [ ] AC1: `scripts/global/otel-gen-ai-content-validator.js` cross-references emitted attrs against the wrapper's declared provider/model
- [ ] AC2: mismatch results in incident-log entry + advisory; production hard-fail behind feature flag
- [ ] AC3: tests cover valid emission + 3 evasion patterns (wrong system, wrong model, wrong operation)
- [ ] AC4: G4/G8 — no PII or credential leakage in incident logs (use existing sanitize)
