# #1340 — Governance drift: Agent Baton section displays stale in-progress tickets with no active orchestrators

> **Source**: github:issue/1340 | **state: CLOSED** | **Labels**: type:task, status:done, priority:P2, area:governance, resolution:completed
> Mirror of `gh issue view 1340` (derived; edit the GitHub item, not this page).

## Body

## Problem

The Agent Baton section in the dashboard displays tickets marked `status:in-progress` that lack active team orchestrators. These tickets appear to show active work but actually represent stalled governance states. Examples: #1271 (Epic, no assignees), #1256 (Task, no assignees, stale collaborator role).

## Purpose

The Agent Baton section should display a **real-time list of in-progress tickets** with:
- Evidence of active team work by orchestrator (role holder with recent activity)
- Human-readable team/role hints (assignee names, epic context)
- Recent status-change timestamps proving liveness
- No display of tickets in `status:in-progress` with:
  - No active role holder for current baton position
  - No activity in last 7 days
  - Missing role-specific label binding (role:manager, role:collaborator, etc.)

## Research Scope

1. **Ticket drift analysis**: Query all `status:in-progress` issues; categorize by baton role holder state (assigned vs. unassigned, activity window, role label match)
2. **Dashboard data source**: Trace baton-flow.js → renderBatonFlow() → data source (logs/, API fetch, router-log); identify where stale tickets enter the render pipeline
3. **Filter & display logic**: Audit `renderBatonFlow()` filter conditions; verify they exclude stale tickets
4. **Role-state consistency**: Cross-check role labels (role:manager, role:collaborator, etc.) against actual assignee; flag mismatches
5. **Fixes required**: Amend ticket states OR enhance baton filter to enforce real-time activity validation

## Acceptance Criteria

- [ ] **AC1**: Drift analysis complete — catalog all `status:in-progress` tickets; mark each as lively/stale/orphaned by (assignee, role, lastActivity, daysStale) tuple
- [ ] **AC2**: Data source traced — baton-flow.js rendering pipeline audited; identify which source (logs, API, event stream) feeds stale tickets
- [ ] **AC3**: Filter logic reviewed — `renderBatonFlow()` and `normalizeBaton()` checked for activity-window and role-holder validation; gaps documented
- [ ] **AC4**: Ticket governance fixed — all currently orphaned in-progress tickets either (a) transitioned to correct status, or (b) assigned to active role holder with activity stamp
- [ ] **AC5**: Dashboard filter hardened — `renderBatonFlow()` amended to exclude tickets with (lastActivityDaysAgo > 7 OR missing role assignee); **no display of in-progress tickets without active orchestrator**
- [ ] **AC6**: Regression test added — verify baton-flow.spec.js includes case: `in-progress ticket with no recent activity does not render`
