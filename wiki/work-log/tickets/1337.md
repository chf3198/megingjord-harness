# #1337 — Auto-tick Epic task-list when child closes via Refs (not Closes)

## Body

## Problem

Epic #1308 had 8 children referenced as `- [ ] #N — title` in the body, but GitHub's native auto-checkbox feature **did not tick** them when the children closed. Only #1309's checkbox was auto-ticked. After investigation: GitHub's auto-checkbox only fires when the closing PR uses `Closes #N` (or `Fixes`/`Resolves`/etc.) keyword. The harness explicitly uses `Refs #N` per repo policy (so Consultant retains close authority), which doesn't trigger auto-checkbox.

The wiki concept page from #1309 oversold this: "GitHub native auto-checkbox handles closure-marking." That's only true for `Closes`-style flows.

Result: Epic #1308's body showed `0 of 8 ✅` until manually ticked. Visual ground-truth drift between body and reality.

## Goal

Bridge the gap: when a child issue closes (regardless of how), tick the corresponding `- [ ] #N` checkbox in the parent Epic's body.

## Acceptance Criteria

- [ ] **AC1 — Parent-Epic discovery.** When an issue closes, scan open Epics for `- [ ] #<closed-issue-N>` in their body. If found, that Epic is the parent.
- [ ] **AC2 — Atomic checkbox update.** For each parent found: edit the Epic body to change `- [ ] #N` → `- [x] #N` for the closed child. Preserve all other body content. Idempotent.
- [ ] **AC3 — Workflow trigger.** On `issues.closed`, run the discovery + update. Skip if the issue is itself a type:epic.
- [ ] **AC4 — Test.** Fixture: synthetic Epic body with mixed checked/unchecked task-list. Closing a child should tick exactly that line. Closing a non-listed child should be a no-op.
- [ ] **AC5 — Wiki concept page correction.** Update `wiki/concepts/distributed-self-anneal.md` to remove the inaccurate claim that GitHub auto-checkbox handles this — note this workflow is the bridge.
