## Why this exists

Recent work exposed a recurring defect shape: the harness can prompt for a credential that is already present in the approved local source because environment hydration is not guaranteed to be consistent across entrypoints. This Phase-0 ticket exists to convert that observation into a concrete plan and storage recommendation before any implementation work begins.

## Scope

- inventory credential-consuming entrypoints and current hydration behavior
- identify hydration drift, missing loads, conflicting precedence rules, and silent failure modes
- evaluate whether local plaintext `.env` is still the right storage model for user-specific secrets in this harness
- compare `.env`, encrypted file, OS keychain / password-manager runtime injection, and centralized secret-manager patterns for this exact tool shape
- recommend the smallest safe Phase-1 normalization plan

## Deliverables

- [ ] D1: Entry-point inventory covering Copilot, Codex, Claude Code, hooks, scripts, MCP servers, browser-assisted research flows, and other credential-consuming runtime surfaces.
- [ ] D2: Hydration map showing, for each entrypoint, source of credentials, load timing, precedence, and failure behavior.
- [ ] D3: Drift report identifying inconsistent or missing hydration paths, plus the user-visible consequences.
- [ ] D4: Goal-lens evaluation of the current local `.env` approach against G1-G10, with special focus on G1, G4, G5, G6, and G8.
- [ ] D5: 2026 external research synthesis on developer-tool secret handling, including runtime injection, OS keychain/password-manager options, centralized secret managers, and short-lived/dynamic credentials.
- [ ] D6: Recommendation matrix with tradeoffs for portability, privacy, operator friction, offline/local usability, observability, cost, and migration complexity.
- [ ] D7: Phase-1 implementation plan defining the canonical hydration contract, regression checks, and rollout order.
- [ ] D8: Consultant-ready closeout documenting whether the harness should retain `.env` as the primary local store, demote it to fallback-only, or replace it.

## Initial research anchors

- OWASP Secrets Management Cheat Sheet on centralization, least privilege, auditability, lifecycle management, and the limits of environment variables.
- 1Password CLI `op run` and related developer-environment features for runtime-only local injection.
- HashiCorp Vault static vs dynamic secrets guidance for the security value of short-lived credentials.
- Current 2026 cross-cloud/open-source tool landscape, including runtime-injection and agent-oriented secret tooling, where relevant to the harness goals.

## Constraints

- No Phase-1 implementation tickets until Epic #2291 phase gate is satisfied.
- Recommendations must explicitly justify any lower-priority goal tradeoff under the harness goal order.
- Any proposed solution must account for local/offline developer workflows and multi-runtime portability.

## Done when

- Epic #2291 AC-R1 through AC-R8 are materially satisfied by this ticket’s outputs.
- A clear recommendation exists for the approved secret-storage model and hydration contract.
- The recommendation is actionable enough to author minimal Phase-1 implementation children without reopening basic design questions.
