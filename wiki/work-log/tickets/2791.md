# #2791 — Increase fleet-model share of code-base development and cross-team review

> **Source**: github:issue/2791 | **state: CLOSED** | **Labels**: type:epic, status:done, priority:P2, area:scripts, resolution:released, phase-gate:research-first

## Body

## Problem / opportunity

HAMR routes governed provider calls for cost + observability, and the task router has a **Fleet**
lane for "medium implementation, known-pattern coding" — but in observed practice the fleet/local
models are used overwhelmingly for **review** (cross-family red-team), not for **code-base
development**. The harness pays a paid-tier (Haiku/Premium) premium for implementation work that a
local fleet model could do, and cross-team review (fleet reviewing *other* teams' work) is
available but under-adopted. This Epic researches & plans a HAMR-layer improvement that **raises
the fleet-model share of (a) code-base development and (b) cross-team review**, directly advancing
**G3 (Zero Cost)** without regressing **G2 (Quality)**.

## Goal-lens framing

- **G3 (primary)** — every implementation/review task that a local fleet model can do correctly is
  a paid-provider call avoided. The premium-share governor (`npm run routing:report`, forces fleet
  lane when premium > 20%/7d) measures the spend; this Epic attacks the *cause* (too little fleet
  development), not just the symptom.
- **G2** — fleet-authored code must clear the same gates as any code (lint, tests, baton, gated
  review). Increasing fleet *share* must not lower the quality bar; it must pair with the trust
  gates so fleet output is mergeable, not merely cheaper.
- **G9 / G8** — cross-team review (one runtime's fleet reviewing another's work) and an auditable
  fleet-development-share metric.

## Scope (what this Epic researches & plans)

1. **Fleet-suitable-task identification** — a capability-aware classifier/heuristic that, given a
   ticket/AC, decides whether the *implementation* (not just review) is fleet-appropriate
   (known-pattern, bounded context, clear schema) vs needs Haiku/Premium. Extends the existing
   `model-routing-policy.json` capability matrix and `per_role_lane_preferences` (#2320).
2. **Fleet-development execution path** — how HAMR/cascade-dispatch *executes* (not just signals)
   fleet implementation with ret/verify loops, building on `cascade-dispatch.js --execute` and the
   free-cloud failover (#2619–#2624), so a fleet attempt that fails capability (not availability)
   escalates by one tier — never silently to Premium.
3. **Trust gates for fleet-authored code** — the quality/guardrail conditions under which fleet
   code is mergeable, in concert with **#2356** (low-capability-model-drift guardrails) and the
   test-methodology matrix. (Dependency, not duplication: #2356 prevents drift; this Epic raises
   the volume that must be safe.)
4. **Cross-team review adoption** — raise the *share* of reviews done cross-team by fleet, reusing
   the shipped cross-family red-team (#2041), per-runtime wrappers (#2285), and the cross-team
   review skill effort (#1899). Not a rebuild — an adoption + routing-default improvement.
5. **Observability** — a fleet-development-share metric (alongside the existing premium-share and
   free-cloud-usage reports) so the G3 gain is measured, not assumed.

## Explicit non-redundancy (the #2283 lesson, codified)

- Cross-family fleet **red-team review** already exists (#2041, closed) — this Epic raises its
  *share/adoption* + adds the **development** dimension; it does not re-implement review.
- Low-capability-model-drift **guardrails** are #2356 (dormant) — a **dependency**; this Epic
  consumes them, does not duplicate them.
- HAMR cost levers (cache, spillover, sticky-route) are owned by `instructions/hamr-routing` — out
  of scope; this Epic is about *what work is routed to fleet*, not the cost mechanics within a lane.
