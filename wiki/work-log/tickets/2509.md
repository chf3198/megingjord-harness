## Body

## Self-anneal trigger

Pattern observed 2026-05-31 across this session: when I (operator) face a yes/no development decision (file follow-on Y/N? two-ticket-or-one? approve qwen finding accept/reject?), I default to asking the client. Client correctly flagged this as governance-misalignment:

> 'Why are you unsure of the follow-ons? I would expect you to automatically file follow-ons and self-annealing tickets without my feedback. Would it be possible for you to utilize another family of model for this type of assurance so that you didn't need to consult with the client for development work?'

Per harness operator-identity contract, client = design + UAT only. Operator should NOT escalate routine dev decisions to client.

## Goal

Wire a cross-family fleet-rater (qwen-7b for fast cases, qwen-32b for high-stakes) as the DEFAULT decision substrate for routine operator-internal yes/no calls, escalating to client only when:
- Fleet returns conflicting verdicts across iterations
- Decision crosses scope boundaries (design direction, UAT, budget)
- Client has previously expressed preference on the specific class of decision

## AC

- AC1: Add scripts/global/fleet-decision-oracle.js — wraps qwen-7b/32b dispatch with structured decision input/output
- AC2: Hook integration: when operator drafts a 'should I do X?' question to client, the hook offers 'route to fleet first?' option
- AC3: Document the decision-routing pattern in instructions/role-baton-routing.instructions.md (operator role behavior section)
- AC4: Tests: unit tests covering routine-decision cases + escalation triggers
- AC5: Add memory anchor 'feedback_route_decisions_to_fleet_not_client' so the pattern persists

## Why this is a self-anneal (not a feature)

This addresses operator-behavioral drift. The harness ALREADY has cross-family rater infrastructure (used today for adversarial review). What's missing is the AUTO-ROUTING: when should the operator use fleet vs ask client. This ticket defines + enforces that routing.

## Out of scope

- Replacing client UAT confirmation with fleet (client is still authority on design direction)
- Eliminating MANAGER_HANDOFF / CONSULTANT_CLOSEOUT roles (governance unchanged)
- Specific qwen prompt templates (separate child once routing wired)
