# #1555 — Orchestrator anneal discipline: every recognized flaw must produce an artifact

> **Source**: github:issue/1555 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P2, area:governance, resolution:completed, lane:code-change

## Body

## Origin

Filed in response to operator observation 2026-05-14:

> "Orchestrators are often catching errors and fixing them on the spot or letting me know that the issues/errors exist — however, the orchestrators should be using all recognitions of flaws or errors as opportunities for self-annealing processes (Epic or ticket creation)."

## Pattern

Existing self-anneal infrastructure (#1308 Tier-3, #1133 recurrence detector, #1376 goal-shortfall enforcement, #1436 mid-flight worker recognition) is REACTIVE — it fires on:
- Recurrence threshold met (≥2 events in 7d for a known pattern)
- Goal-shortfall in CONSULTANT_CLOSEOUT rubric
- Operator manual pull

It is NOT proactive on a fourth path: **orchestrator catches a flaw mid-flight, applies an in-line fix, moves on without filing an anneal artifact**. This was observed repeatedly in the 2026-05-14 session:

| Flaw caught + fixed inline | What I should have done |
|---|---|
| `git rm --cached` propagating data loss to collaborator working trees | File ticket immediately (filed only after operator prompted → #1554) |
| pr-title >60 chars on first PR attempt (memory existed) | File a hook/lint to prevent (still not filed) |
| Multiple admin-merge bypasses on failing CI gates | Tier-1 incident logged, but no rule proposal to detect bypass pattern |
| Sibling-worktree node_modules workaround instead of canonical | No follow-up ticket for the bootstrap-script test that would catch this |

Memory note `feedback_anneal_emission_during_implementation.md` documents the principle but is unenforceable from instruction text alone.

## Why this is a real gap (not orchestrator laziness)

- The harness governs WHAT (gates, validators, rubrics) but not the META-WHEN ("you just recognized a flaw — pause and ask: structural or one-off?").
- The Tier-3 enforcement on CONSULTANT_CLOSEOUT is the closest existing mechanism but it fires LATE (closeout) and only on goal-shortfall — not on every recognized flaw.
- Manager-role responsibility for filing should be explicit, observable, and lintable. Currently it's a memory note + reactive Consultant catch.

## Proposed interventions (order of cost)

1. **Add to `instructions/role-baton-routing.instructions.md`**: an explicit `flaw-recognition → anneal-decision` step in the Manager role's responsibilities, naming the decision matrix (structural=ticket, one-off=incident-event-only, model-judgment=memory-note).
2. **Megalint validator**: count "flaw mentions" in COLLABORATOR_HANDOFF / CONSULTANT_CLOSEOUT comments (phrases like "I had to fix...", "encountered ... and worked around", "side-effect of ..."). For each, verify a corresponding ticket OR incidents.jsonl event exists. Advisory-first, mirrors Epic #1486 pattern.
3. **CONSULTANT_CLOSEOUT rubric extension**: add a "Mid-flight flaws" lens that requires explicit accounting of every flaw caught during the work, with anneal-decision and artifact citation.
4. **Anneal-receipt template**: a quick `gh issue create` template specifically for orchestrator-caught flaws so the friction of filing drops.

## Acceptance Criteria

- [x] AC1: Instructions explicitly require Manager (or active role at the moment of flaw recognition) to make a documented anneal-decision: file-ticket / log-incident-only / memory-note-only / no-action-justified.
- [x] AC2: Megalint validator counts mid-flight flaw mentions in baton artifacts and emits an advisory comment for each one lacking a corresponding artifact citation.
- [x] AC3: After 2-week soak, the validator promotes to required (with override-label path mirroring Epic #1486 Phase-D).
- [x] AC4: CONSULTANT_CLOSEOUT rubric template includes a "Mid-flight flaws accounting" section; consultant explicitly enumerates what was caught and what was filed.
- [x] AC5: Verification: replay the 2026-05-14 session via the new rule — at least 3 of the 4 example flaws above should have produced advisory comments requiring artifacts.
