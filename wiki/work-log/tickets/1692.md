# #1692 — research(governance): cross-orchestrator governance without file duplication

> **Source**: github:issue/1692 | **state: CLOSED** | **Labels**: type:research, status:done, priority:P1, area:governance, lane:research
> Mirror of `gh issue view 1692` (derived; edit the GitHub item, not this page).

## Body

## Objective
Research the optimal architecture for **maximum orchestrator compatibility** (Copilot, Cline, Claude Code, Continue, and adjacent tools) **without duplicating governance content** across many file trees.

## Research scope (completed)
- Current instruction/rule loading patterns in major tools
- Compatibility hooks between formats (e.g., AGENTS.md reuse)
- Modern policy-as-code and schema validation patterns for deterministic governance pipelines

## External evidence (May 2026)
1. **VS Code / Copilot custom instructions** support:
   - `.github/copilot-instructions.md`, `AGENTS.md`, `.instructions.md`, and `CLAUDE.md`
   - file-scoped instruction loading (`applyTo`) and configurable discovery locations
   Source: https://code.visualstudio.com/docs/copilot/customization/custom-instructions

2. **Cline rules** support:
   - native `.clinerules/`
   - compatibility detection for `.cursorrules`, `.windsurfrules`, and `AGENTS.md`
   - conditional path-based rule activation (`paths` frontmatter)
   Source: https://docs.cline.bot/customization/cline-rules

3. **Claude Code memory/rules** support:
   - `CLAUDE.md`, `.claude/rules/*.md`, `paths` scoping
   - explicit guidance to import `AGENTS.md` from `CLAUDE.md` to avoid duplication
   Source: https://code.claude.com/docs/en/memory

4. **Continue rules** support:
   - `.continue/rules` local rule sets + shared Hub rules
   Source: https://docs.continue.dev/customize/rules

5. **MCP standardization direction**:
   - capability negotiation and host/server interoperability; strong fit for portable policy execution interfaces
   Source: https://modelcontextprotocol.io/specification

6. **Policy-as-code baseline**:
   - OPA decouples policy decision from enforcement and supports structured inputs/outputs
   Source: https://openpolicyagent.org/docs/latest/

## Key findings
- The ecosystem is converging on **Markdown instruction artifacts + optional frontmatter path scoping**.
- Hard incompatibility is mostly in **discovery paths and minor metadata keys**, not policy semantics.
- Several tools already accept `AGENTS.md` directly or via import/interop path, making it a practical bridge format.
- Redundancy risk is highest when teams hand-edit every tool’s native location.

## Recommendation (target architecture)
Adopt a **single-source governance compiler** model:

1. **Canonical source layer (human-authored)**
   - One provider-neutral governance corpus in repo source-of-truth paths (existing harness pattern).
   - Add machine-readable manifest per document (ID, scope, priority, appliesTo, tags).

2. **Schema layer (deterministic contract)**
   - Add JSON Schema for governance manifest + rule metadata.
   - Validate at lint/CI time to prevent drift and malformed directives.

3. **Adapter emission layer (generated artifacts)**
   - Generate tool-specific files from canonical source:
     - Copilot: `.github/copilot-instructions.md`, `.github/instructions/*.instructions.md`
     - Cline: `.clinerules/*.md`
     - Claude: `CLAUDE.md`, `.claude/rules/*.md`
     - Continue: `.continue/rules/*.md`
   - Use imports/symlinks where tool-native behavior allows; fall back to generated files elsewhere.

4. **Drift control layer**
   - Generated files carry provenance header (`source`, hash, generated-at).
   - CI gate fails when generated files are stale (`governance:sync-check`).

5. **Policy decision layer (optional but recommended)**
   - For high-risk controls, move from “instruction-only” to executable checks via OPA or equivalent in hooks/CI.

## Why this is optimal
- **Extends compatibility** to new orchestrators by adding a small adapter, not duplicating all governance content.
- **Minimizes maintenance** with one canonical authoring flow.
- **Increases correctness** via schema validation and generated artifact drift gates.
- **Preserves portability** with markdown-first governance and tool-specific output adapters.

## Output of this research ticket
- Architecture selected: **Canonical Governance Graph + Adapter Emitters + Drift Gates**.
- Ready for implementation planning in Epic ACs and child tickets.
