# #2603 — Pre-merge gate false-blocks on indeterminate unknown CI state (retry before blocking)

> **Source**: github:issue/2603 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P2, area:hooks, area:governance, lane:code-change, governance:close-without-merge
> Mirror of `gh issue view 2603` (derived; edit the GitHub item, not this page).

## Body

## Problem (G1 governance / G6 resilience)
The pre-merge gate in `hooks/scripts/pretool_guard.py` conflates the **indeterminate** `"unknown"` CI state with `"failing"`:

```py
ci_state = ci_gate_status(pr_ref, cwd)
if ci_state == "pending-only": deny("still pending")
if ci_state in {"failing", "unknown"}: deny("not fully green")   # <-- bug
```

`ci_gate_status` returns `"unknown"` when it cannot determine status — e.g. `gh pr checks --json` returns empty/flaky AND the plain-exit fallback returns a non-standard code (transient gh/API/network blip, GitHub checks eventual-consistency right after completion). That is *"could not determine"*, not *"definitely red"*. Treating it as a hard block **false-blocks legitimate green merges**.

Firsthand: across this session, `gh pr merge --admin` repeatedly blocked with "required CI checks are not fully green (live API check)" while `ci_gate_status` returned `green` on replay seconds later (confirmed 3 ways: plain `gh pr checks` exit 0, `statusCheckRollup` all SUCCESS, `ci_gate_status`=green). Forced a `gh api PUT .../merge` workaround each time.

## Objective
Make the live CI merge-gate robust to a transient indeterminate state: re-query before blocking on `"unknown"`. Preserve the gate — persistent `failing`/`pending-only`/`unknown` still blocks.

## Acceptance Criteria
- AC1: New `ci_gate_status_stable(pr_ref, cwd, attempts=3, sleep_fn=None)` in `live_checks.py` — retries while the result is `"unknown"` (bounded delays), returns the first definitive state (`green`/`failing`/`pending-only`) or `"unknown"` if it persists. `sleep_fn` injectable for tests.
- AC2: `pretool_guard.py` merge branch uses `ci_gate_status_stable`. A transient `unknown`→`green` now ALLOWS; persistent non-green still DENIES; `failing`/`pending-only` semantics unchanged.
- AC3: Unit tests (mocked `ci_gate_status`, no real sleep): `unknown→green` ⇒ green (no block); `unknown×3` ⇒ unknown (block); immediate `green`/`failing`/`pending-only` ⇒ unchanged, no retry.
- AC4: No weakening — `failing` still blocks immediately; green still requires an affirmative signal.
- AC5: CHANGELOG fragment.

## Goal Lens
- G1 Governance: gate must be correct — an indeterminate query must not masquerade as a policy failure.
- G6 Resilience: bounded retry absorbs transient flakiness.
- G7 Throughput: removes the false-block that forced manual API-merge workarounds.
