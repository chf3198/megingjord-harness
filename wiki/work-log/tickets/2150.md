# #2150 — Harden Tailscale and HAMR guardrails with programmatic enforcement

> **Source**: github:issue/2150 | **state: CLOSED** | **Labels**: type:epic, status:done, priority:P1, area:infra, area:governance, resolution:released, phase-gate:research-first
> Mirror of `gh issue view 2150` (derived; edit the GitHub item, not this page).

## Body

## Problem

Fleet and HAMR usage guardrails repeatedly regress after temporary improvements. The current pattern appears too dependent on instruction-layer compliance and not sufficiently enforced by deterministic code-level controls.

Observed failure mode:
- fleet/HAMR hardening lands, then behavior drifts back over subsequent sessions
- route/timeout/context handling is inconsistently enforced
- governance expectations are not consistently backed by blocking gates and replayable tests

## Objective

Make Tailscale fleet + HAMR usage durable via programmatic assurance so guardrails remain effective without relying on instruction memory or manual discipline.

## Expected outcome

Fleet-first + HAMR-enabled behavior for governed work is enforced by code, tests, and CI policy gates, with explicit evidence and anti-regression checks.

## Scope

- Routing policy enforcement for fleet-first execution where capability permits
- Timeout policy hardening with adaptive, deterministic rules per lane/model/workload
- HAMR-assisted context reduction contracts (API/RAG over oversized prompt transfer)
- Cross-role review checkpoints that require strongest available fleet reasoning model for adversarial critique
- Drift detection and auto-filed remediation when fleet/HAMR guardrails are bypassed or weakened

## Acceptance criteria

- [ ] AC1: Add deterministic router gate that fails governed runs when fleet-eligible tasks bypass fleet/HAMR without an approved override artifact.
- [ ] AC2: Add adaptive timeout controller (policy + tests) keyed by lane/model/workload class, with explicit non-hanging bounds and evidence logs.
- [ ] AC3: Add prompt-payload budget policy for fleet reasoning calls with chunking/RAG fallback path and enforcement tests.
- [ ] AC4: Require programmatic adversarial-review checkpoints at Collaborator/Admin/Consultant with strongest fleet model available; missing evidence blocks baton advancement.
- [ ] AC5: Add replay-eval suite covering historical fleet/HAMR regressions; CI fails on regression reintroduction.
- [ ] AC6: Add observability dashboard/report metrics for fleet utilization, HAMR-assisted retrieval usage, timeout causes, and override frequency.
- [ ] AC7: Add auto-anneal trigger for repeated fleet/HAMR governance drift that files a child remediation ticket after threshold breach.

## Initial child-ticket plan

1. Router enforcement gate for fleet-eligible tasks
2. Adaptive timeout policy engine + test harness
3. Prompt budget + chunking/RAG enforcement
4. Baton review evidence gate for strongest fleet model
5. Replay-eval anti-regression corpus for fleet/HAMR drift
6. Fleet/HAMR observability and weekly compliance report
7. Drift-triggered auto-remediation ticketing

## Evidence model

All child deliveries must include:
- deterministic test coverage
- CI gate enforcement behavior
- before/after compliance metrics
- rollback-safe migration notes
