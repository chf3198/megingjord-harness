# #1898 — Tier-2 anneal: pre-commit docs:compile gate on package.json changes

## Body

## Problem

`Doc update required` CI gate fails when `README.md` is out of sync with `package.json` npm scripts. This trap has recurred multiple times in 24 hours:

- PR #1885 (#1612 ship) — failed, fixed mid-flight
- PR #1887 (#1259 ship) — failed, fixed mid-flight
- PR #1895 (#1888 ship) — currently failing for this exact reason

The fix is always the same: `npm run docs:compile` then commit the README change. The trap is also always preceded by adding npm scripts to `package.json`. A pre-commit hook that runs `docs:compile --check` when package.json changes would eliminate the recurrence.

## Goal

Pre-commit hook that runs `docs:compile --check` when `package.json` is in the staged diff; if README is out of sync, auto-run `docs:compile` (or fail with clear remediation).

## Acceptance Criteria

- [ ] AC1: New lefthook entry `docs-compile-on-package-json` runs on pre-commit when `package.json` is staged.
- [ ] AC2: Either auto-fixes (runs `docs:compile` + auto-stages README) OR fails with one-line remediation.
- [ ] AC3: Decision (auto-fix vs fail) documented; consensus: auto-fix is lower friction but auto-stage carries the audit-trail risk of committing changes the author didn't explicitly stage. Recommend FAIL-WITH-FIX-COMMAND for transparency.
- [ ] AC4: Tests covering: README in sync passes, README out of sync fails with clear remediation, package.json untouched no-op.
- [ ] AC5: harness:self-test registry entry.

## Why programmatic

Both teams (claude-code, copilot) keep hitting this trap. Instructional governance ("remember to run docs:compile") clearly insufficient. A pre-commit gate catches it deterministically.

## Composition

- Existing `docs-compile` CI gate (post-PR)
- This validator is the local-pre-commit sibling
