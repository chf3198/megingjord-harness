# #3001 — Refine #2995 shell-write guard to avoid quoted/heredoc over-block

> **Source**: github:issue/3001 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P1, area:hooks, lane:code-change, governance:close-without-merge
> Mirror of `gh issue view 3001` (derived; edit the GitHub item, not this page).

## Body

## Problem
The #2995 shell-write canonical-main guard (`pretool_guard.shell_write_targets` +
`check_terminal`) scans the raw command string for redirect / `sed -i` / `tee` / `cp` / `mv` /
`dd` patterns. It matches these even when they appear **inside a quoted/heredoc string or
comment**, not as an actual shell redirection. When the mentioned path is a real tracked main
file, `evaluate_path` denies the whole command — a false positive.

## Evidence (self-observed, 2026-06-14)
While finalizing #2995, two `gh issue comment` / `gh issue create` calls whose heredoc bodies
merely *described* the guard (text mentioning a redirect to a tracked hook path) were DENIED,
because the redirect regex matched the literal inside the heredoc. Worked around by authoring the
text via the editor tool instead of a shell heredoc.

## Scope / severity
Narrow: only triggers when a command string literally contains a write-idiom token targeting an
**actual tracked main path**. Most common victim: documenting/echoing the guard itself. Low
frequency, but a real correctness/UX papercut in a hook that gates all tool calls.

## Acceptance criteria
- Reduce false positives where a write-idiom appears only inside a single/double-quoted string or
  heredoc body (best-effort shell-quote awareness), WITHOUT reopening the real-redirect bypass.
- Keep fail-open + `evaluate_path` as the deny authority.
- Tests: quoted/heredoc mention of a redirect-to-tracked-path is NOT denied; a real
  redirect-to-tracked-path IS denied; mixed cases.

## Goal-lens
G2 quality / G10 maintainability refinement of the G1 guard shipped in #2995. No security
regression (tightening over-block only).
