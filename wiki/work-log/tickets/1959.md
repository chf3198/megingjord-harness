## Body

Parent Epic: #1956

## Problem

When a breaking change ships before detection (the #1917 pattern), there is no named, repeatable recovery protocol. The actual recovery — file #1951, fix in #1952, fix paths in commit c857ce8, revert the in-flight #1953, refile as #1954 — worked but was ad-hoc. No instruction file defines the playbook so future incidents can follow the same path with the same evidence requirements.

## Scope

Define a named breaking-change recovery workflow with rollback evidence, triage-ticket schema, and smoke-evidence-on-re-merge requirements.

## Acceptance criteria

- [ ] New instruction file at `instructions/breaking-change-recovery.instructions.md` defines the protocol with these phases:
  1. **Detect** — signal sources (CI fail, runtime crash, user report); severity classification
  2. **Revert** — authority and evidence (commit revert, scope-bounded; preserves in-flight casualties for re-author)
  3. **Triage ticket** — schema, labels, manager-handoff content
  4. **Fix** — branch convention, validation requirements, baton flow
  5. **Re-merge with smoke evidence** — what smoke proves the fix works in the target runtime
  6. **Casualty re-author** — how in-flight work that got reverted-as-collateral gets refiled
- [ ] Cross-referenced from `instructions/workflow-resilience.instructions.md` (Tier-3 escalation handoff point).
- [ ] Includes a worked example using the #1917 → #1951 → #1952 → c857ce8 → #1953 → #1954 chain as the canonical reference.
- [ ] Sibling skill `breaking-change-recovery` adapter (under `.claude/commands/`, `.codex/commands/`, `.copilot/skills/`) routes the trigger phrases.
