# #2816 — Phase-2: VS Code Webview transport bridge + CSP audit

## Body

## Problem

`transport.js` currently detects only `demo` and `local` modes. The `vscode` mode path (via `acquireVsCodeApi`) is unimplemented. Running the dashboard as a VS Code Webview fails because:

- `connectSSE()` tries to open an `EventSource` to `/api/events/stream` — invalid in a webview context
- No postMessage bridge exists to receive events from the extension host
- `HOST_ENV` is always `local` in webview (should be `vscode`)

## Scope

Deliver the third mode of the Tri-Mode Architecture (#2803): VS Code Webview Mode.

## Acceptance Criteria

- AC1: `transport.js` sets `HOST_ENV = 'vscode'` when `typeof acquireVsCodeApi !== 'undefined'`
- AC2: `dashboard/js/vscode-transport.js` implements postMessage bridge — acquires the VS Code API handle once and overrides `connectSSE()` to use `window.addEventListener('message', ...)` instead of SSE
- AC3: `event-source.js` routes to vscode bridge when `HOST_ENV === 'vscode'`
- AC4: CSP audit passes — no inline `on*=` attribute event handlers in dashboard HTML, no `eval()`, no `javascript:` URI schemes
- AC5: All new/modified files ≤100 lines; `npm run lint` clean

## Out of Scope

Full ES module refactor (deferred to Epic #2508 implementation phase).
