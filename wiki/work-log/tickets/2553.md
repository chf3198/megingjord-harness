# #2553 — Automate xteam-mcp server registration into harness deploy workflow

> **Source**: github:issue/2553 | **state: OPEN** | **Labels**: type:epic, priority:P2, area:scripts, area:infra, role:manager, status:triage, phase-gate:research-first
> Mirror of `gh issue view 2553` (derived; edit the GitHub item, not this page).

## Body

## Objective

Automate `megingjord-xteam` MCP server registration into the harness deploy workflow so operator deploys leave the active runtimes ready to use `/xteam`, `/xteam-create`, and `/xteam-status` without manual registration steps.

## Problem

`npm run deploy:apply` deploys skills, instructions, hooks, scripts, agents, wiki, and dashboard to runtime homes, but it does not register the `megingjord-xteam` MCP server. After a clean deploy, the xteam prompts are not available until each runtime is configured separately.

## Scope

Automate idempotent `megingjord-xteam` MCP registration for the active runtimes using their corrected config surfaces:

| Runtime | MCP config location | Method | Team env var |
|---|---|---|---|
| Claude Code | `~/.claude.json` | JSON merge at root `mcpServers` (user scope) | `MEGINGJORD_XTEAM_TEAM=claude-code` |
| VS Code Copilot | `~/.config/Code/User/mcp.json` | JSON merge at `servers` (default profile only in this epic) | `MEGINGJORD_XTEAM_TEAM=copilot` |
| Codex | `~/.codex/config.toml` | TOML merge at `[mcp_servers.*]` | `MEGINGJORD_XTEAM_TEAM=codex` |
| Antigravity | `~/.config/Antigravity/User/mcp.json` | JSON merge at `servers` | `MEGINGJORD_XTEAM_TEAM=antigravity` |

## Out of Scope

- Gemini CLI
- New xteam-mcp server features
- Changing xteam-mcp business behavior
- VS Code multi-profile path discovery in this epic (tracked by #2567)

## Constraints

- `deploy.sh` remains at exactly 100 lines; registration logic lives in a separate helper.
- Registration must be idempotent and preserve unrelated existing config entries.
- Registration writes must fail safely without corrupting existing target config.
- Server path must be absolute and resolved from repo root at deploy time.
- Config registration is not treated as proof of prompt activation; runtime proof is tracked by #2565 and consumed by C5 #2559.
