# #1911 — Tier-2 anneal: post-merge-sweep validator (force-close drifters from PR Closes-keywords)

## Body

Parent Epic: #1894

## Problem

GitHub's auto-close-via-PR-keyword is unreliable across PRs with multiple close-keywords:

**Recurrence log (last 24h)**:
- PR #1887 (#1113 children): body had `Closes #1259 #1260 #1262`. Auto-close fired for #1259+#1260 but NOT #1262. Epic Close-Readiness Gate auto-reopened #1113 → required manual `gh issue close 1262`.
- PR #1763 (#1758 family): body had `Closes #1758` only. Commit message had `Closes #1758, #1760, #1761, #1762`. Auto-close fired for #1758 only — but the Epic Close-Readiness Gate then auto-reopened #1758 because #1760/#1761/#1762 stayed OPEN. Required manual close of all 4 today.

The unreliability has two distinct surfaces:
1. **Selective miss**: PR body multi-keyword → some auto-close, some don't (mechanism unclear)
2. **Commit-vs-PR-body**: GitHub only reads PR BODY keywords, not commit message keywords

Either pattern produces orphaned-OPEN tickets that the close-readiness gate detects only when their parent Epic is closed — making the gate the trap-detector but not the trap-fixer.

## Goal

Post-merge-sweep validator: when a PR merges, parse ALL `Closes #N`/`Fixes #N`/`Resolves #N` keywords from BOTH the PR body AND commit messages on the merged branch. For each cited #N, assert state=CLOSED within 30 seconds of merge. Force-close any drifters with audit-log entry.

## Acceptance Criteria

- [ ] AC1: `scripts/global/post-merge-sweep.js` — runs on `pull_request` event `closed` with `merged=true`
- [ ] AC2: Parses `(Closes|Fixes|Resolves)\s+#(\d+)` from PR body + every commit message in the merged push
- [ ] AC3: For each cited ticket, polls `gh issue view --json state` (up to 30s) waiting for state=CLOSED
- [ ] AC4: Tickets still OPEN after poll window get force-closed with comment: `Auto-closed by post-merge-sweep — PR #N body+commits cited Closes #M but GitHub auto-close did not fire`
- [ ] AC5: Audit log entry per drift to `~/.megingjord/post-merge-sweep.jsonl`
- [ ] AC6: Workflow `.github/workflows/post-merge-sweep.yml` (advisory-mode initially per Epic #1612 promotion model)
- [ ] AC7: Tests: PR with single keyword (no-op), PR with multi-keyword (all closed → no-op), PR with multi-keyword (one drifter → force-closes), edge case (cited ticket already closed before merge).
- [ ] AC8: harness:self-test registry entry.

## Why programmatic

Three recurrences in 24h on this specific trap class. Instructional governance ("verify all Closes keywords took effect") is unsustainable for less-capable models AND for capable-model fatigue. A post-merge listener is the durable fix.
