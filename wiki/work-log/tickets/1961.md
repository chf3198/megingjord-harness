# #1961 — Prevent line-cap violations escaping to main via incremental expansion

> **Source**: github:issue/1961 | **state: OPEN** | **Labels**: type:epic, status:backlog, priority:P2, area:hooks, area:scripts, role:manager
> Mirror of `gh issue view 1961` (derived; edit the GitHub item, not this page).

## Body

## Objective

Line-cap violations (≤100 lines, lint-enforced) can accumulate in main when a file is created within the cap but grows across subsequent commits. No local hook fires at `git pull` time to surface pre-existing violations, so a collaborator discovers the debt only at CI (`lint-required` failure on PR), requiring `--admin` bypass every time.

**Known instance:** `inventory/orchestrator-governance-parity.json` grew from ~49 lines (creation, #1912) to 150 lines after wiki-parity additions (#1933). This now causes `lint-required` to fail on every PR opened against main, requiring `--admin` bypass as a recurring tax.

## Root cause

1. No `post-merge` lefthook — pulling from main does not surface pre-existing violations before the collaborator starts work.
2. File grew via incremental expansion of a logically distinct section (`wikiDocsParity`) into an existing file rather than a new file. The pre-push `lint-line-cap` hook would have caught this if lefthook was installed in that checkout; evidence suggests it was not (commit `453082c`).

## Expected outcome

- Advisory `post-merge` lefthook runs `npm run lint` and surfaces violations immediately on pull — before any new code is written on top of debt.
- Pre-existing `orchestrator-governance-parity.json` violation resolved by splitting the file (extract `wikiDocsParity` block → `orchestrator-wiki-parity.json`).
- Zero `lint-required` CI failures caused by pre-existing violations on future PRs.

## Research-first acceptance criteria

### AC-R1 — Full violation audit
Enumerate all files currently exceeding the 100-line cap (or within 10 lines of it as a leading indicator). Produce a prioritized remediation list. Close with Consultant rubric ≥7.

### AC-R2 — Post-merge hook design
Research advisory vs blocking semantics, output format, and whether `post-merge` fires after `git pull --rebase` vs `git merge`. Confirm lefthook event support. Produce a hook design doc. Close with Consultant rubric ≥7.

### AC-R3 — File split strategy for parity.json
Confirm the `wikiDocsParity` extraction approach: target files, line counts, any consumers that reference the key by path (scripts, CI, tests). Produce a safe split plan. Close with Consultant rubric ≥7.
