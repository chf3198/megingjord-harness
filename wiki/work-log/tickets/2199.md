## Body

## Problem

Session repeatedly enters stuck states during web-search tool invocation, requiring manual Cancel to recover. This has recurred multiple times in a single workflow and is now a throughput + reliability defect.

## Evidence

- User reported at least 4 forced Cancels to free the session.
- Pattern: long-running "Searching the web for..." state without completion signal.
- Recovery currently depends on manual intervention (Cancel/reload).

## Impact

- Breaks operator flow and increases partial/exploratory dead-ends.
- Encourages instruction-heavy workaround behavior instead of deterministic tool routing.
- Violates resilience expectations for governed execution.

## Acceptance Criteria

- AC1: Add bounded timeout + fallback wrapper around web-search tool calls.
- AC2: Add per-session circuit-breaker after repeated web-search stalls (auto-disable that tool path for session).
- AC3: Add deterministic fallback path (local repo evidence + GH API) when web-search path degrades.
- AC4: Emit structured anneal/drift event when stall threshold breached.
- AC5: Add regression tests for "stuck on searching" pattern.

## Notes

Immediate tactical mitigation in current session: no web-search tool usage.
