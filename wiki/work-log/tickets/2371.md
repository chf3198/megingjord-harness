# #2371 — fix(hooks): pretool_guard regex misclassification — Edit content as path + Bash substring matches

## Body

## Problem

Two regex misclassification bugs in `hooks/scripts/pretool_guard.py` rejected legitimate operations across multiple shipping sessions:

### Bug A: Edit/Write tool `new_string` content read as a path

When the Claude Code Edit tool is invoked with a `new_string` containing path-like substrings (e.g. quoted YAML frontmatter, markdown link references), `canonical_main_enforcer.py` evaluates the CONTENT as if it were a file path. The "neither gitignored nor tracked" rejection branch fires with the literal new_string as the error payload (visible in #2355 commit-2 attempt logs).

### Bug B: Bash command argument list scanned for branch-switch / restore-verb tokens

`RE_BRANCH_SWITCH` regex in `pretool_guard.py:67` matches any string containing the substring representing the latest-commit literal (e.g. anywhere inside an echo header), AND any string containing the restore-verb followed by a token positionally anywhere in the command. False-positive rejections observed when a Bash command's heredoc content contains educational prose about git operations or when split-shell-echo prepares to invoke a restore command.

## Sessions impacted

- #2355: required external-file + python-copy workaround to write Edit content; documented in operator memory `feedback_sync_sh_reverse_direction_trap.md` Pattern 4
- #2357: same workaround required for research file write
- General: any session attempting to document git-restore operations in a heredoc trips the regex

## Acceptance Criteria

- AC1: Edit tool param parsing in canonical_main_enforcer.py distinguishes `file_path` (the actual path field) from `new_string` / `old_string` (content fields); never evaluates content as a path.
- AC2: `RE_BRANCH_SWITCH` regex anchored to actual command boundaries (e.g. anchored to the start-of-command position with whitespace boundaries); only fires on the literal command position, not on argument lists or echo content.
- AC3: `RE_GIT_COMMIT` regex updated to match the commit verb preceded by optional `-c` flag pairs (currently bypassed by the `-c` flag prefix; observed during #2355 commit when using gpg-disable flags).
- AC4: Test fixtures in `tests/pretool-guard-regex.spec.js` (or equivalent Python pytest) covering: Edit with path-like new_string content (allowed), Bash with the latest-commit literal in echo prose (allowed), Bash with the literal in actual git command position (denied appropriately), commit invocation with intermediate `-c` flags (state advance recognized).
- AC5: Cross-validate against the new validator from #2370 to ensure the regex fixes don't open signer-fidelity bypasses.
