# #3051 — Guardrail the Admin's mandatory ticket-worktree merge before ADMIN_HANDOFF

## Epic Summary

**Self-anneal (client-flagged).** Across a long session the **Admin baton role repeatedly deferred its mandatory merge to the client** instead of performing it on a passing review — 6+ times (incident `admin-handoff-without-mandatory-merge`, high). The Admin contract is: review the Collaborator's work, and **if the PR rates at or above 93 against the harness goals AND CI is green, the Admin MUST merge the ticket's worktree before emitting `ADMIN_HANDOFF`**. There is **no guardrail** that blocks `ADMIN_HANDOFF` while the linked PR is unmerged-but-mergeable — so the step is silently skippable, and the work piles up unmerged.

This Epic builds that guardrail: **`ADMIN_HANDOFF` is invalid (gate-blocked) when the ticket's worktree PR is mergeable (green CI) and reviewed at-or-above 93 but not merged.**

## Root cause (to confirm in Phase-0)

- The baton schema requires `ADMIN_HANDOFF` to carry `branch:`/`commit:`/signer-independence, but **does not require merge evidence** (a `merged_at`/merge-commit) as a precondition for the handoff.
- The Admin's "merge" is tracked in `admin_ops` (stop-hook `check_admin_ops`) but emitting the handoff artifact is not coupled to it.
- The deferral was rationalized as "needs client authorization" — which **inverts** closed ticket **#2578 (Prevent client arbitration for internal governance conflicts)**: a green, reviewed PR is an *internal* decision, not a client one.

## Incorrect-presumption check (from this session)

- Presumed "admin-merge to default branch always needs explicit client auth" — WRONG for green+reviewed internal PRs (#2578). The classifier correctly distinguished "merge on a passing review" (allowed, Admin duty) from "bypass RED CI" (not allowed). The guardrail must encode BOTH: require the merge when green+reviewed; never bypass red CI.
