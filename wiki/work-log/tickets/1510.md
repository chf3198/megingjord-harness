# #1510 — Epic: Lint coverage + model-facing coding-practice instruction hardening

## Body

## Problem

Two related gaps surfaced this session:

1. **Lint coverage gap.** Today's lint surface enforces only a sliver of the harness's coding standards:
   - `scripts/lint.js`: file size ≤ 100 lines (one architectural rule)
   - `scripts/lint-readability.js`: single-letter vars, magic numbers, function-length ≤ 30 (three readability rules)
   - `prettier --check`: format on a narrow file allowlist (not all source)
   - `eslint` (devenv config): recommended subset + JSDoc Tier 1
   - `markdownlint-cli2`, `ruff`, `shellcheck`: language-specific basics

   Best practices the harness *expects* but does **not** lint for include: cyclomatic complexity, duplication, unused exports, dependency-import boundaries (e.g., `scripts/global/megalint/*` should be pure), security smells (e.g., `--no-verify`, `eval()`, untrusted shell interpolation), naming conventions beyond single-letter, instruction-file structure validation, test-discoverability (bare `assert` outside Playwright runner — already filed as #1489), and PR/commit/branch convention enforcement at the editor level.

2. **AC anti-pattern.** Several recent tickets/Epics contain ACs like *\"AC8: All new files ≤ 100 lines\"* embedded in the ticket body. Linting is automatic and pre-push-enforced; baking it into ACs:
   - Conflates domain acceptance with tooling enforcement.
   - Implies the AC is the source of truth when the lint rule IS the source.
   - Lets an author claim AC completion by reading vs. running.
   - Doesn't generalize: only the rules an author happens to think of get listed.

3. **Model-instruction channel optimization.** Coding standards are scattered across `CLAUDE.md`, `instructions/global-standards.instructions.md`, `AGENTS.md`, individual `.claude/commands/*.md` skills, and ad-hoc memory files. There's no audit of which channel actually reaches the model at the lowest token cost with highest signal — and no enforcement that new standards land in the optimal channel.

## Goal

(a) Drive lint coverage toward parity with the harness's documented coding practices so authors don't need to remember rules.
(b) Eliminate the AC anti-pattern via a megalint rule that flags lint-as-AC.
(c) Identify the optimal channel for each class of coding-practice instruction and consolidate.

## Phase 0 — R&D (required before children, per Phase Gate Rule #1397)

The Phase-0 child filed alongside this Epic will:

1. **Lint-coverage audit.** Enumerate every coding practice referenced in `instructions/**` and `CLAUDE.md`. Mark each: covered-by-lint / partially-covered / uncovered. Quantify gap size.
2. **AC anti-pattern survey.** Scan recently-closed tickets for ACs that read like lint rules. Quantify recurrence.
3. **Instruction-channel inventory.** Map each existing instruction file → its loading mechanism → estimated context-window tokens spent → coverage of distinct practice categories. Identify duplication and dead weight.
4. **Recommendation matrix.** For each gap, recommend the best surface (new lint rule / new megalint validator / CLAUDE.md addition / instruction file merge / removal) with goal-lens justification.

Phase-0 deliverable: `research/epic-NEW-phase-0-design-2026-05-13.md` proposing 2-3 implementation paths for each gap class.

## Children (Phase 1+, depend on Phase-0)

To be filed after Phase-0 design lands. Likely shape:

- Phase-1a: AC-anti-pattern megalint validator (flag ACs that embed lint requirements like \"all files ≤ N lines\", \"prettier clean\", etc.)
- Phase-1b: Top-N missing lint rules implemented (cyclomatic, duplication, dependency boundaries, security smells — prioritized by Phase-0)
