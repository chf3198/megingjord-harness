# #1628 — G5 Portability backing: extend definition to operator-environment variance + opt-in/fallback contract

## Parent

Standalone governance-contract update. Identified during adoption planning for the #1624 GitHub-primitives research and Epic #1604's implementation children.

## Problem

The current G5 Portability definition in \`instructions/harness-goals.instructions.md\` line 22 reads:

> G5 Portability: avoid user-specific coupling; settings-driven behavior preferred.

This covers **code portability** (no hardcoded paths/credentials, runtime config drives behavior) but does **not** cover three real expectations that have surfaced during multi-team operation:

1. **Opt-in classification for resource-dependent features**: features that require keys, secrets, remote services, or expected fleet resources should either be opt-in (if not integral to harness goals) OR ship with explicit minimal-resource fallback.

2. **Minimal-resource fallback when resources are baseline-absent**: G6 Resilience covers \"graceful degradation and fallback paths for partial outages\" — that's *temporary* unavailability. A different case: the operator NEVER had the resource. Air-gapped operators don't have GitHub access; local-only operators don't have Codespaces; some operators don't have Copilot subscriptions; some don't have fleet GPUs. This is the baseline, not an outage. G6 doesn't cover it; G5's current wording is too narrow.

3. **Each operator has a unique resource set**: Claude Code Team, Copilot Team, Codex Team — even within one team, different operators have different local + fleet + remote baselines. The harness should treat operator-environment variance as a first-class portability dimension, not edge-case exception handling.

## Goal

Extend G5's definition to explicitly cover operator-environment variance + opt-in/fallback contract, OR add a new sub-clause / sibling goal that does. Decision space:

- **Option A**: extend G5 wording inline (lowest disruption; same priority position).
- **Option B**: add a sub-clause G5a explicitly addressing resource-availability variance.
- **Option C**: introduce a new G dimension at the appropriate priority slot.

## Out of scope

## Why now

The user explicitly raised this expectation during adoption planning for Epic #1604 implementation children. Without an explicit G5 extension, the implementation children risk shipping features that assume baseline resources not every operator has. Examples that surfaced in #1624's primitive inventory:

- F6 Agentic Workflows (gh-aw) — preview-tier; requires gh extension install + may need Copilot subscription
- F7 Codespaces + Live Share — requires paid plan or hits free-tier limits; needs VS Code
- F9 Claude Code agent teams — requires Claude Code subscription
- F4 Webhooks RECEIVERS — require public endpoint (air-gapped operators can't receive)

The G5 extension is the contract gate that ensures each of these is either deferred to opt-in or shipped with explicit fallback.
