# #1892 — Tier-2 anneal: commit-message-ticket-consistency validator

> **Source**: github:issue/1892 | **state: OPEN** | **Labels**: type:task, status:backlog, priority:P2, area:governance, lane:code-change, anneal:tier-2
> Mirror of `gh issue view 1892` (derived; edit the GitHub item, not this page).

## Body

## Problem

Commit messages contain `#N` references that are NOT validated against the branch's lead ticket or the PR's Refs/Closes lines. When the commit-creating hook picks up a wrong ticket number (from lease context, stale environment variable, or prior session bleed), the audit trail silently corrupts.

**Pattern instance** (2026-05-18):
- Copilot Team's commit on branch `feat/1399-research-first-epic-gate` shipped with subject: `docs(governance): codify research-first epic phase gate (#1612)`
- PR body said `Refs #1399`, branch encoded `1399`, but commit subject embedded `#1612` (claude-code's concurrent Epic)
- Copilot's local pre-commit hook ran from stale lease context and accepted the wrong number
- The CONSULTANT_CLOSEOUT noted "local hook context mismatch enforced #1612 reference during commit" — but no validator caught it before merge

This is downstream of #1891 (lease-worktree mismatch) but independently worth catching, because the trap can also occur with stale env vars, partial branch creation, or operator typo.

## Goal

Pre-merge check: commit-subject ticket references must match either (a) the branch's lead ticket extracted from `feat|fix|hotfix/<N>-...` OR (b) appear in the PR body's `Refs #N`/`Closes #N` lines.

## Acceptance Criteria

- [ ] AC1: `scripts/global/megalint/commit-ticket-consistency.js` parses commit messages on a PR (via `gh pr view <N> --json commits`).
- [ ] AC2: For each commit, extracts all `#N` references from subject + body.
- [ ] AC3: Validates each `#N` against: branch lead-N (from branch name), PR body Refs N (regex `/Refs\s+#(\d+)/`), PR body Closes N (regex `/Closes\s+#(\d+)/`). Set union must contain every `#N` from commits, or violation fires.
- [ ] AC4: Workflow runs on `pull_request` events `[opened, synchronize, reopened]`.
- [ ] AC5: Advisory-mode initially; promote to required after replay-eval on 100 historical PRs shows < 5% FP rate.
- [ ] AC6: Tests covering: (a) matching commits pass, (b) wrong-ticket commits flagged, (c) prose ticket references in commit body don't false-positive when also cited in Refs.
- [ ] AC7: Stress test (perf budget): p99 < 100ms per PR with 20 commits.
- [ ] AC8: harness:self-test registry wiring.

## Test strategy

`tdd-pyramid+stress-test` per Epic #1875.

## Why programmatic

Lower-capability models in Copilot auto-mode don't reliably notice when their local environment leaks state into commit messages. A CI-time consistency check is the durable defense.

## Composition

- Branch-name-required workflow (existing) provides branch lead-N extraction
- evidence-completeness (existing) already validates Refs in PR body
- This validator joins the two existing surfaces
