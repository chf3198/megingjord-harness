# #1342 — Define Epic status:in-progress exit condition (auto-transition to dormant)

> **Source**: github:issue/1342 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P2, area:governance, resolution:completed, lane:code-change
> Mirror of `gh issue view 1342` (derived; edit the GitHub item, not this page).

## Body

## Problem

`instructions/role-baton-routing.instructions.md` and `instructions/epic-governance.instructions.md` define when an Epic *enters* `status:in-progress` ("first child ticket moves to `status:in-progress`") but never define when it *exits*. As a result, Epics accumulate in `status:in-progress` indefinitely — even when no child is currently working and no PR has recent activity.

Live evidence (2026-05-11 audit): 5 Epics found in `status:in-progress` with zero active children — #1271, #1245, #1133, #1130, #1113. Manager out-of-band swept them to `status:dormant` with EPIC_PAUSE comments naming resume triggers.

This is the **third instance** of state-truthfulness drift the operator has flagged this session (first: #1103 done+open; second: #1340 done+open; third: this 5-Epic sweep). The systemic gap deserves codification + automation.

## Goal

Define and enforce the `status:in-progress → status:dormant` transition condition. Eliminate the indefinite-in-progress drift class.

## Acceptance Criteria

- [x] **AC1 — Codify exit condition** in `instructions/epic-governance.instructions.md`. Proposed wording: *"Epic exits `status:in-progress` to `status:dormant` when ALL of the following hold for ≥7 days: (a) no child ticket at `status:in-progress`, (b) no linked PR with activity (commit, comment, or review) in the last 7 days, (c) no Manager comment with explicit `EPIC_ACTIVE` marker overriding the auto-transition."* Operator can tune the 7-day window.
- [x] **AC2 — Auto-transition workflow.** New `.github/workflows/epic-state-sync.yml` (or extend an existing) runs daily. For each Epic at `status:in-progress`, evaluates AC1 conditions; if dormant criteria met, posts EPIC_AUTO_PAUSE comment naming the implicit resume trigger (the next child action) and swaps `status:in-progress → status:dormant`. Idempotent.
- [x] **AC3 — Override marker.** Manager can post `EPIC_ACTIVE: <reason>` comment to opt-out of the auto-transition for one cycle. Useful when implementation is genuinely in flight but children-state hasn't caught up.
- [x] **AC4 — Inverse transition** (dormant → in-progress) when a child moves to `status:in-progress` OR a new PR is opened against a child. Mirrors the entry rule from the existing taxonomy.
- [x] **AC5 — Tests.** Golden-file workflow fixtures: Epic with active child → no transition. Epic with all-backlog children + no recent PRs → transitions to dormant. Epic with `EPIC_ACTIVE` marker → no transition.
- [x] **AC6 — Retrospective sweep.** One-shot script run after merge that audits all `status:in-progress` Epics and applies the new rule. Should ideally find ZERO further drift (the 2026-05-11 manual sweep handled the current backlog).

## Test strategy

`tdd-pyramid` — pure-function evaluator in `scripts/global/` + workflow golden-file fixtures.

## Out of scope

- Auto-cancellation (`dormant → cancelled`) — explicit Manager authority remains; no auto-path
- `status:review → done` auto-transition — different rule, not part of this ticket
- Project field / GitHub Projects board sync — workflow scope only
