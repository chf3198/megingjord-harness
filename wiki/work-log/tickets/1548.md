# #1548 — P1: node_modules symlink is tracked in git — #1539 regression source

## Body

## Problem

`node_modules` (the symlink itself, not its contents) is tracked in git history since commit `7135455` (ed25519 signatures merge #1298). The tracked value is the path string `/home/curtisfranks/devenv-ops/node_modules` — a self-reference.

`.gitignore` has `node_modules/` (trailing slash, directory pattern). It does NOT match `node_modules` when that name refers to a symlink-as-file. So `git add node_modules` (which someone did during the ed25519 merge) succeeded, and every subsequent checkout/pull restores the broken self-link.

## Evidence

```
$ git ls-files node_modules
node_modules

$ git show HEAD:node_modules
/home/curtisfranks/devenv-ops/node_modules

$ git log --oneline -- node_modules
7135455 feat(governance): add additive ed25519 artifact signatures (#1298) (#1444)
daa184a feat(governance): merge #1298 ed25519 signatures with #1376 Tier-3
```

## Why this explains #1539's regression

After #1539's filesystem-level fix (mv real-dir into main, re-symlink worktrees), the next `git checkout`/`git pull` on main reset main's `node_modules` back to the tracked self-symlink. Verified directly: main's node_modules is currently the self-symlink again, with mtime matching the most recent main-checkout pull.

This means my #1539 fix was non-persistent. The 7 Codex worktrees + locked auto-worktree may already be re-broken or will re-break on their next checkout.

## Fix

1. `git rm --cached node_modules` — untrack the symlink
2. `.gitignore`: change `node_modules/` to `node_modules` (covers both directory and symlink forms)
3. After merge: re-do the #1539 filesystem fix (rm broken symlink, restore real dir at main)
4. Bootstrap script sanity check (#1540) becomes the safety net for future occurrences
