# #2282 — Phase-1 P1-2: XGrammar baton-artifact grammar (T1 — prose-collision elimination)

## Body

## Phase-1 P1-2 child of Epic #2261

T1 (constrained decoding) deliverable. Eliminates the prose-collision trap class entirely by making it physically impossible for the model to emit peer-artifact-name tokens outside the canonical header position of a baton artifact.

## Problem

Baton-artifact validators do plain substring matching (`body.includes('CONSULTANT_CLOSEOUT')`). When a Manager/Collaborator/Admin comment references a peer artifact name in prose (e.g. "evidence cited in CONSULTANT_CLOSEOUT"), the validator creates a phantom artifact entry and trips a role-mismatch failure.

Documented recurrences this session: #2063 + #1337 + plus #1614 (the existing anneal ticket tracking this trap class). Each recurrence costs a comment-PATCH + CI rerun.

## Why T1 (not T2)

Parser-side fix (#1614 path) requires updating EVERY validator that does substring matching. Generation-side fix (T1) prevents the input from ever existing — the model literally cannot emit the bad token sequence. Both paths are complementary defense-in-depth; T1 is the structural elimination.

## Suggested approach

Use XGrammar (or OpenAI/Anthropic strict-mode JSON Schema with grammar extension) to define a baton-artifact grammar where:
- Canonical header lines (`## MANAGER_HANDOFF`, `## COLLABORATOR_HANDOFF`, `## ADMIN_HANDOFF`, `## CONSULTANT_CLOSEOUT`) are the ONLY position where the artifact-name token sequence may appear
- All other positions in the body use hyphenated/quoted forms (`consultant-closeout-artifact`, `the consultant closeout`, etc.) or single-token references
- Apply to the baton-comment-build helper script (`scripts/global/baton-comment-build.js`) so all agent-generated artifacts pass through constrained generation
