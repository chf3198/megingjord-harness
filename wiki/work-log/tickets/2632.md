# #2632 — Define deterministic epic and ticket state semantics

> **Source**: github:issue/2632 | **state: OPEN** | **Labels**: type:epic, priority:P1, role:manager, area:governance, status:deferred

## Body

## Goal
Establish deterministic, programmatic enforcement of Epic and ticket state so GitHub issue filtering yields predictable, truth-preserving results across teams.

## Problem statement
Current workflow can produce semantically ambiguous states (for example, Epics closed while follow-on work remains open under mixed scope interpretations). This degrades query reliability and governance confidence.

## Scope (Phase-0 only)
This Epic is **research-first** and starts with one research/planning child ticket only.

The research phase must:
- Map current state lifecycle semantics for Epics and child tickets
- Identify mismatch patterns between intended governance and observed label/state reality
- Propose deterministic state-transition rules and enforcement points
- Define measurable acceptance for "query reliability" (expected filters produce expected sets)
- Produce an implementation plan with rollout, migration, and rollback strategy

## Hard gate for Phase-1
No AC checklists and no development child tickets are authorized until the research/planning ticket is complete and receives cross-model review consensus with rating **greater than A-** between:
- Primary planning reviewer
- Cross-model red-team reviewer

## Initial child
To be created immediately as the only child:
- Research + planning ticket (Parent: this Epic)
