## Body

## Parent

Epic #1604. Adopts finding F3 from research #1624.

## Goal

Replace the current prose-level \`Refs Epic #N\` convention with GitHub's native Sub-issues primitive (up to 100 children per parent, 8 levels deep, native parent/child API, automatic progress rollup). Eliminates the trap class my #1614 + #1624 prose-collision memories document at the relationship layer.

## Portability annotation (per #1628)

- **Resource dependency**: Sub-issues + Issue Types GA on GitHub. No new credential or service beyond standard GitHub access.
- **Network dependency**: GitHub API. Same baseline as Issues — air-gapped operators handle via existing local-only workflows.
- **Classification**: integral to harness goals; no special opt-out needed beyond GitHub-access opt-out which is already established.

## Acceptance Criteria

- [ ] AC1: Enable Sub-issues + Issue Types on the org/repo (one-time GitHub Settings op).
- [ ] AC2: scripts/global/sub-issue-link.js pure helper to link child to parent via REST/GraphQL.
- [ ] AC3: Update instructions/role-baton-routing.instructions.md and instructions/epic-governance.instructions.md to reference Sub-issues as the canonical parent/child relationship.
- [ ] AC4: Provide a one-time migration script that walks closed Epics with \`Refs Epic #N\` children and emits the Sub-issue link.
- [ ] AC5: Update closeout-schema validator (#1336 era) to prefer Sub-issue relationship over prose Refs scanning. The prose-scan fallback remains for backward compatibility.
- [ ] AC6: Tests covering link creation, parent-rollup behavior, validator preferring Sub-issue link.
