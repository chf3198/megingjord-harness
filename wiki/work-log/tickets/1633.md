# #1633 — Split Discussions (decisional) from Issues (actionable) cross-team coordination (#1624 F5)

## Parent

Epic #1604. Adopts F5 from research #1624.

## Goal

Separate the *decisional* layer (GitHub Discussions: should we adopt X?) from the *actionable* layer (Issues: ship X by Y). Today we conflate them — design discussions happen in Issue comments alongside acceptance criteria, which makes both noisier.

## Portability annotation (per #1628)

- **Resource dependency**: GitHub Discussions GA. Same baseline as Issues.
- **Classification**: integral to harness goals; no special opt-out needed.

## Acceptance Criteria

- [ ] AC1: Create Discussion categories matching the harness's decisional patterns: Architecture, Cross-team protocol, Tooling research, Operations notes, Q&A.
- [ ] AC2: Update instructions/global-standards.instructions.md with a "decisional-vs-actionable" rule: any open question that doesn't have a concrete deliverable/AC goes to a Discussion, not an Issue.
- [ ] AC3: Document the conversion path (Discussion → Issue when a decision crystallizes; gh CLI supports this).
- [ ] AC4: One-time light migration of historical decisional comment threads from existing Issues to Discussions (operator judgment, not bulk).
- [ ] AC5: Add a check to closeout-schema or a new advisory: flag Issues that have >20 comments and no resolution as candidates for Discussion split.

## test_strategy

peer-review per docs-only lane. Deliverable is the instruction edit + Discussion categories + migration light-touch.
