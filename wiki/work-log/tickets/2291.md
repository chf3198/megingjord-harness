## Body

## Self-anneal trigger

- Operator-surfaced 2026-05-27 after repeated agent prompts for a Tavily API key already present in the local `.env`.
- Trigger type: manual-pull plus recurring entrypoint inconsistency.
- Severity: medium-high. The system still functions, but inconsistent hydration across entrypoints degrades governance, privacy, portability, and operator trust.
- Recurrence anchor: any harness surface that needs local credentials may hydrate environment differently or skip hydration entirely.

## Gap analysis

### What exists today

- Local `.env` is already used in some flows and was sufficient for the current Tavily credential.
- Recent guardrails now enforce env-backed Tavily wiring in tracked config and templates.
- Repo governance strongly prefers settings-driven, portable configuration and auditable runtime behavior.

### What is missing

1. No single canonical environment hydration contract across CLI entrypoints, orchestrators, MCP adapters, scripts, and research tooling.
2. No deterministic inventory of which entrypoints load `.env`, which inherit shell env only, and which bypass both.
3. No explicit decision record for whether plaintext local `.env` remains optimal for user-specific secrets under the harness goal order.
4. No cross-entrypoint regression check that proves a secret present in the approved local source is visible where intended and absent where not intended.
5. No documented fallback or migration path if the harness should move from plaintext `.env` toward runtime injection, OS keychain, or dedicated secret-manager-backed flows.

## Phase-0 research deliverables

- [ ] AC-R1: Catalog every credential-consuming harness entrypoint and classify its hydration path: inherited shell env, explicit dotenv load, runtime adapter load, config indirection, or missing/ambiguous.
- [ ] AC-R2: Identify drift, conflicts, and gaps in current hydration behavior across Copilot, Codex, Claude Code, scripts, MCP servers, hooks, and browser-assisted research flows.
- [ ] AC-R3: Evaluate whether local plaintext `.env` remains the optimal credential storage pattern against the harness goals G1-G10, with explicit treatment of G1 Governance, G4 Privacy, G5 Portability, G6 Resilience, and G8 Observability.

- [ ] AC-R6: Define a canonical environment hydration contract for the harness, including approved source(s) of truth, precedence rules, failure semantics, redaction expectations, and auditability.

## Initial external research anchors

- OWASP Secrets Management Cheat Sheet: centralize and standardize secret handling, enforce least privilege, audit access, and prefer automated rotation / dynamic secrets where practical.
- OWASP notes environment variables are a weaker fallback than in-memory/file injection mechanisms in orchestrated environments because they are broadly accessible to processes and can leak via logs or dumps.
- 1Password CLI `op run` demonstrates runtime injection for local development so secrets are only exposed to the subprocess duration, while also warning that env vars remain visible to same-user processes.
- HashiCorp Vault guidance on static vs dynamic secrets highlights the security advantage of unique, short-lived credentials over shared long-lived static secrets when the underlying system supports them.
- Current 2026 cross-cloud tool landscape indicates that portable developer-focused secrets systems are converging on runtime injection, granular access control, audit logs, and short-lived or dynamically issued credentials rather than plaintext local files as the long-term ideal.
