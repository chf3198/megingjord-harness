## Agentic MCP loop driver for fleet-model development (P1-0 child of #2802)

Parent: #2802 (P1-0 fleet context-access layer). Phase-0 source: #2792 (D12). Consumes the shipped
substrate: context bundle/render/dispatch (#2815/#2817/#2823/#2831), the sandbox trust gate (#2844),
and the governed MCP tool adapters (#2847).

### Problem
The substrate exists but nothing drives it as an **iterative agentic loop**. #2802 AC1 requires wrapping
the fleet model in an OpenHands/Aider-class loop: assemble context → dispatch to fleet → let the model
call governed tools (#2847) → validate any proposed change in the sandbox (#2844) → feed results back →
repeat until done or escalate. This is the central P1-0 deliverable still unshipped.

### Objective
A loop driver that orchestrates: (1) context assembly + dispatch, (2) governed tool-call handling via
`fleet-mcp-broker.invokeTool` (OA2-gated), (3) proposed-change validation via
`validateProposedChange` (fail-closed), (4) bounded iteration with an escalation exit. Pure orchestration
reusing shipped primitives — no new RAG/tool surface.

### Acceptance criteria
- AC1: bounded loop (max-iterations + budget guard) that interleaves dispatch ↔ governed tool calls ↔
  sandbox validation; a rejected tool call or failed validation does not abort the loop (records + continues/escalates).
- AC2: only authorized tool calls execute (delegates to #2847); only sandbox-trusted changes are promoted.
- AC3: escalation exit on max-iterations / repeated failure / fleet low-confidence (hand back to operator).
- AC4: side-effects injectable (dispatch/tool/sandbox) → loop unit-testable network/process-free.
- AC5: <=100 lines/file; readability <=475.
- AC6: tdd-pyramid spec + **stress spec covering MCP loop lifecycle** (satisfies #2802 AC7 for the loop):
  >=1 chaos path (tool-deny / validation-fail / iteration-exhaustion) + >=1 p99 budget.

### Lane / strategy
