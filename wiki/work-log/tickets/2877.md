# #2877 — Extend consultant-activation-decision.js with stale-baton-detector

## Body

## Summary
Extend `consultant-activation-decision.js` with a stale-baton-detector so that issues with COLLABORATOR_HANDOFF evidence but unadvanced labels (triage-stranded) are not silently skipped by the Consultant activation check.

## Problem
`consultant-activation-decision.js` has an explicit guard that skips activation if `status:testing` is absent. This means issues where the Collaborator posted a COLLABORATOR_HANDOFF but forgot to advance labels from `status:in-progress → status:testing` silently bypass Consultant activation and remain stuck indefinitely. Confirmed by AC-R2 research (#2872).

## Expected Outcome
When `decide()` is called on an issue that has a COLLABORATOR_HANDOFF comment but lacks `status:testing`, it emits `action: 'warn-and-advance'` with a message and the correct target state, instead of `action: 'skip'`. This enables `post-merge-automation.yml` to surface the anomaly and optionally advance labels.

## Acceptance Criteria
- AC-I2.1: `decide()` in `scripts/global/consultant-activation-decision.js` detects: issue has label `status:in-progress` AND COLLABORATOR_HANDOFF comment present AND `status:testing` label absent. Returns `{ action: 'warn-and-advance', reason: 'stale-baton-collab-handoff-not-advanced', target: 'status:testing' }`.
- AC-I2.2: Existing `action: 'skip'` path for `status:testing` absent without COLLABORATOR_HANDOFF remains unchanged (no regression).
- AC-I2.3: `post-merge-automation.yml` handles the new `warn-and-advance` action: posts a GitHub comment on the stranded issue noting the stale baton state (advisory; not a hard block).
- AC-I2.4: Unit tests for `decide()` covering: (a) COLLABORATOR_HANDOFF present + status:in-progress → warn-and-advance, (b) COLLABORATOR_HANDOFF absent + status:in-progress → skip (existing behaviour), (c) status:testing present → activate (existing behaviour).

## Implementation Entry Point
- `scripts/global/consultant-activation-decision.js`: modify `decide()` function; add stale-detector rule before the existing `status:testing` guard exit.
- `.github/workflows/post-merge-automation.yml`: extend the step that handles `decide()` output to handle `warn-and-advance` case.
