## Body

Parent: #2802 (P1-0 fleet context-access layer). Child slice 4 — closes the dogfood loop.

## Objective
Slices 1–3 built the context substrate (assemble → render → dispatch) but `dispatchWithContext`'s
`dispatch` fn is injectable/stubbed. This child wires it to the **real** fleet/free-cloud call and
adds a thin CLI, so a fleet model actually receives auto-assembled context (ticket + repo-map + wiki)
before a dev/review task — the operational end of the loop we have been hand-rolling for reviews.

## Acceptance criteria
- AC1: New `scripts/global/fleet-context-cli.js` — a thin entry that composes `dispatchWithContext`
  (slice 3) with a real `dispatch` backed by the existing zero-cost cascade
  (`cascade-dispatch.js` / `free-cloud-dispatch.js`); no new provider surface, no duplicated routing.
- AC2: CLI parses `--ticket N`, `--paths a,b,c`, `--wiki "query"`, `--task "..."`,
  `--max-context-chars N`; assembles context, dispatches, prints the result + a one-line manifest summary.
- AC3: Pure wiring is unit-tested network-free (injected fake dispatch); the real-provider path is
  manual-verify only (no network in CI). G6 graceful: missing keys / unreachable fleet degrade per the
  existing cascade, never throw unhandled.
- AC4: ≤100 lines, lint clean, readability ≤475; cross-family dogfood review with gemini-2.5-pro
  (rigorous tier per memory) reaches ACCEPT, min(G1–G9) ≥ 7.
