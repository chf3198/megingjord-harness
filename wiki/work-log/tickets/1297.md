# #1297 — Evaluate and migrate to Policy-as-Code substrate for governance gates

## Body

## Problem

Megingjord governance gates today are written as bespoke JavaScript inside `.github/workflows/*.yml` files (e.g., `closeout-lint.yml`, `epic-close-readiness.yml`, label-lint, baton gates). This pattern works but does not produce an auditable decision-log automatically, makes rule-evolution hard to review, and has no clear scaling path as the gate count grows past ~20.

The 2026 industry standard for governance enforcement is **Policy-as-Code (PaC)** — committing rules in a dedicated declarative language separate from the system being governed. The most-adopted tool is **OPA + Rego** (29% market share per [arXiv 2601.05555 Empirical Study of PaC Adoption 2026](https://arxiv.org/html/2601.05555v1)); rising alternatives include **AWS Cedar** (4%, 42–60× faster than Rego, formal verification per [Permit.io 2026](https://www.permit.io/blog/policy-engine-showdown-opa-vs-openfga-vs-cedar)) and **Microsoft Agent Governance Toolkit** (April 2026, agent-runtime-specific, addresses OWASP Agentic Top 10).

This Epic evaluates the cutting-edge PaC landscape, picks a substrate (or hybrid), and migrates the existing governance gates to it.

## Goal

Reduce gate-authoring friction, gain free auditable decision-logs, scale governance to 50+ gates, and align with cutting-edge AI-agent code governance industry standard.

## Phase-0 R&D Gate (required before implementation children)

Mirroring the Epic #1271 pattern: 3 team R&D plans (CC/CP/CX) evaluating OPA, Cedar, Microsoft AGT, and OPAL/Permit.io abstraction. Cross-team review cycle. Operator-delegated synthesis.
