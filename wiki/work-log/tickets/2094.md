# #2094 — HAMR: enforce harness guardrail parity for fleet models (Ollama) vs non-fleet runtimes

> **Source**: github:issue/2094 | **state: CLOSED** | **Labels**: type:epic, status:done, priority:P2, area:infra, resolution:released, lane:research, phase-gate:research-first
> Mirror of `gh issue view 2094` (derived; edit the GitHub item, not this page).

## Body

## Objective

Non-fleet runtimes (Claude Code, Copilot, Codex) operate with full local harness access: they can call `rubric-score.js`, `consultant-checks.js`, `wiki-search.js`, and any `scripts/global/` tool via direct filesystem + terminal access. Fleet models (Ollama instances on fleet devices) route through HAMR but have no equivalent access to these harness primitives.

This Epic tracks the HAMR upgrades required to give fleet models feature parity with non-fleet runtimes for governance-critical harness operations.

## Problem

HAMR currently provides three capabilities to fleet models:
1. **Cost/routing layer** — caching, 429 spillover across 6 providers, sticky-route per tier
2. **Observability layer** — signed telemetry, cache-hit tracking, DPoP auth
3. **Capability dispatch** — `/mcp` endpoint: `bundle:fetch`, `doctor:probe`, `mailbox:read`

What is absent: fleet models have no mechanism to invoke governance-critical tools that run locally on the orchestrator machine:
- `scripts/global/consultant-checks.js` — 16 governance/tool/fleet checks against live GitHub API + telemetry
- `scripts/global/rubric-score.js` — deterministic v2 rubric scoring against issue trail + diff
- `~/.copilot/scripts/wiki-search.js` — semantic search across 163 harness wiki pages
- `scripts/global/emit-event.js`, `consultant-feedback.js`, `baton-comment-build.js`

A fleet model acting as Consultant cannot produce a compliant CLOSEOUT because the tools that populate mandatory fields (`checks_run`, `wiki_health`, `fleet_utilization`, `drift_score`) are not accessible through HAMR's current surface.

## Proposed Approach (research required)

Option A: **HAMR tool-proxy endpoints** — expose select `scripts/global/` tools as HAMR `/mcp` actions (e.g. `tool:consultant-checks`, `tool:rubric-score`, `tool:wiki-search`). Fleet model POSTs to HAMR; Worker executes tool on the orchestrator and returns JSON.

Option B: **Harness bundle enrichment** — extend `bundle:fetch` to return pre-computed governance artifacts (latest consultant-checks output, rubric score, wiki search index snapshot) so fleet models can consume results without tool invocation.

Option C: **Hybrid** — pre-computed bundle for read-only checks (wiki search, rubric scoring against cached trail); live proxy only for write-path tools (emit-event, consultant-feedback).

## Acceptance Criteria

- AC-R1: Research: evaluate Option A, B, C against G1 (governance completeness), G3 (zero-cost), G4 (privacy — tool output may contain issue content), G5 (portability — orchestrator must be reachable from fleet device), G6 (resilience — fleet model must degrade gracefully if HAMR tool endpoint is unavailable)
- AC-R2: Research: audit which `scripts/global/` tools are safe to proxy (pure read + GitHub API) vs. unsafe (write to local filesystem, require local secrets)
- AC-R3: Research: define the minimal HAMR surface that enables a fleet model to produce a fully compliant `role-consultant-critique` CLOSEOUT
- AC-1: Implement chosen option for `consultant-checks` + `rubric-score` tool access via HAMR
- AC-2: Implement `wiki-search` access via HAMR (can use `bundle:fetch` extension or new endpoint)
- AC-3: Add integration test: fleet model (Ollama) invoked as Consultant produces CLOSEOUT with all mandatory fields populated via HAMR-proxied tools
- AC-4: Update `closeout-schema` validator to distinguish CLOSEOUT authored by fleet model (HAMR-proxied tools) vs non-fleet model (direct local tools) — parity check, not separate standard
