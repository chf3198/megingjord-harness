# #2767 — Dangerfile ccPattern allowlist drifts from canonical conventional-commits enum

> **Source**: github:issue/2767 | **state: OPEN** | **Labels**: type:bug, status:backlog, priority:P3, area:governance, lane:code-change
> Mirror of `gh issue view 2767` (derived; edit the GitHub item, not this page).

## Body

## Problem

`Dangerfile.js` `ccPattern` = `/^(feat|fix|chore|docs|style|test|refactor|perf)(\([^)]+\))?!?: .+/` rejects PR titles using `content`, `skill`, or `hotfix` — but those three ARE in the canonical 11-type set (`scripts/global/conventional-commits-enum.js`, per CLAUDE.md github-governance). So a valid `content(research): …` title is falsely blocked by the `danger-required` gate.

## Evidence
- Surfaced on PR #2766 (Phase-0 #2292): `content(research): …` committed cleanly (commit-msg hook accepted, enum-valid) but `danger-required` failed; retitled to `docs(research):` to unblock.
- `Dangerfile.js:30` ccPattern omits `content|skill|hotfix`. `Dangerfile.js:20` branchPattern similarly omits `skill|hotfix` (branch-name-required DOES allow them — separate drift).

## Fix
Dangerfile.js should consume the canonical enum from `conventional-commits-enum.js` (single source of truth) rather than a hand-maintained inline regex. Add a golden-file test asserting parity.

G1 (gate correctness) + G10 (maintainability — DRY the type list). Incident: `~/.megingjord/incidents.jsonl` pattern_id `dangerfile-cctype-allowlist-drifts-from-canonical-enum`.
