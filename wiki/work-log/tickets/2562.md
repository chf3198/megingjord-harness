# #2562 — collaborator-gate false-fails: baton-gates passes strings, validator expects objects

> **Source**: github:issue/2562 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P1, area:scripts, lane:code-change, governance:close-without-merge
> Mirror of `gh issue view 2562` (derived; edit the GitHub item, not this page).

## Body

severity: HIGH — collaborator-gate currently false-FAILS on every PR (governance gate effectively disabled).

## Root cause
`.github/workflows/baton-gates.yml` collaborator-gate calls:
```
const result = validate({ comments: comments.map(c => c.body), labels });
```
passing an array of **strings**. But `scripts/global/megalint/collaborator-handoff.js` `findCollaboratorHandoff` does `headerRe.test(c.body || "")`, expecting **objects** with a `.body` property. So every comment becomes a string, `c.body` is undefined, the finder matches nothing, and validate returns `missing-collaborator-handoff` even when a valid COLLABORATOR_HANDOFF is present.

Introduced by #2439 (PR #2549, merged 2026-06-01) which refactored the validator to take objects without updating the workflow caller (and likely admin-gate/consultant-gate calls at baton-gates.yml lines ~85, ~128 — verify all three).

## Reproduction
```
validate({ comments: comments.map(c => c.body) })  -> ok:false missing-collaborator-handoff
validate({ comments })                              -> ok:true
```
Verified against issue #1911 live comments.

## Fix
Pass objects (`comments` / `comments.map(c => ({body:c.body, user:c.user}))`) at all three baton-gates call sites, OR make the validators accept both string and object elements. Add a regression test asserting the workflow-shaped input.

## Impact
Blocks collaborator-gate (and likely admin/consultant gates) for all PRs; PRs must currently merge via Admin baseline-drift override. Surfaced while shipping #1911.
