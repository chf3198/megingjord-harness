# #2973 — fix(fleet): cascade-dispatch escalates to premium (haiku) on availability failure instead of $0 free-cloud

> **Source**: github:issue/2973 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P2, area:scripts, resolution:completed, lane:code-change
> Mirror of `gh issue view 2973` (derived; edit the GitHub item, not this page).

## Body

## Severity: MEDIUM (G3 zero-cost + G6 resilience)

## Symptom (from run log, not yet code-confirmed)
`node scripts/global/cascade-dispatch.js "<prompt>"` with the LiteLLM gateway down and local Ollama
unusable produced:
```
[fleet] litellm gateway-unhealthy → direct Ollama fallback
[cascade] escalate→haiku: HTTP 400
```
It escalated to **haiku (premium Anthropic)** and then 400'd → no output. The direct
`free-cloud-dispatch` path (Gemini / Llama-3.3-70B via Groq/OpenRouter) works fine when called
directly (it produced the HE1/HE2 consensus ratings).

## Conflict
`cascade-dispatch.js` design comment: escalation "hard-pins availability failures to free-cloud
(NEVER premium) and steps capability failures one paid tier up." Observed behavior contradicts this:
an availability failure (gateway down + Ollama down) routed to premium.

## Likely cause
The escalation classifier in `fleet-escalation-policy.js` mis-labels "gateway-unhealthy +
fleet-unreachable" as a capability failure (→ paid tier up) rather than availability (→ free-cloud).

## Impact
- G3: reaches for a paid model when free models are available and working.
- G6: returns nothing, when the free lane would have succeeded.

## Fix direction
Classify gateway-unhealthy + fleet-unreachable as availability → call `dispatchFreeCloud` BEFORE any
premium tier. Add a regression test for this failure shape. (Relevant to HE2, whose $0 consensus
judging depends on the free lane.)
