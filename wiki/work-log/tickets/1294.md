## Body

Refs #1271

## Scope

Add rate-limit guard middleware `scripts/global/rate-limit-guard.js` with ETag cache. Centralized so C6 dependency DAG and other future GitHub API consumers share a single rate-aware client.

## Wave

Wave 3 (depth + ops infrastructure — supports C6)

## Problem

Per CP FDPR §IV + CC FDPR R3 + 2026 GitHub REST rate-limit reality (80 content-creating req/min secondary limit; semantic search 10 req/min). Bulk-mode operations risk secondary-rate-limit lockout without centralized budget.

## Acceptance criteria

- [ ] `scripts/global/rate-limit-guard.js` provides rate-aware GitHub API client wrapper
- [ ] Per-endpoint budgets: REST 80/min, semantic search 10/min, dep API low-write
- [ ] ETag cache: stores response + ETag; subsequent calls send `If-None-Match`; 304 Not Modified counts as 1 point in budget
- [ ] Backoff strategy: 503/429 triggers exponential backoff (1s → 2s → 4s → 8s); after 4 retries, falls through with error
- [ ] Adopted by C6 dependency DAG (proven first); other consumers can opt in incrementally
- [ ] Telemetry: rate-limit-budget-exhaustion events surfaced to dashboard / governance audit
- [ ] Test fixtures: under-budget (passes), at-limit (uses cache), over-limit (backs off), 304-not-modified (cache hit)
