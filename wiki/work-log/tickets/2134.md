# #2134 — Rubric scoring does not reflect G1>G2>...>G10 goal precedence order

> **Source**: github:issue/2134 | **state: CLOSED** | **Labels**: type:epic, status:done, priority:P2, area:scripts, role:manager, resolution:released
> Mirror of `gh issue view 2134` (derived; edit the GitHub item, not this page).

## Body

## Problem

The harness declares a strict goal priority order:
`G1 Governance > G2 Quality > G3 Zero Cost > ... > G10 Maintainability`

However, `scripts/global/rubric-score.js` computes a flat arithmetic mean across
all goals with equal weight:

```javascript
out.mean = scores.reduce((a, b) => a + b, 0) / scores.length;
```

A G1 (Governance) failure and a G10 (Maintainability) failure contribute
identically to the final score. A ticket with zero governance artifacts but
perfect maintainability can score the same as the inverse. This contradicts the
stated constitution and makes the rubric unsuitable as a governance gate.

`inventory/rubric-g1-g9-v2.json` likewise has no `weight`, `priority`, or
`hard_fail` fields — all goals are equal by schema construction.

## Goal

Make `rubric-score.js` and the rubric schema reflect actual G1>G2>...>G10
precedence so that the Consultant closeout score is a meaningful governance
signal.

## Acceptance Criteria (Epic-level)

- AC1: A precedence model is specified and documented (lexicographic minimum,
  priority weights, or hard-fail flag — selected via Phase-0 research child).
- AC2: `rubric-score.js` implements the chosen model; output includes a
  `precedence_verdict: pass|fail` field alongside the existing `mean`.
- AC3: `rubric-g1-g9-v2.json` (or successor) carries the precedence metadata
  required by the new scorer.
- AC4: Existing tests updated; coverage ≥ 80% for the scorer.
- AC5: A transition plan documents how historical closeout scores are
  interpreted under the new model (no retroactive re-scoring required, but
  behavior change is documented).

## Scope

**In scope**: `rubric-score.js`, `rubric-g1-g9-v2.json`, associated tests,
`consultant-checks.js` if it reads scores, scorer documentation.

**Out of scope**: Re-scoring closed historical tickets; changes to baton gate
CI checks (follow-on if needed).
