# #1827 — Epic: Collaborator-side rebase discipline + conflict prevention for long-lived branches

## Body

## Epic Summary

Optimize the git workflow contract so **Collaborators handle conflict resolution incrementally** against `main` throughout their work, **eliminating the typical Admin-side conflict-resolution burden at merge time**. Today the harness has a single one-line guidance ("Pull latest main into feature branch before creating PR") in `global-standards.instructions.md` and an advisory `git-state-drift-sensor.js` (MAX_BEHIND=5). This Epic builds the missing contract: when to rebase, when to fail, what tooling assists conflict prediction, and how the baton handoff sequence enforces it.

## v2 (2026-05-17 operator correction): velocity-relative, NOT calendar-relative

**An earlier draft used a "Graduated branch-age budget" keyed to calendar days (0-1, 2+, 4+, 7+). Operator correctly rejected this as the same anti-pattern Epic #1771 killed for soaks.** In parallel AI agent orchestration, multiple teams (Claude Code, Copilot, Codex) can produce **100× the commit + merge traffic of a human dev environment**. A "1-day-old" branch in such a system may already be 50+ commits behind main. Calendar measurements don't survive agentic velocity. This Epic now reframes around **trunk-velocity-relative** metrics.

## Problem

2026-05-17 stress test caught the live symptom: the Copilot Team's feature branch was 5 commits behind main when they ran governance checks. Investigation showed the branch base predated my #1822 merge, so their local repo source lacked the #1821 UI-scoping fix. **The 8 reported failures collapsed to one root cause: stale branch.** While #1824 added a hook-parity verifier to discriminate the diagnosis, the underlying gap is that the harness has no contract telling collaborators to rebase mid-flight at agentic velocity, nor any CI gate that fails when COLLABORATOR_HANDOFF emits from a stale branch.

Three observable failure modes today:

1. **Branch staleness at handoff** — Collaborator emits COLLABORATOR_HANDOFF when branch is N commits behind main; Admin discovers conflicts at merge time.
2. **Late discovery of file overlap** — two teams editing the same file via parallel branches don't surface the collision until PR open; #1608 partially addresses this but is still backlog.
3. **No conflict-resolution practice for collaborators** — git workflow learning happens reactively when merges fail, not proactively as a Collaborator-phase deliverable.

## Frontier 2026 research (embedded so this Epic is pick-uppable cold)

### F1 — Trunk-based development is the high-performance baseline

[Java Code Geeks 2026 trunk-based development guide](https://www.javacodegeeks.com/2026/02/trunk-based-development-the-git-strategy-powering-elite-engineering-teams.html) and [Atlassian](https://www.atlassian.com/continuous-delivery/continuous-integration/trunk-based-development): high-performing teams merge to main **at least once per day** in HUMAN-scale teams. State of DevOps Report: TBD-adopting teams **deploy 208× more frequently**. "The branch never survives the working day."

**Apply (agentic adaptation)**: in multi-orchestrator parallel work, "the working day" is meaningless — the relevant time unit is **commits-on-main per hour**. A branch should never survive more than N commits-of-trunk-drift without rebase. N is a velocity-relative threshold (see F8 below).

### F2 — Long-lived branches mathematically guarantee conflicts

[MinimumCD anti-patterns](https://beyond.minimumcd.org/docs/anti-patterns/branching-integration/long-lived-feature-branches/), [Sparkbox](https://sparkbox.com/foundry/how_to_not_dread_rebases_when_managing_long_lived_feature_branches): two developers editing the same file on different branches **will not discover the collision until merge**. A developer rebasing a three-week branch may introduce bugs because three-week-old assumptions are stale.

**Apply (agentic adaptation)**: "Long-lived" is measured in commits-behind, not weeks. A branch can become "long-lived" in 30 minutes if trunk takes 100 commits in that window.

### F3 — Rebase frequency = inverse conflict cost

[GitHub community discussion #161932](https://github.com/orgs/community/discussions/161932): "the simplest answer to minimize conflicts is to rebase the feature branch frequently onto master." Small updates = fewer conflicts.

**Apply (agentic adaptation)**: rebase frequency should be PROPORTIONAL to trunk velocity, not constant. High-velocity windows demand pre-push rebase; low-velocity windows tolerate longer intervals.

### F4 — AI-powered conflict prevention (2026)

[GitKraken Team View](https://www.gitkraken.com/features/merge-conflict-resolution-tool) flags overlapping file edits **before PR open**. [Graphite AI merge resolution](https://www.graphite.com/guides/ai-code-merge-conflict-resolution) provides first-pass conflict resolutions with explanations. [GitWand](https://gitwand.devlint.fr/) is a native git client with AI conflict resolution.

**Apply**: cross-PR file-overlap detection at every push and before COLLABORATOR_HANDOFF. Composes with #1608.

### F5 — Merge queue with automated rebase

[Graphite merge queue](https://graphite.com/docs/merge-pull-requests) automates rebase management ensuring "green trunk." [GitLab merge trains](https://docs.gitlab.com/ci/pipelines/merge_trains/) provide similar serialization.

**Apply (agentic adaptation)**: merge queues are **especially valuable** at agentic velocity because they serialize the high-traffic flow into main. Evaluate adoption as Phase 3 deliverable. Without a queue, Collaborator-side rebase contract is the floor.

### F6 — Branch protection requires up-to-date with main

[Alloy April 2026 guide](https://alloy.app/library/how-to-avoid-merge-conflicts-in-deployment) and GitHub branch protection: require branches up-to-date with main **before merging**, forcing conflict resolution before PR approval.

**Apply**: extend `git-state-drift-sensor.js` to emit a HARD failure (not just advisory) when threshold exceeded AND the operation is COLLABORATOR_HANDOFF emission.

### F7 — Git worktrees enable parallel safe work

[Nx blog 2026](https://nx.dev/blog/git-worktrees-ai-agents) and [Medium 2026](https://medium.com/@raminmammadzada/solving-parallel-workflow-conflicts-between-ai-agents-and-developers-in-shared-codebases-286504422125): create branch in a worktree, spin up AI agent per-worktree.

**Apply**: codify the existing harness worktree pattern (`research/concurrent-agent-worktrees-2026-04-24.md`) as required for multi-commit Collaborator work.

### F8 — Velocity-relative staleness is the agentic-native metric

**Operator-supplied finding (2026-05-17)** — the prior draft of this Epic used calendar-day thresholds (0-1, 2+, 4+, 7+ days). Operator correctly identified this as the same anti-pattern Epic #1771 killed for "soaks." In agentic orchestration with N parallel teams committing at machine speed, a 1-hour-old branch can be 100+ commits behind main. **Calendar measurements don't survive agentic velocity.**

The correct metric family:

1. **Absolute behind-count**: `behind_commits = origin/main HEAD - merge-base(branch, origin/main)`. Cheap to compute via `git rev-list`.
2. **Trunk-velocity-per-hour**: `velocity = commits_to_main_in_last_window / window_hours`. Computed from `git log origin/main --since=...`.
3. **Velocity-normalized staleness**: `effective_drift = behind_commits / max(velocity, 1)`. Captures "how many hours worth of trunk activity am I behind?"
4. **Behind:branch ratio**: `ratio = behind_commits / max(commits_on_my_branch, 1)`. Captures "has trunk moved disproportionately during my branch's lifetime?"

Initial thresholds (subject to Phase 2 calibration via replay-eval):

| Tier | Behind (abs) | Velocity-normalized | Ratio | Action |
|---|---|---|---|---|
| ok | ≤ 3 | ≤ 1.0 hr-of-trunk | ≤ 1:1 | None |
| advisory | 4 – 10 | 1.0 – 3.0 hr | 1:1 – 3:1 | Recommend rebase before next push |
| pre-handoff-block | 11 – 30 | 3.0 – 8.0 hr | 3:1 – 10:1 | COLLABORATOR_HANDOFF blocked; rebase required |
| re-scope | > 30 | > 8.0 hr | > 10:1 | Ticket flips to Manager re-scope review |

The harness's existing `GIT_DRIFT_MAX_BEHIND=5` is the floor of the "advisory" tier — already velocity-aware-ish, just not promoted to a Collaborator-handoff gate.

**Calibration is replay-eval-driven, not chosen a priori**, per Epic #1771 lesson. Phase 5 replays against last 50 merged PRs to find thresholds that maximize precision (real conflicts caught) and recall (false positives minimized).

## Proposed contract (subject to Phase 2 design)

A new `instructions/collaborator-rebase-discipline.instructions.md` contract specifying:

1. **Velocity-relative staleness sampler** (per F8): primary metric is `behind_commits` + `trunk_velocity`. Calendar-day thresholds NOT used.

2. **Pre-handoff gate**: `npm run git:freshness-check` runs as part of pre-push lefthook. Evaluates against the 4-tier table from F8. Fails (exit 1) at `pre-handoff-block`; fails-loud at `re-scope`.

3. **Pre-PR conflict prediction**: `npm run git:conflict-predict` scans every other open PR for overlapping files (Phase 3 implementation). Composes with #1608.

4. **COLLABORATOR_HANDOFF schema extension**: add required fields `behind_at_handoff: <int>` and `rebase_freshness: <ISO8601>` per #1745 audit-trail extension pattern. Closeout-schema validator emits advisory if absent.

5. **Admin-handoff contract**: Admin MAY refuse merge if branch crosses pre-handoff-block AFTER initial Collaborator rebase due to subsequent trunk traffic (race condition that's especially common at agentic velocity). Note `BLOCKER_NOTE: pre-merge-rebase-required`; Collaborator re-claims, rebases, re-emits.

6. **Adaptive cadence at agentic velocity**: when trunk velocity > 10 commits/hour, pre-push rebase is mandatory on every push (not just before handoff). Sampler emits this signal.
