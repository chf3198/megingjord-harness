# #2552 — Make worktree merge-detection squash-merge-aware (cleanup blind spot)

> **Source**: github:issue/2552 | **state: CLOSED** | **Labels**: type:task, status:done, priority:P1, area:governance, lane:code-change, anneal:tier-2
> Mirror of `gh issue view 2552` (derived; edit the GitHub item, not this page).

## Body

## Problem (Tier-2 self-anneal — recurring stale worktree accumulation)

`scripts/global/worktree-inventory.js` determines merged status via ancestry:

```
mergedToMain = git merge-base --is-ancestor <branch.head> origin/main
```

This repository **squash-merges every PR**. A squash merge creates a *new* commit
on `main`; the original branch HEAD is therefore **never** an ancestor of
`origin/main`. Structural consequences:

- `mergedToMain` is permanently `false` for every squash-merged branch.
- Because `worktree-cleanup-plan.js` `classify()` checks `mainAhead > 0 →
  quarantine` **before** it checks `mergedToMain → remove`, the squash-merged
  entry is trapped in `quarantine` regardless of `mergedToMain`'s value.
- Net effect: a genuinely merged, ticket-CLOSED worktree is permanently
  misclassified as `quarantine` and can never be recommended for removal.

## Evidence

On 2026-06-01 an audit found 6 stale worktrees. All four squash-merged ticket
worktrees (#2437/#2438/#2439/#2541) were classified `quarantine` with
`removalSafe:false`, despite their PRs being merged and (for #2541) the ticket
closed. Manual verification via `gh pr list --head <branch> --state merged`
confirmed all four were merged. Operator removed them by hand — exactly the toil
Epic #2248 set out to eliminate.

**Cost of current bug**: 4 stale worktrees × ~10 min manual removal per audit =
~40 min operator toil per audit cycle. At current audit frequency (weekly), this
accumulates to ~35 hours/year of wasted agent and operator time with zero
decision value.

## Root cause (two-part — both must be fixed)

**Part 1 — Detection gap**: `worktree-inventory.js` `mergedToMain()` uses
ancestry-based detection (`git merge-base --is-ancestor`) which structurally
cannot detect squash merges. It always returns `false` for squash-merged branches.

**Part 2 — Guard ordering bug**: `worktree-cleanup-plan.js` `classify()` checks
`aheadOfMain > 0 → quarantine` **before** `mergedToMain → remove` (lines 27 vs
29 in the current code). Even if the inventory correctly set `mergedToMain: true`
for a squash-merged branch, `classify()` would still return `quarantine` because
squash-merged branches always have `mainAhead > 0` (the original pre-squash
commit is not an ancestor of main). Both defects interact; fixing only one is
insufficient.

## Proposed solution

**Part A — squash detection in `worktree-inventory.js`:**
Extend `mergedToMain()` to also probe `gh pr list --head <branch> --state merged
--json number`. If the probe returns ≥1 result, the branch is squash-merged and
`mergedToMain` is set to `true` (only when the worktree is also clean). Probe
result is cached in-memory per branch within a single invocation (one `gh` call
per distinct branch). If `gh` is unavailable or errors, fall back to `false`
(safe/conservative — no false positives for removal).

**Part B — guard reordering in `worktree-cleanup-plan.js`:**
Move `if (entry.mergedToMain) return 'remove';` to **before**
`if (aheadOfMain > 0) return 'quarantine';` in `classify()`. The dirty/untracked
check remains first so dirty squash-merged worktrees still quarantine.

  now carries an explicit pass gate command so verification is unambiguous.
- **G3 Zero Cost**: Primary detection path uses `gh` CLI — a free tool that
  consumes zero LLM tokens. The fix eliminates recurring manual operator toil:
  the 2026-06-01 audit found 4 squash-merged worktrees requiring manual removal
  (~40 min of wasted operator/agent time per audit run; ~35 hours/year at weekly
  cadence). With this fix, automated cleanup reclaims those directories without
  any paid-provider calls. Offline fallback also uses zero tokens and zero
  external resources. Every worktree audit cycle previously blocked on manual
  intervention can now run unattended.
- **G5 Portability**: Solution uses only POSIX `git` commands and the `gh` CLI,
  available across all supported runtimes (Copilot, Codex, Claude Code). No
  platform-specific paths, no environment-specific dependencies. The `runGh`
  injection pattern allows the same module to run in CI (real `gh`) or tests
  (mock) without code changes.
