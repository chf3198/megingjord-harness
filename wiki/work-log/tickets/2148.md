## Body

## Goal

Every change merged to `main` (and thus published to the GitHub repo) MUST be meticulously documented across four parallel surfaces, with that documentation **enforced by hard gates** (CI / pre-merge required-checks / branch-protection rules), not by instruction-based hope-it-happens. The system must operate reliably with **non-frontier fleet models** that cannot be trusted to follow nuanced governance instructions — so the gates must be evidence-checkable, model-agnostic, and binary pass/fail.

## The four documentation surfaces

1. **GitHub Issues** — the canonical log of all work. Every merge to main must trace to at least one issue (`Closes #N` or `Refs #N`) and that issue must carry a complete baton trail (Manager scope, Collaborator handoff, Admin handoff, Consultant closeout).
2. **Release notes** — every shipped change appears in `.changes/unreleased/<N>.md` (per #1132 fragment pattern) → aggregated to `CHANGELOG.md` at release-cut.
3. **Project documentation** — README, design docs, dev guide, user guide. Behavior-changing PRs MUST update the relevant doc surface or explicitly mark N/A with justification.
4. **Wikis** — per the three-wiki typology (#1942 / #1943): code-base wiki, work-log wiki, wisdom wiki (project + global). Work that touches code patterns, governance, fleet routing, or new infra MUST update the relevant wiki page.

## Fleet-model compatibility constraint (load-bearing)

Frontier models can follow `instructions/*.md` advisory text. Fleet models (qwen2.5-coder, gemma3, Ollama variants) cannot reliably parse multi-page nuanced instructions. The governance MUST therefore be:

- **Schema-checkable**: validators run independently of agent reasoning
- **Evidence-based**: each surface has a file artifact the gate inspects
- **Binary**: pass/fail with clear required-action remediation message
- **Documented in operator-facing runbooks**, not buried in instruction prose

## Existing harness surfaces to integrate (NOT replace)

- `closeout-schema` workflow + `evidence-completeness` validator (per-baton-step evidence)
- `docs-drift-maintenance` skill (post-merge advisory)
- `repo-profile-governance` skill (community-health files)
- `release-version-integrity` skill (tag/manifest/changelog alignment)
- `.changes/unreleased/` per-PR fragment + `scripts/global/changelog-aggregate.js` (#1132, #2120 refit)
- `wiki/wisdom/project/` + `wiki/work-log/` + `wiki/code/` (Three-Wiki typology #1942)

The new gate set composes these into a **merge-time required-check union** that fails any PR missing evidence for any surface where it should apply.

## Operator clarification (2026-05-25 round 2)

This Epic is explicitly framed as **adding/hardening a guardrailed maintenance task to the Collaborator role** — a Technical Writer sub-capability that activates during the Collaborator phase and ensures EVERY human-readable artifact (logs, docs, READMEs, guides, profiles, wikis) is updated to reflect the work in the current ticket. The 4-role baton invariant (Manager → Collaborator → Admin → Consultant) is preserved; the Tech Writer specialization lives WITHIN the Collaborator phase as a mandatory sub-step before COLLABORATOR_HANDOFF.

### Tech-Writer-Collaborator sub-phase contract

```
Collaborator phase begins
  │
  ▼
1. IMPLEMENT scoped changes (existing Collaborator deliverable)
  │
  ▼
2. TECH-WRITER sub-phase (NEW — this Epic adds):
   a. Query doc-link graph: which docs reference the changed symbols?
   b. Update each affected doc OR mark explicit N/A + justification
   c. Pre-handoff verification: doc-graph lint clean
  │
  ▼
3. COLLABORATOR_HANDOFF (existing) — now includes per-doc-surface evidence block
```

### Doc-link graph (efficiency mechanism per operator request)

A **knowledge graph** linking code symbols / concepts / config keys / commands / etc. (nodes) to the doc files that reference them (edges) — so the Tech-Writer-Collaborator can quickly identify ALL revision targets per change without manual scanning. Two architecture candidates surveyed in research round-2 on #2149:

| Approach | Source | Fit |
|---|---|---|
| **Graphify-pattern** (full knowledge graph + MCP query interface) | knightli.com 2026-05 | Strong fit; confidence-tagged edges (EXTRACTED/INFERRED/AMBIGUOUS) preserve G8 auditability |
| **Red Hat Code-to-Docs** (semantic-index, NOT graph) | developers.redhat.com 2026-04 | Rejected — black-box LLM summaries violate G1 auditability |
| **Obsidian-style bidirectional wikilinks** (file-based) | Karpathy LLM Wiki + Obsidian/Logseq/Roam | Strong fit; already present in `wiki/`, can extend to `docs/` + `instructions/` + `README.md` |
| **Hybrid (recommended)** | Graphify-style code-to-doc edges + Obsidian-style wikilinks | Best fit — minimal greenfield, G3 zero-cost, G5 portable, file-based no external DB |
