## Tier-2 self-anneal: human-calendar cadence in agentic governance

### Pattern
The operator (and harness instructions) repeatedly default to **human-team
calendar cadence** when designing timers, crons, SLAs, dormancy windows,
reminder schedules, and time-forecasts — intervals measured in **days** that
are mis-scaled by 1–2 orders of magnitude against agentic throughput.

### Recurrence evidence (≥2 in 7d → Tier-2)
- **2026-06-13 (this session):** designed/encoded day-scale cadences as routine:
  - in-flight child **#2920** = "Add GitHub Actions cron for Epic dormant **90-day**
    review reminders" — the AC premise itself is human-calendar.
  - prose framing: "**7-day** dormancy", "**24h** ready-SLA", "spans many sessions".
- **Prior anneals (#1771 / #1827 / #1809):** fixed *measurement-window* language
  ("N-day soak" → replay-eval) and established velocity-relative metrics — but
  scoped to **measurement/eval windows only**. They did NOT cover **operational
  cadence** (cron intervals, SLA/reminder timers, dormancy/auto-pause windows,
  operator wall-clock forecasting). That gap is why the pattern recurs.
- Operator memory `feedback_calendar_thresholds_in_agentic_systems` + Client
  surfaced this directly again on 2026-06-13.

### Why it's wrong (evidence)
Agentic dev cycles are 1–2 orders of magnitude faster than human cycles:
- **METR Time Horizon 1.1 (Jan 2026):** AI-agent 50%-reliability task length
  doubling **~4.3 months** (post-2023); agents now do hour-scale tasks trending
  to day/week-scale.
- Concrete compression: agent completes in **<45 min** what takes a human a full
  workday; prototypes a service in **20–30 min**; full Docker test bench in
  **8–12 h**; end-to-end dev+test+deploy in **6–7 days** vs months (~10–30×).
- Agentic operation is **continuous / event-driven** (no work-hours, no fatigue);
  much "24/7" is event-triggered (minutes per event), so **event-driven beats any
  polling timer**.

Consequence in this harness: a **90-day** dormancy review never fires within a
meaningful agentic Epic lifecycle (the fleet can finish an Epic in hours–days);
a **7-day** "stale" window is, in agentic terms, an eternity; a **24h** ready-SLA
is ~100× looser than fleet pickup latency.

### Root cause
`stale/incomplete guardrail` — prior calendar-threshold guardrail was scoped to
*measurement windows*, leaving *operational cadence + time-forecasting* uncovered.

### Proposed guardrail (docs/process only — implementation via normal baton)
A **cadence-selection rule** in harness instructions:

1. **Default order for any new timer/cron/SLA/reminder/threshold:**
   `event-driven  >  velocity-relative (commits-behind, idle-relative-to-recent-throughput)  >  wall-clock-in-hours (justified)  >>  wall-clock-in-days+ (essentially forbidden)`.
2. **Day-scale-or-longer wall-clock cadence is forbidden by default.** It is
   permitted ONLY when bound to a genuine **external human/system constraint**
   (third-party beta window, human UAT turnaround, legal/compliance/billing
   cycle) — and that constraint must be named inline.
3. **Operator time-forecasting in prose** ("spans days", "N-day soak", "weekly")
   must translate to agentic units (hours, or event/velocity terms) unless tied
   to an external constraint.
4. Cross-link existing `feedback_calendar_thresholds_in_agentic_systems`,
   Epic #1771/#1827, and the soak-to-replay translation guide — extending their
   scope from *measurement* to *operational cadence*.

Candidate target files (Manager to confirm at scoping):
`instructions/global-standards.instructions.md` (or a focused
`instructions/agentic-cadence.instructions.md`), plus a one-line goal-lens hook;
update the existing operator memory; add a `drift-lint`/doc-lint check that flags
`\b\d+\s*-?\s*(day|days|week|weeks|month|months|d)\b` in new cron/SLA/timer
definitions absent an `external-constraint:` justification.
