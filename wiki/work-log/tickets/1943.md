## Three-Wiki typology (operator-defined 2026-05-19)

### Wiki A — Code-Base Wiki (per-project)

Holds all code-base details. Failure mode addressed:

> "Models often need to search the code base via lengthy CLI research to find dispersed references in order to deal with prompts that ask questions of the project (code base). Having an optimally built Wiki that holds all details of the code base would be a much more efficient and potentially honest answer source to questions and prompts that need high to low level information on features, dependencies, known issues, and all other information that would be found in a developer's guide or a set of historical release documents."

Tokens spent per model query on dispersed-reference reconstruction is the G3 cost vector this Wiki type closes.

### Wiki B — Project Work-Log Wiki (per-project)

> "The third consideration for another Wiki was to provide a local store (log) of all work done — mirroring that found in the GitHub tickets."

Local mirror of GitHub ticket state. Reduces `gh issue view` / `gh pr view` round-trip cost (G3) and enables outage-time queries (G7).

### Wiki C — Research-Based-Wisdom Wiki (dual-scope)

> "The original Karpathy LLM Wiki of the harness was an attempt to build a local collection of research-based wisdom."

Dual scope per operator:
- **Global Wisdom Wiki**: general knowledge helpful to ALL projects (existing `wiki/concepts/`, `wiki/entities/`, `wiki/syntheses/`)
- **Project-specific Wisdom Wiki**: research knowledge specific to a particular project

Operator quote: *"the research-based-wisdom Wiki would need both a globally available Wiki with general knowledge helpful to all projects, and a project specific research-based-wisdom Wiki when knowledge is deemed specific to a particular project."*

## Operator capture (full discussion 2026-05-19)

> "The idea of the Karpathy LLM Wiki is to increase the efficiency of the harness towards its goals, particularly G3 and G7. We had design discussions on multiple uses of the Karpathy LLM Wiki design for several different informational scopes that the fleet's models often need. For example the models often need to search the code base via lengthy CLI research to find dispersed references in order to deal with prompts that ask questions of the project (code base). Having an optimally built Wiki that holds all details of the code base would be a much more efficient and potentially honest answer source to questions and prompts that need high to low level information on features, dependencies, known issues, and all other information that would be found in a developer's guide or a set of historical release documents. The original Karpathy LLM Wiki of the harness was an attempt to build a local collection of research-based wisdom. The third consideration for another Wiki was to provide a local store (log) of all work done — mirroring that found in the GitHub tickets. Lastly, while the code-base Wiki and the project work-log Wiki are obviously specific to individual projects, the research-based-wisdom Wiki would need both a globally available Wiki with general knowledge helpful to all projects, and a project specific research-based-wisdom Wiki when knowledge is deemed specific to a particular project."

This quote is the authoritative typology source. Implementation children must preserve the dual-scope contract on Wiki C.
