# #2967 — Enforce one-ticket Agile role execution per worktree

> **Source**: github:issue/2967 | **state: OPEN** | **Labels**: type:task, priority:P1, area:infra, role:manager, status:triage, lane:code-change
> Mirror of `gh issue view 2967` (derived; edit the GitHub item, not this page).

## Body

## Objective
Add deterministic guardrails that block running Manager/Collaborator/Admin/Consultant baton artifacts for multiple tickets from the same active worktree execution flow, with provider-neutral behavior across Claude Code, Copilot, Codex, and Antigravity runtimes.

## Problem
A recent closeout sequence processed multiple tickets in one pass from the same worktree context instead of iterating full Agile role workflow per ticket. This creates governance drift risk in Tier B++ mode and can diverge across ADK runtimes unless the guardrail is runtime-neutral.

## Acceptance Criteria
- Pretool guard fails when a second ticket is targeted in the same worktree while an active ticket has unresolved baton state.
- Guardrail enforces one-ticket-per-worktree for Agile role artifacts (MANAGER_HANDOFF, COLLABORATOR_HANDOFF, ADMIN_HANDOFF, CONSULTANT_CLOSEOUT).
- Guardrail logic is provider-neutral and does not rely on runtime-specific aliases; it works consistently for Claude Code, Copilot, Codex, and Antigravity adapters.
- Clear remediation guidance is emitted: finish or explicitly handoff/close current ticket before switching.
- Regression tests cover allowed and denied transitions, including false-positive prevention.
- Documentation updated for operator workflow and failure recovery path, including the cross-ADK compatibility contract.

## Verification
- Unit tests for guard logic and state transitions.
- Hook-level integration test proving deny path when switching ticket mid-baton.
- Positive test proving sequential per-ticket iteration still works.
- Compatibility matrix assertions covering Claude Code, Copilot, Codex, and Antigravity execution surfaces.

## Notes
This is a self-annealing follow-on to strengthen governance controls after observed multi-ticket one-pass handling behavior.
