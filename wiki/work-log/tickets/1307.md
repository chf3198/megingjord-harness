# #1307 — ADR-010 label-scan: false-positive role:collaborator required on Epics in-progress

> **Source**: github:issue/1307 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P1, area:governance, resolution:completed, lane:code-change
> Mirror of `gh issue view 1307` (derived; edit the GitHub item, not this page).

## Body

## Problem

`.github/workflows/label-scan.yml` (the daily ADR-010 audit workflow) declares this rule:

```js
const STATUS_ROLE_REQUIRED = {
  'status:in-progress': 'role:collaborator',
  'status:testing':     'role:admin',
  'status:review':      'role:consultant',
};
```

…and applies it indiscriminately to all issues including Epics. This **contradicts Epic Rule E3** in `instructions/epic-governance.instructions.md`, which mandates:

> Epic always carries `role:manager` — never changes. Active agent role lives on the child ticket, not the epic.

Result: every `status:in-progress` Epic gets a false-positive `<!-- adr-010-label-scan -->` comment posted by the daily cron. Live evidence (2026-05-10 cron run posted to all four):

- Epic #1245 (Harden git governance) — `status:in-progress`, has `role:manager`, **incorrectly flagged**
- Epic #1133 (Self-annealing) — same shape, **incorrectly flagged**
- Epic #1130 (HAMR universal coverage) — same shape, **incorrectly flagged**
- Epic #1113 (Multi-layer self-annealing) — same shape, **incorrectly flagged**

The same false-positive class will fire on Epics in `status:testing` (would require `role:admin`) and `status:review` (would require `role:consultant`) even though Epic Rule E3 forbids those role labels on Epics.

## Impact

- **Comment-stream pollution**: every active Epic accrues false-positive lint comments daily, drowning out real Manager-progress comments.
- **Operator confusion**: comments appear authoritative ("ADR-010 Label Scan Violation") but contradict the canonical rules.
- **Trust erosion in lint signals**: when audit cries wolf, real violations are easier to ignore.
- **Conflicts with `label-lint.yml`**: the per-event lint correctly exempts Epics; the daily scan does not. Two governance gates disagreeing is itself drift.

## Acceptance Criteria

- [ ] **AC1 — Skip Epics in role-required check.** When evaluating `STATUS_ROLE_REQUIRED`, skip issues where `labels` contains `type:epic`. Epics validate via the separate Epic-invariant check (which also belongs in this scan: must carry exactly `role:manager`, no other role).
- [ ] **AC2 — Add Epic-invariant rule.** New rule in `label-scan.yml`: if `type:epic` is present, exactly one `role:*` label must be present and it must be `role:manager`. Anything else is a violation.
- [ ] **AC3 — Reconciliation comment cleanup.** Add a step that detects existing `<!-- adr-010-label-scan -->` comments on issues that are no longer violating, and either deletes them or appends a `RESOLVED` line. Prevents ghost comments lingering after the rule fix.
- [ ] **AC4 — Parity with `label-lint.yml`.** Both workflows reference a single shared rule set (extract to `scripts/global/label-rules.js` or similar). Two workflows must not disagree on what constitutes a violation.
- [ ] **AC5 — Golden-file regression.** Fixtures under `tests/fixtures/label-scan/`: real `status:in-progress` Epic with `role:manager` (must NOT flag); non-Epic `status:in-progress` missing `role:collaborator` (must flag); Epic with `role:collaborator` (must flag — Epic invariant violation).
- [ ] **AC6 — Live verification.** After fix, manual `workflow_dispatch` re-run produces zero violations on Epics #1245/#1133/#1130/#1113. Pre-existing false-positive comments cleaned up per AC3.
