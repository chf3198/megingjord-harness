## Body

## Goal

Establish governance + observability for cross-team fleet resource utilization so AI agent teams (Claude Code, Codex, Copilot, Antigravity) (a) are patient with slow fleet resources rather than timing out and falling back, (b) actively utilize ALL available fleet resources rather than fixating on one node, and (c) have shared visibility into which team is using which resource at any time.

## Motivating evidence (today's session, 2026-05-31)

PR #2516 / #2515 iter-2 dispatch to qwen-32b on 36gbwinresource (priority-100 fleet node) timed out at 1200s (20 min). Operator fell back to qwen-7b on the same node. Two issues:

1. **Patience failure**: 1200s is short for qwen-32b on a 4GB-VRAM GPU with CPU offload (~19GB model into 28GB RAM after VRAM exhaustion). A 30-60 min wait would have been honest. The 1200s timeout was arbitrary, not informed by the model's known latency profile.
2. **Resource map blindness**: Operator did NOT consider routing to OpenClaw Host (windows-laptop, 16GB CPU-inference at priority 40) for the fallback. Operator did NOT consider routing to a paid Copilot-Pro free-tier model (claude-haiku-4.5, gpt-5-mini, mult=0). Fallback was qwen-7b on the SAME host, missing the cross-resource fanout opportunity.
3. **Cross-team blindness**: Operator did NOT know whether another team (Codex, Copilot, Antigravity) was concurrently holding the 32b model on a different ticket, causing the slow response.

## Proposed solution areas

### 1. Patience policy (per-resource latency profile)

- inventory/fleet-latency-profile.json: per-model × per-host expected latency (p50, p99) at standard prompt sizes
- scripts/global/fleet-dispatch-with-patience.js: wraps Ollama dispatch with backoff aware of model latency profile
- Default timeouts derived from p99 × 2 instead of arbitrary 1200s
- Per-model exponential backoff when model is loaded but slow

### 2. Resource utilization map (route to best-fit, not default)

- scripts/global/fleet-router.js: given task class (search, refactor, critique, etc.), routes to best available host
- Consults devices.json + ai-models.json + per-host current-load (see #3)
- Respects priority ordering but considers contention
- Cost-aware: prefer mult=0 cloud models OR fleet over mult>0 paid models per G3

### 3. Cross-team fleet observability

- New HAMR endpoint: GET /fleet/in-flight → { host, model, holding_team, ticket, started_at, eta_seconds }
- Each team's fleet dispatch announces via HAMR /fleet/claim/{host}/{model} BEFORE dispatch
- Other teams query /fleet/in-flight before dispatching same host/model → queue or pick alternative
- Append to baton-events.jsonl per Epic #2451 Move 2 schema v3 for audit

### 4. Operator-visible status panel

- Dashboard panel showing live /fleet/in-flight state
- Each team's session shows "qwen-32b on 36gbwinresource HELD by codex on #N, ETA 3min" in stderr or session log
