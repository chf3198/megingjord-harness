# #2587 — Scope pretool_guard no-active-ticket deny to in-repo paths only

## Body

Refs Epic #2585

## Problem (Finding 2 — over-broad block)
The no-active-ticket deny in `pretool_guard.py` fires for **every** Write/Edit tool, regardless of where the target path lives. The ticket-gate is meant to protect **repository code**, but it also blocks writes to paths **outside any repo** — e.g. the operator memory store at `~/.claude/projects/.../memory/`. Evidence: after clearing `active_ticket` for hygiene at #2564 close, saving a memory note was denied "no active ticket," requiring a re-set workaround. Memory: `feedback_worktree_edit_ticket_gate_reads_main_state` (corollary).

## Remediation (runtime-neutral, strictly-additive)
Gate-scope narrowing: apply the no-active-ticket deny **only** when the edited path resolves inside a governed repository working tree.
1. Read edited path(s) via `admin_patterns.iter_paths(tool_input)` (no hardcoded keys; all 4 runtimes).
2. Classify each path: is it inside a governed checkout (`~/devenv-ops` or `~/devenv-ops-*`, via `git -C <dir> rev-parse --show-toplevel` matched against the governed-root pattern)?
3. If ALL edited paths are **outside** any governed repo → the ticket-gate does not apply → allow.
4. If any path is inside a governed repo → existing ticket-gate behavior is unchanged.
On any resolution failure → fall back to current (deny-on-no-ticket) behavior, i.e. fail safe toward the status quo.

Coordinate with Finding 1 (same function): both reuse one shared "resolve governed worktree for path" helper. Order: scope-check (Finding 2) gates whether the ticket-check runs at all; path-derived resolution (Finding 1) supplies the ticket when it does.

## Cross-SDK compatibility analysis
Same single shared file across Claude/Copilot/Codex/Antigravity; path reads via `iter_paths` (key-normalized); no `DEVENT_OPS_RUNTIME` branching; no new payload fields. **Strictly-additive** — only *removes* denials for non-repo paths; never changes any decision for in-repo paths, so no runtime regresses and repo governance is untouched.
