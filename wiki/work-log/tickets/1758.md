# #1758 — Governance: configurable agent authorization profile with default owner-mode

## Body

## Problem
Agent behavior can still drift toward permission-seeking or partial-environment assumptions even when the harness intent is full operator ownership.

## Goal
Make agent authorization level an explicit, configurable governance contract, with **default owner-mode**: agents treat the full environment as their own (root/admin capable across local hardware, software, and approved remote services), and execute tasks directly unless a stricter profile is selected.

## Default policy (required)
- Default profile = `owner`
- `owner` means: full operational authority over local system + approved remote services, including installing tools, upgrading dependencies, running privileged commands, and performing infrastructure actions consistent with governance rules.
- Agents should not frame missing tool/version as a blocker when install/upgrade is permissible under active profile.

## Acceptance Criteria
- [x] AC1: Add a canonical authorization-profile spec (e.g., `owner`, `guarded`, `restricted`) with explicit capability matrix and override semantics.
- [x] AC2: Wire the active profile into runtime instructions/hooks so agent prompts consistently receive the effective authority contract.
- [x] AC3: Ensure `owner` profile is default in harness startup path unless explicitly overridden by operator policy.
- [x] AC4: Add conformance checks/tests proving agent behavior differences by profile (install/upgrade allowed in `owner`, constrained in stricter modes).
- [x] AC5: Update operator and team guidance with examples of expected behavior under each profile.
- [x] AC6: Add observability evidence (logs/telemetry) showing which profile was active for each governed execution.
