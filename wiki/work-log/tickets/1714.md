# #1714 — Self-anneal: formalize multi-Close PR batching governance contract

## Body

## Symptom

In the 2026-05-16 session, 7 batched PRs each closed 2-5 tickets with multi-Close lines. Each batch had FULL baton evidence on only the LEAD ticket; sibling tickets received brief close-evidence comments referencing the lead. The `closeout-schema` validator only fires on the issue extracted from the branch name, so sibling tickets close via PR auto-close without full validator pass.

Codex Team (gpt-5.5) called this "governance-laundering where sibling tickets closed with only brief evidence and no full baton, because the claimed protocol is ticket-scoped, not lead-ticket-scoped."

## Recurrence evidence

- PR #1690 closed #1670 + #1671 (#1671 brief evidence only)
- PR #1702 closed #1641 + #1642 + #1643 (#1642 #1643 brief evidence only)
- PR #1703 closed #1655 + #1656 + #1657 + #1658 + #1659 (4 brief)
- PR #1704 closed #1647 + #1648 + #1649 + #1650 + #1651 (4 brief)
- PR #1705 closed #1662 + #1663 + #1664 + #1665 + #1666 (4 brief)

Plus historical precedent (PR #1599 multi-closed multiple tickets without controversy).

## Pattern class

The harness has multi-Close PRs **in practice** but no **formal contract** defining when batching is governance-compliant vs governance-laundering. Codex's critique is valid in spirit: full validator pass is per-issue, not per-PR.

## Acceptance Criteria

- [ ] AC1: Decide policy: either (a) allow batching with formal rules, or (b) require per-issue baton dance even when batched.
- [ ] AC2: If (a): add a Multi-Close Batching section to `instructions/role-baton-routing.instructions.md` defining: when batching is allowed (related ACs, single diff, single test surface); required content of sibling brief-evidence comments (must include Signed-by, Team&Model, Role, verification-timestamp, rubric_rating fields); validator extension to recognize "resolved as part of batch with #LEAD" as valid closeout evidence.
- [ ] AC3: If (b): require each closing-issue to have its own MANAGER_HANDOFF + CONSULTANT_CLOSEOUT structured fields. Add per-issue closeout-schema fire.
- [ ] AC4: Add unit test for whichever path is chosen.
- [ ] AC5: Update `[[feedback-all-baton-artifacts-before-pr]]` memory to reflect the formalized contract.

## test_strategy

## Out of scope

## Grandfathering (explicit)

**Historic batches are GRANDFATHERED regardless of which option (a) or (b) lands.** Specifically: the 7 batched PRs from the 2026-05-16 session (PRs #1690, #1702, #1703, #1704, #1705, #1706, #1707 — multi-Close PRs that closed siblings without per-issue baton) are accepted as historical record. The chosen policy applies only to PRs opened AFTER the option-decision lands. This grandfathering is non-negotiable to avoid creating retroactive scope-trimming pressure.
