# #2706 — Pre-flight guard + auto-Tier-1 bridge for admin-override bypass exception

> **Source**: github:issue/2706 | **state: CLOSED** | **Labels**: type:task, status:done, priority:P2, area:hooks, area:governance, lane:code-change, governance:close-without-merge, phase-gate:phase-1, merge-bypass:admin-exception
> Mirror of `gh issue view 2706` (derived; edit the GitHub item, not this page).

## Body

## Problem

Enforcement of the Epic #2517 admin-bypass exception is **reactive-only**: the exception
(`merge-bypass:admin-exception` label or PR-body `BLOCKER_NOTE` with `bypass_reason:`+`approver:`)
is validated **after** the fact by `merge-bypass-gates.yml` → `scripts/global/megalint/admin-merge-exception.js`.

There is **no pre-flight guard**: `hooks/scripts/pretool_guard.py` already intercepts admin-override
merge commands (it aggressively keyword-matches and will block even a read-only `grep` containing the
token) yet has **0 references** to the exception precondition. So an operator can perform an
admin-override merge on a *false-BLOCKED* state, pass nothing, and only discover the missing exception
when a post-merge check goes red — forcing rework.

This violates the standards' own mandate: *"Prefer prevention over reaction: local guardrails first,
CI backstops second."* The guardrail here is backstop-only.

## Evidence (origin incident)

- #2703 / PR #2705: merged via admin-override on a false-BLOCKED state (30/30 checks green,
  `required_approving_review_count=null`) **without** a pre-recorded exception.
- Post-merge `merge-bypass-gates` went red → manual remediation required (label + BLOCKER_NOTE + audit
  comment) after the fact. The red check-run is now a frozen historical artifact on the merged PR.
- Tier-1 incident: `~/.megingjord/incidents.jsonl` pattern_id `admin-bypass-merge-without-pre-recorded-exception`.

## Why this did not self-anneal (meta-finding)

1. No Tier-1 event was emitted at failure time, so the recurrence detector had nothing to count.
2. The flaw was misclassified as `memory-note-only` instead of `file-ticket` (now corrected).
3. First-occurrence does not auto-escalate by design (≥2 needed) — which *depends* on Tier-1 logging
   happening. There is **no automation** that converts an operator-caused gate failure + remediation
   into a Tier-1 event; it relies on the operator remembering.

## Goal lens

G3 Zero-Cost (avoidable rework burns tokens), G1 Governance (bypass reached a non-compliant state),
G6 Resilience (no pre-merge fallback/guard), G2 Quality (avoidable failure).

## Proposed scope (two structural fixes)

- **AC1 — Pre-flight guard (prevention):** extend `pretool_guard.py` so that before an admin-override
  merge it checks the linked issue for the exception label OR the PR body for the `BLOCKER_NOTE`
  (`bypass_reason:`+`approver:`); if absent, emit a **deny with the exact remediation** (record the
  exception first). Reuse `admin-merge-exception.js#hasException` — single source of truth, no logic fork.
- **AC2 — Auto-Tier-1 bridge (observability → anneal):** add a hook/path that, when an operator-caused
  governed gate goes red and is then remediated in-session, appends a Tier-1 `incidents.jsonl` event
  automatically, so escalation no longer depends on the operator manually logging it.
- **AC3** — guard precision: avoid over-blocking read-only commands that merely *mention* the token
  (the current guard blocked a `grep`); match the actual merge invocation, not substring mentions.
- **AC4** — tests: stress/behavioral coverage for the guard (deny-without-exception, allow-with-label,
  allow-with-blocker-note, read-only-not-blocked) per the test-methodology matrix (side-effect-bearing gate → tdd-pyramid + stress).
