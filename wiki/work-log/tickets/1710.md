# #1710 — Self-anneal: github-dispatcher needs execute() that actually invokes the chosen provider

> **Source**: github:issue/1710 | **state: CLOSED** | **Labels**: type:task, status:done, priority:P2, area:scripts, resolution:completed, lane:code-change
> Mirror of `gh issue view 1710` (derived; edit the GitHub item, not this page).

## Body

## Symptom

`scripts/global/github-dispatcher.js` (shipped via PR #1702, #1643) returns `{provider, tool}` selection metadata but does NOT execute the call. `dispatch('create-issue', ...)` returns the tool name; no helper takes that result and invokes either an MCP server or `gh` CLI. The "MCP-driven Issue create" claimed in #1643's AC is tested at the routing-decision level, not the actual invocation level.

## Recurrence evidence

Surfaced via Codex Team (gpt-5.5) MCP-driven repo inspection on 2026-05-16 during Epic #1604 closeout review. Codex correctly identified that the dispatcher is "a provider selector, not a real MCP migration layer."

## Pattern class

Routing-only helpers without an execution layer; same orphan-utility class as #1708, #1709.

## Acceptance Criteria

- [ ] AC1: Add `executeViaCli(operation, params)` that shells out to `gh` CLI with the correct arguments per operation.
- [ ] AC2: Add `executeViaMcp(operation, params, mcpClient)` that invokes the MCP tool by name. (MCP client is the standardized MCP SDK shape; injected for testability.)
- [ ] AC3: Add `execute(operation, params, env)` that selects via `dispatch()` and invokes the appropriate executor.
- [ ] AC4: 10+ unit tests covering both execution paths with mocked CLI/MCP responses.
- [ ] AC5: Integration smoke: dispatcher-execute create-issue → real issue exists.

tdd-pyramid

## Out of scope

- Migrating existing callers from direct `gh` CLI to the dispatcher — per-helper tickets per the #1641 audit (see #1710 batch).
- Live MCP server in test env — mock-based unit tests only; smoke test only.
