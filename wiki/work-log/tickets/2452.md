## Industry research (2026)

### Coordination topologies

Per the 2026 multi-agent orchestration survey, three topologies dominate:

1. **Orchestrator-worker** (~70% production deployments): single coordinator owns state, dispatches workers, aggregates results. Simpler to reason about; SPOF + bottleneck risk.
2. **Peer-to-peer / swarm**: emergent coordination, no central controller. Resilient; ~impossible to audit (kills G1+G8).
3. **Hierarchical supervisor**: tree of coordinators; rare in code-review workflows.

Megingjord today is best described as "orchestrator-worker with a stale local cache" — gets neither simplicity nor distributed resilience.

### State management patterns

Per state-management taxonomy from 2026 frameworks survey:
- **Checkpointed** (LangGraph): periodic snapshots of full state; replay from last checkpoint
- **Ephemeral** (CrewAI defaults): in-memory state, no persistence beyond run
- **Event-sourced** (ESAA, Microsoft Agent Framework): immutable event log; current state = fold(events)

State management is "the primary technical challenge across every production multi-agent account" (Augment Code 2026).

### Specific 2026 frameworks evaluated

| Framework | Relevance to Megingjord |
|---|---|
| **POLARIS** (arXiv Feb 2026) | "Typed plan synthesis with DAG-based planning, rubric-guided selection, validator-gated execution, **compiled policy guardrails**" — directly applicable: guardrails compile from single source of truth |
| **ESAA** (arXiv 2602.23193, Feb 2026) | "Event Sourcing for Autonomous Agents in LLM-Based Software Engineering" — separates cognitive intention from project state mutation; directly addresses our prose-collision and state-desync classes |
| **LangGraph** (LangChain) | State-machine for LLM workflows with built-in checkpointer; orchestrator-worker friendly |
| **Microsoft Agent Framework** (April 2026 public preview) | Policy-as-code, cost control, audit logging, enterprise identity integration |
| **Open Agent Control Plane** | Pre-execution policy enforcement, approval gates, audit trails; works with LangChain/CrewAI/MCP |

### Economics

- Independent multi-agent setups: ~58% extra token overhead vs single agent
- Centralized orchestration: ~285% extra token overhead vs single agent
- Multi-agent only pays off when task genuinely benefits from specialization, parallelism, or critique
- Implication for Megingjord: G3 (zero cost) lens favors minimizing orchestration overhead — collapsing the local cache is a net token reduction, not addition

## Fledgling 3-move plan

### Move 1 — Local state file becomes ephemeral read-cache (60s TTL)

Current:
```
tool_activity.py writes roles{}, flags{}, admin_ops{} → state file
stop_reminder.py reads state file → check_admin_ops()
```

Proposed:
```
stop_reminder.py invokes derive_roles_from_github(active_ticket_n)
  → gh issue view N --json labels,comments (cached 60s)
  → parse role:* labels + 4 named handoff artifacts
  → return {manager, collaborator, admin, consultant: bool}
```

Local cache becomes read-only TTL store; no writes from baton transitions.

### Move 2 — Event-source the baton

```
~/.megingjord/baton-events.jsonl (append-only, schema v3)
  {ts, version: 3, service: "baton", event: "role-handoff",
   from: "collaborator", to: "admin", ticket: N, signer: "Orla Harper",
   trace_id, session_id}
```

Used for: replay, audit, eval-harness, dashboard.
NOT used for: live gate decisions (those derive from GitHub).

### Move 3 — Cross-team merge-claim primitive

HAMR Worker exposes:
```
POST /merge-claim/acquire/{ticket-n}  → {claim_id, ttl_s: 60}
POST /merge-claim/release/{claim_id}  → ack
GET  /merge-claim/status/{ticket-n}   → {held_by_team, expires_at}
```

Admin role acquires claim before `gh pr merge`; releases after merge or on failure.
Closes the "Base branch was modified" silent-race class.
