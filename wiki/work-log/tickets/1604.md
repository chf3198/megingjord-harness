# #1604 — Establish conflict-free cross-team harness governance

## Body

## Epic Summary

Critically examine and upgrade the harness governance that coordinates Claude Code Team, Copilot Team, and Codex Team so parallel work becomes conflict-free, observable, and genuinely collaborative rather than merely isolated.

## Problem

The harness already has strong ticket, branch, worktree, and baton rules, but the three teams still create friction:

- Branch/worktree clutter accumulates in VS Code Source Control.
- Team sessions can stale-lock or stale-own tickets after work completes.
- Teams can choose overlapping files or adjacent governance surfaces without seeing each other's intent soon enough.
- Global instructions, skills, repo instructions, and runtime hooks are not yet normalized into one clear cross-team contract.
- Communication is mostly indirect through GitHub issue comments after work begins, not a pre-edit coordination protocol.

## Research Basis

Initial research child: to be completed under this Epic before implementation children are picked up.

Early external signals to validate:

- OpenAI Codex cloud tasks run in separate task environments and can work in parallel, which supports isolation as a baseline but not cross-tool coordination.
- GitHub Copilot coding agent works from assigned issues in isolated development environments and creates PRs, making issue-level dispatch the durable shared contract.
- Claude Code guidance emphasizes separate git worktrees for parallel sessions and distinguishes worktrees, subagents, and agent teams.
- VS Code Copilot cloud agent documentation reinforces that autonomous agent work should be reviewable through PRs and task state.
- Recent AI-agent PR research shows agent workflows increasingly carry operational initiative, while governance/review authority remains a separate control layer.

## Proposed Direction

Create a cross-team coordination plane layered above worktrees and tickets:

1. A single issue-backed claim/lease registry for ticket, branch, file/path, port, and runtime-surface ownership.
2. Pre-edit conflict prediction that blocks or warns before overlapping work begins.
3. Team communication artifacts that are structured, append-only, and readable by every team.
4. A dashboard/CLI view that turns many worktrees and branches into a managed queue rather than VS Code clutter.
5. A normalized instruction/skill contract shared across Claude Code, Copilot, and Codex, with team-specific adapters only where necessary.
6. Cleanup automation that closes the loop after merge: labels, local branch, remote branch, stale state, and VS Code workspace visibility.
