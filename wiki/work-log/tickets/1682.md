## Body

## Context
Part of Epic #1626. Independent.

`logs/wiki-metrics.json` contains a `pages` map that is always empty `{}`. The root cause: the access-tracking function fires with a section name but no page slug for the majority of read calls. Page-level popularity has zero effective signal.

## Objective
Fix the access tracking call sites so that each wiki page read records its slug in the `pages` map. Surface page-level popularity in the dashboard wiki panel.

## Implementation notes
- Audit all callers of the access-tracking function in `scripts/wiki-metrics.js` and `scripts/wiki-reader.js` — identify where slug is not passed
- Fix callers to pass the page slug; fall back to `section/filename` if frontmatter title unavailable
- Update the dashboard wiki panel (`dashboard/js/wiki-panel.js` or `dashboard-wiki.js`) to display a "Top pages" section sourced from the `pages` map
- `pages` object should store `{ [slug]: count }` using atomic read-modify-write on `wiki-metrics.json`

## Acceptance criteria
- [ ] After 5 wiki read operations, `logs/wiki-metrics.json` `pages` map contains ≥ 1 slug entry
- [ ] Dashboard wiki panel displays a visible "Top pages" section with at least placeholder data
- [ ] No existing section-level telemetry is removed or broken
- [ ] `npm run wiki:lint` passes
