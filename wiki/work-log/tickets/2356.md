# #2356 — Harden governance guardrails against low-capability model drift

> **Source**: github:issue/2356 | **state: CLOSED** | **Labels**: type:epic, priority:P1, area:governance, status:cancelled, coordinator:cross-team-needs-hand-off
> Mirror of `gh issue view 2356` (derived; edit the GitHub item, not this page).

## Body

## Problem

The Megingjord governance harness enforces compliance primarily through CI gates, lint, and pre-commit hooks. These mechanisms catch file-level violations (line count, JSON schema, naming conventions) but cannot detect missed *process* steps — specifically whether required baton artifacts (MANAGER_HANDOFF, COLLABORATOR_HANDOFF, ADMIN_HANDOFF), changelog fragments, and cross-link updates were actually posted.

Issue #2254 exposed this gap directly. The implementation work was performed by an Auto-tier model, which satisfied the content requirements of all 5 ACs while systematically bypassing every governance process requirement. A subsequent review by Claude Sonnet 4.6 High identified the full scope of violations — the harness had no mechanism to surface them during execution.

The violating model produced work that passes lint and would pass CI while being non-compliant with G1 Governance at a 5/10 rating. This is a **trust boundary failure**.

## Root Cause (Phase-0 finding)

The harness enforces two fundamentally different kinds of compliance:

- **Mechanical compliance** (file structure, line count, branch names) — enforced deterministically by pre-commit hooks, lefthook pre-push, CI lint
- **Process compliance** (baton artifacts, changelog fragments, cross-links) — entirely voluntary; no deterministic enforcement at any lifecycle layer

Every process compliance step that a capable model observes voluntarily is a gap that a less capable model will silently skip. The harness's enforcement architecture is therefore model-capability-dependent at its core.

### The gate bypass that exposed the gap

`baton-gates.yml` checks for COLLABORATOR_HANDOFF in issue comments, but **explicitly skips this check for `lane:docs-research`** (and `lane:docs-only`, `lane:trivial`, `lane:no-code-remediation`). Issue #2254 carries `lane:docs-research`. The one CI gate closest to catching the violation was configured to not fire for this exact ticket type.

### The event model mismatch

All existing process-compliance gates fire at PR creation (post-code, post-commit, post-push). Baton state — whether MANAGER_HANDOFF exists before work begins, whether COLLABORATOR_HANDOFF was posted before Admin picks up — is never checked at the transition event itself. The two event streams (baton state changes vs. git events) are structurally decoupled.

## Why This Is Critical

The severity scales directly with how often Auto/lower-tier model selection is used. Docs and research tickets (lane:docs-research, lane:docs-only) are disproportionately likely to be assigned to Auto-tier, and they carry the exact bypass that removes CI enforcement for those lanes.

## Revised Success Criteria

The original success criterion ("at least one blocking gate at Admin pickup") was wrong in three ways: it targeted a single transition rather than every transition; it targeted a gate that is bypassed for lightweight lanes; and it accepted minimum coverage rather than complete coverage.

The correct success criteria for this Epic are:

1. **Complete taxonomy** — every required governance step for every lane type is classified as: (a) mechanically enforced at the correct lifecycle event, (b) requires new enforcement, or (c) intentionally advisory with documented justification. No unclassified steps remain.

2. **Transition-time enforcement** — each baton phase transition has a deterministic, model-capability-independent gate that fires at or before the transition event, not after code is committed. Specifically:
   - Before first file write on a ticket branch: MANAGER_HANDOFF verified present on linked issue
   - Before commit: changelog fragment presence verified (for non-trivial lanes)
   - Before Admin baton pickup: COLLABORATOR_HANDOFF verified present on linked issue
   - These gates must fire for ALL lanes including docs-research

3. **Bypass elimination** — the lightweight-lane bypass in `baton-gates.yml` is removed or narrowed to only exclude genuinely unconstrained lanes (e.g., `lane:trivial`) with explicit documented rationale for each exclusion.

4. **Session-start gate** — `pretool_guard.py` verifies baton state (MANAGER_HANDOFF present on linked issue) before permitting file writes, not relying on model voluntary compliance.

5. **Regression test** — a synthetic non-compliant session exercise demonstrates that gates fire for a docs-research lane ticket where baton artifacts are absent.

6. **apply_patch / canonical-main enforcer bug** — resolved or formally deferred with an explicit Tier-2 anneal ticket and operator workaround documented at the repo level.
