## Self-anneal trigger

Tier-2 mid-flight pivot triggered by ≥3 recurrences in 7-day window per `instructions/workflow-resilience.instructions.md`. Pattern: flaw-emission validator (`scripts/global/megalint/flaw-emission.js`) over-triggers on marker words (`flaw`, `bug`, `failure`, `incident`) used inside legitimate per-goal-rubric prose or per-AC verification narratives in baton artifacts. Each recurrence cost an iteration of `gh api PATCH` to rephrase + CI re-run.

All hits were FALSE positives — none were actual mid-flight flaws lacking citation; they were feature-descriptive prose using common English compound forms.

## Root cause

`scripts/global/megalint/flaw-emission.js` line 4:
```js
const MARKERS = [/\bI had to\b/i, /\bworked around\b/i, /\bside-?effect\b/i, /\bflaw\b/i, /\bbug\b/i, /\bfailure\b/i, /\bincident\b/i];
```

Per-marker word-boundary regex is broad enough to catch:
- `failure mode` (technical idiom for "error condition")
- `bug-fix` (compound noun for "defect-fix")
- `flaw=...` (structured-field key inside a `mid_flight_flaws:` block where the citation IS present, just on adjacent lines)
- `failure modes`, `error path failure`, etc. — common engineering vocabulary

The validator's intent is to catch flaw mentions in NARRATIVE prose (like "we had a bug here but didn't file a ticket") that lack the required artifact citation. It catches that — but also catches all legitimate non-flaw uses of the same words.

## Memory anchor (existing, codified)

`memory/feedback_flaw_emission_failure_word.md` documents the pattern. Memory `feedback_baton_artifact_format_pitfalls.md` documents the broader "validator regex collision" class. The memories codify operator workaround (rephrase the marker word) but don't fix the underlying validator.

## Phase-0 research deliverables

- [ ] AC-R1: Catalog the exact patterns that legitimately use marker words (e.g. per-goal-rubric prose, per-AC verification narratives, structured-field keys like `Flaw=`). Build a positive-test corpus of >=15 examples drawn from the 3 recurrences this session.
- [ ] AC-R2: Catalog the patterns that are TRUE-positive mid-flight flaw mentions lacking citation. Build a negative-test corpus of >=10 examples (some real from incident log, some synthesized).
- [ ] AC-R3: Survey 2026 NLP-validator patterns for "intent-aware" flaw detection. Options to evaluate: (a) negative-lookahead regex (skip if marker is followed by `mode`, `-fix`, `=`, etc.); (b) context-window heuristics (skip if marker is inside a `mid_flight_flaws:` block or per-goal-rubric line); (c) LLM-judge-as-validator with the marker corpus as fixtures; (d) require explicit `<!-- flaw-mention -->` HTML comment to opt IN to validation; (e) shift to structured-field validation (only check `mid_flight_flaws:` blocks for proper citation).
- [ ] AC-R4: Design the validator-softening proposal. Recommended path (subject to research): option (a)+(b) hybrid — skip negative-lookahead idioms AND skip inside known-safe section markers.
- [ ] AC-R5: Replay-eval against the historical corpus: re-run the candidate validator against ALL recent baton artifacts; measure (true-positive-recall, false-positive-precision) vs current; promote only if precision improves >=20% with recall preserved.
- [ ] AC-R6: Consultant peer-review rubric >=7 across G1-G9 plus cross-family fleet rater check.
