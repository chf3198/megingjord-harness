# #1923 — Fix evidence-completeness: allow Epic-named branches for multi-close batch PRs

> **Source**: github:issue/1923 | **state: OPEN** | **Labels**: type:bug, priority:P2, area:infra, role:manager, status:triage, lane:code-change
> Mirror of `gh issue view 1923` (derived; edit the GitHub item, not this page).

## Body

Parent Epic: #1894

## Problem

The `evidence-completeness` CI check fails when a branch is named after a parent Epic (e.g., `feat/1894-pre-pr-gate`) and the PR body contains child `Refs #N` links where N ≠ the Epic number.

**Failing example:**
- Branch: `feat/1894-pre-pr-gate`
- PR body: `Refs #1894` (Epic) + `Refs #1896` (child)
- Check finds first non-epic Refs = #1896
- Violation: `Branch #1894 ≠ Refs #1896`

The current check does not handle Multi-Close PR batching (per governance rules) where the branch is named after the parent Epic but delivers multiple children.

## Expected Behavior

Per governance multi-close batch rules: if the branch is named after a known Epic (issue with `type:epic` label) AND the PR body contains `Refs Epic #N` where N matches the branch number, the branch-binding check should pass.

## Acceptance Criteria

- [ ] AC1: Update `evidence-completeness.yml` branch-binding check to skip the number mismatch when the branch number matches an Epic referenced in the PR body
- [ ] AC2: Regression test: PR with `feat/1894-...` branch + `Refs Epic #1894` + `Refs #1896` passes the check
- [ ] AC3: Existing check still enforces branch-to-issue binding for non-batch PRs
- [ ] AC4: Add integration test in `tests/evidence-completeness.spec.js` (or equivalent)
