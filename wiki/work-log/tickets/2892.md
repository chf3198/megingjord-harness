# #2892 — User-configurable harness guardrail overrides per repo

> **Source**: github:issue/2892 | **state: OPEN** | **Labels**: type:epic, status:backlog, priority:P2, area:instructions, role:manager
> Mirror of `gh issue view 2892` (derived; edit the GitHub item, not this page).

## Body

## Objective

Design a mechanism that allows repository operators to override or alter specific harness guardrails on a per-repo basis, without modifying the harness source itself.

## Problem

Once governance guardrails are machine-enforced (Epic #2891), they become deterministic blocks. Some operators may have legitimate reasons to loosen, tighten, or substitute a specific guardrail for a given repository context (e.g., a research-only repo that does not need cross-family preflight; a solo-operator repo that does not need signer-independence checks). Without an override mechanism, harness guardrails become a one-size-fits-all constraint.

## Scope (Phase-0 to be defined)

Research questions:
- What is the right override surface? (repo-local config file, label, env var, per-workflow input)
- Which guardrails should never be overridable (G1 hard-floor controls)?
- How do overrides compose when the harness updates (drift risk)?
- How should overrides be audited/visible (G8 observability)?
- Should overrides require approval or be self-serve?

## Dependencies

- Epic #2891 (Governance guardrails assessment) must reach Phase-1 before Phase-0 of this Epic can be scoped — we need to know which guardrails exist before designing their override surface.
