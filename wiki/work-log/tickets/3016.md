# #3016 — Activate doc-coverage collaborator gate by deriving lane from labels

> **Source**: github:issue/3016 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P1, area:governance, lane:code-change
> Mirror of `gh issue view 3016` (derived; edit the GitHub item, not this page).

## Body

## Problem
The doc-coverage Tech-Writer gate (validator shipped by #2712) is **dead code in CI**: `collaborator-handoff.js` gates the doc-coverage and cross-family checks on `input.lane === 'lane:code-change'`, but `.github/workflows/baton-gates.yml` calls `validate({ comments, labels })` and never passes a `lane` scalar. So both checks are silently skipped on every PR — no PR has ever been blocked for missing documentation.

Two secondary defects compound it:
- Parser key mismatch: `parseDocBlock` matches `doc-coverage:` (hyphen) but the canonical `baton-comment-build.js` renders the field as `doc_coverage:` (underscore).
- Fail-open on matrix load: `try { loadMatrix() } catch { matrix = null }` then skips the check, so a matrix-load error silently disables enforcement.

Evidence: PRs #3007/#3010/#3011 (this is the lead's sibling context) merged green with non-conforming doc-coverage blocks.

## Scope
Activate the gate (make it actually block lane:code-change PRs) with a safe escape hatch. Validator semantics (N/A enum #2714, matrix #2713) are NOT re-litigated.

## Acceptance Criteria
- Derive code-change from `input.labels` when `input.lane` is absent, so the doc-coverage + cross-family checks run under the real CI call signature.
- `parseDocBlock` accepts both `doc-coverage:` and `doc_coverage:`; `baton-comment-build.js` aligned to the canonical key.
- Matrix load failure FAILS CLOSED (blocking violation), not silent skip.
- `LEGACY_DOC_SKIP` escape hatch requires a BLOCKER_NOTE on the linked issue (per #2707 AC1).
- Regression tests reproduce the dead-gate (labels-only call must enforce) and the key/fail-closed fixes.

Refs #2707
