## Objective

AC-R2 Phase-1 child of Epic #2356. Design the transition-time enforcement mechanism for all baton transitions across all lanes per the Phase-0 audit deliverable `research/governance-guardrail-audit-2026-05-28.md` §2.

## Background

Phase-0 audit (#2357, deliverable shipped via PR #2359 to research/governance-guardrail-audit-2026-05-28.md) established the gap: process compliance is voluntary across multiple surfaces; lightweight lanes (docs-research, trivial, no-code-remediation, docs-only) bypass CI baton-gates entirely. The audit recommended Phase-1 implementation order with step 1 = `config/governance-decision-policy.json` and step 2 = `scripts/global/governance-decision-engine.js`.

This ticket scopes the design + initial schema for the decision-policy data and the engine's evaluate(context) → {decision, checks[], degradations[], auditTrace} contract. Implementation of the engine itself is a separate child.

## Acceptance Criteria

- AC1: `config/governance-decision-policy.json` schema defined and documented (per-transition × per-lane-severity × per-runtime-profile → check_set mapping)
- AC2: Decision engine contract specification (input shape, output shape, audit-trace fields, degradation behavior per runtime profile)
- AC3: Per-lane lighter-check-set assignment for trivial / docs-research / no-code-remediation lanes per audit §3 quantitative table
- AC4: Migration plan from current per-validator hardcoded logic to the engine
- AC5: Cross-link to sibling Phase-1 children to be filed: AC-R3 (session-start gate), AC-R4 (lightweight-lane bypass narrowing in baton-gates.yml), AC-R5 (pre-commit changelog fragment enforcement), AC-R6 (regression test)
