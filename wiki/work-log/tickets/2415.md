# #2415 — Research: programmatic doc-coverage guardrail patterns for hard-blocking gate design

## Body

## Scope (Phase-0 child of Epic #2414)

Survey existing programmatic documentation-coverage guardrail implementations to inform the design of a hard-blocking doc-coverage gate for the Megingjord baton workflow.

## Why

Epic #2414 promotes doc-coverage from advisory to blocking. Before designing the gate, we need to understand: what patterns exist in the wild? What has worked, what hasn't, and what can we adapt rather than invent?

## Research Questions

- RQ1: Do any CI/CD systems enforce "docs must be updated before merge" as a hard gate? If so, how?
- RQ2: Are there tools (Danger.js plugins, GitHub Actions, pre-commit hooks, megalint-style validators) that enforce required-file-change patterns?
- RQ3: How do major OSS projects (Kubernetes, Next.js, Django, Rust stdlib, Go stdlib) enforce documentation updates alongside code changes?
- RQ4: What does a "doc-coverage block" need to contain for a validator to verify it deterministically?
- RQ5: What are the known failure modes of hard-blocking doc gates (false positives, contributor friction, bypass patterns)?

## Acceptance Criteria

- AC1: RQ1–RQ5 answered with cited sources (URLs, GitHub refs, or internal codebase paths)
- AC2: At least 3 real-world examples of programmatic doc guardrails found and described
- AC3: A recommended gate design pattern derived from findings, suitable for Megingjord's megalint + GitHub Actions architecture
- AC4: Known failure modes catalogued with proposed mitigations
- AC5: Findings posted as comment on this ticket; Epic #2414 receives EPIC_RESCOPE comment before Phase-1 work begins
