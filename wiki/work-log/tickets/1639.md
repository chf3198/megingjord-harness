# #1639 — Self-anneal: closeout-preflight hardcodes lane:code-change blocking docs-only tickets

> **Source**: github:issue/1639 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P3, area:scripts, resolution:completed, lane:code-change
> Mirror of `gh issue view 1639` (derived; edit the GitHub item, not this page).

## Body

## Symptom

`scripts/global/closeout-preflight.js` hardcodes `lane: 'lane:code-change'` in `toValidatorInput()` (line ~36), regardless of the issue's actual `lane:*` label. The validator then fails with `lane-mismatch` on any non-code-change MANAGER_HANDOFF, blocking pre-push on otherwise-valid tickets.

## Recurrence evidence

Surfaced during #1628 push (lane:docs-only). The MANAGER_HANDOFF correctly declared `lane: lane:docs-only` and the issue carried the `lane:docs-only` label — but the preflight failed because its hardcoded `lane:code-change` did not match. Required `SKIP_CLOSEOUT_PREFLIGHT=1` to push.

## Pattern class

This is a latent recurring issue: every docs-only, docs-research, config-only, research, or trivial-lane ticket will hit this. CI workflows derive lane from labels correctly; only the pre-push preflight has the hardcoded value.

## Root cause

`toValidatorInput()` does not derive `lane` from the labels array it already constructs. The line `lane: 'lane:code-change',` should be replaced with logic that extracts the `lane:*` label from `normalizeLabels(issue.labels)`.

## Acceptance

- AC1: closeout-preflight derives `lane` from the first `lane:*` label on the issue (default `lane:code-change` if none).
- AC2: unit test added in `tests/closeout-preflight.spec.js` covering each lane value.
- AC3: re-run on #1628 (closed) would now pass without `SKIP_CLOSEOUT_PREFLIGHT=1`.
- AC4: changelog fragment + no behavior change to CI workflows.

## Out of scope

- Re-architecting closeout-preflight beyond the lane-derivation fix.
- Changing the validator schemas themselves.
