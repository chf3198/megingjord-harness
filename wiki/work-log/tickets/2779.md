# #2779 — Enforce 100-line split-and-link design contract as repo guardrail

> **Source**: github:issue/2779 | **state: CLOSED** | **Labels**: status:done, resolution:released, governance:close-without-merge
> Mirror of `gh issue view 2779` (derived; edit the GitHub item, not this page).

## Body

## Problem

The `≤100 lines` lint rule blocks over-limit files but gives no design guidance.
Developers respond by compressing content to fit the limit (losing clarity and
completeness). The correct response is to split into linked files.

## Objective

Make the split-and-link pattern the enforced guardrail — the 100-line limit means
**one focused unit of responsibility**, not a content budget to optimize against.

## Acceptance Criteria

- AC1: `scripts/lint.js` error message states the design contract and directs
  to split into linked files rather than compressing content.
- AC2: `docs/howto/100-line-design-contract.md` created with full guidance for
  all file types: markdown, JS/TS modules, CSS, HTML, shell scripts.
- AC3: `.github/copilot-instructions.md` (100 lines, 7 compressed one-liners)
  expanded into proper sections + companion file for advanced contracts.
- AC4: Root `WIKI.md` (100 lines) expanded to a navigation hub linking to the
  `wiki/WIKI.md` + `wiki/WIKI-operations.md` + `wiki/WIKI-typology.md` suite.
- AC5: `CONTRIBUTING.md` references `docs/howto/100-line-design-contract.md`.
- AC6: `npm run lint` passes (all files ≤100 lines) after expansion.
