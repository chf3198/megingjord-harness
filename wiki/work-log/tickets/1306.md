# #1306 — Epic close-readiness gate: text-match matcher reports false-positive children

> **Source**: github:issue/1306 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P1, area:governance, resolution:completed, lane:code-change

## Body

## Problem

The Epic Close-Readiness Gate workflow auto-reopens legitimately-closed Epics when its child-detection matcher reports false positives. Live evidence: Epic #1103.

On 2026-05-08T19:31:46Z, Epic #1103 received a valid CONSULTANT_CLOSEOUT (Orla Vale, all 10 D-001..D-011 children merged) and was closed. 18 seconds later (19:32:04Z), the gate auto-reopened with this comment:

> **## Epic Close-Readiness Violation**
>
> Open child issues detected for epic #1103:
>
> - #1112: Productize multi-team parallel R&D collaboration as harness feature _(match: text)_
> - #1113: Multi-layer self-annealing goal-governance system _(match: text)_
> - #1114: Design Goal Health Score and self-annealing actuator matrix _(match: indirect-via-#1113)_
> - #1125: AI-augmented dependency graph for Epic and ticket landscape _(match: indirect-via-#1114)_
> - #1126: R&D: design AI-augmented dependency proposal system _(match: indirect-via-#1125)_
> - #1130: Optimize and harden HAMR for universal harness governance _(match: text)_
> - #1131: Cross-team R&D: design universal HAMR coverage strategy _(match: indirect-via-#1130)_
> - #1132: Adopt per-ticket CHANGELOG fragments to eliminate merge conflicts _(match: text)_
> - #1133: Automate self-annealing for recurring workflow failures _(match: indirect-via-#1132)_
> - #1134: Cross-team R&D: design automated workflow-anneal detector + pattern catalog _(match: indirect-via-#1133)_

**None of these are children of #1103.** They are sibling Epics, follow-on Epics, or R&D bootstraps that #1103's body merely *mentions*. The matcher's `match: text` and `match: indirect-via-#N` heuristics over-collect.

## Impact

- Legitimately-completed Epics cannot close (#1103 has been stuck for 2 days).
- Operator must manually re-close, gate re-opens, and the cycle repeats.
- Bypasses the actual close-readiness check (real children of #1103 are #1115-#1124, all merged).
- A second-order bug: when the gate auto-reopens, it does NOT revert the `status:done` label to a pre-close value, leaving the issue in a forbidden `status:done` + open state (caught by label-lint as drift).

## Acceptance Criteria

- [ ] **AC1 — Matcher uses task-list-only edges by default.** A child reference must be a checkbox bullet `- [ ]` / `- [x]` in the Epic body OR a GitHub-native sub-issue link. Plain prose mentions of issue numbers no longer count as parent-child.
- [ ] **AC2 — Indirect-via traversal disabled (or explicitly opted-in).** The `match: indirect-via-#N` heuristic is removed unless an explicit setting enables it for a specific use case. Document the rationale in the workflow file.
- [ ] **AC3 — Status-label restoration on auto-reopen.** When the gate reopens an issue, it must also revert `status:done` (and resolution labels) to the pre-close value (or `status:review` as a safe default).
- [ ] **AC4 — Dry-run / preview mode.** Add a workflow_dispatch input or PR-comment trigger that lists what the matcher *would* flag, without taking any action. Use this to vet the matcher before re-enabling enforcement.
- [ ] **AC5 — Live verification on #1103.** Re-run the gate against #1103. With AC1+AC2 in place, the gate should report zero open children and allow close.
- [ ] **AC6 — Regression test.** Golden-file fixture of #1103's body + sibling Epics; assert matcher returns zero children. Add a fixture for an Epic with real task-list children; assert it still detects them correctly.
