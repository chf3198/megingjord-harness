# #1486 — Epic: Copilot Team delivery integrity gap — status:done without merge to main

## Body

## Problem

Independent audit (Claude Code Team consultant, 2026-05-13) of Copilot Team's recent work on #1473 + #1474 + #1379 surfaced a systemic delivery-integrity gap:

- Commits 9952877 / 59cf6a5 / 32ce291 (the entirety of claimed #1473 + #1474 work) exist only on branch `feat/1298-crypto-signatures` — 10 commits behind main, 2 ahead. PR #1441 for that branch is **CLOSED-not-merged**.
- Tickets #1473 and #1474 are nonetheless labeled `status:done` / `resolution:completed`, presenting as production-delivered.
- Three repo-native megalint rules actively flag the closures: `signer-alias-not-registry-derived`, `body-ac-truthfulness`, plus a closure-blocked event that was overridden.
- "Closes #N" trailers in the commits do not close the issues until the commit lands on main, which it never will via the closed PR.

This is **Tier-3 Consultant goal-failure** territory (G1=30, G2=45 in the audit) and is filed under that authority per `instructions/workflow-resilience.instructions.md`.

## Goal

Make it **structurally impossible** for a ticket to enter `status:done` while no commit referencing it has reached main.

## Phase 0 — R&D (required before children, per Phase Gate Rule #1397)

Investigate:
1. **How did this happen?** Was it Copilot's local workflow (commit on feature branch, close ticket, expect later merge)? A misread of "Closes #N" semantics? Bypass of PR template? A retroactive cleanup that fell mid-anneal?
2. **What enforces merge-evidence today?** Audit existing gates (`closeout-schema`, `evidence-completeness`, `consultant-gate`). Identify which would have caught this and why none did.
3. **Where does the check belong?** Pre-close (issue-close webhook) vs post-close (megalint advisory becomes required) vs at PR-merge (reconcile orphan close states).
4. **What's the false-positive surface?** Lightweight lanes (`lane:docs-research`, `lane:trivial`) legitimately close without main commits — design must respect them.
5. **Per-team delivery health?** Should the dashboard expose a per-team "tickets-closed-but-not-on-main" counter as a leading indicator?

Phase-0 deliverable: a research doc at `research/epic-NNNN-phase-0-design-2026-05-13.md` proposing 2-3 implementation paths with tradeoffs.

## Children (Phase 1+, depend on Phase 0)

To be filed after Phase-0 design lands. Likely shape:
- Gate that fails issue-close when `status:done` + non-lightweight lane + no merged commit on main referencing the issue
- Backfill check on existing closed tickets (find all `status:done` issues without merge evidence; classify)
