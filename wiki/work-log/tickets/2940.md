# #2940 — HAMR-bypass detection required check skips on config/docs-only PRs, blocking merges

> **Source**: github:issue/2940 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P2, area:infra, merge-bypass:admin-exception
> Mirror of `gh issue view 2940` (derived; edit the GitHub item, not this page).

## Body

## Problem

The required status check **HAMR-bypass detection** (`.github/workflows/hamr-bypass-lint.yml`) triggers only on:
```yaml
on:
  pull_request:
    paths: ['scripts/**','dashboard/**','hooks/**','tests/**']
```
A PR that touches none of those paths (e.g. `config/**`-only or `docs/**`/`instructions/**`-only) never triggers the job, so the **required** check stays permanently `expected` and the PR cannot satisfy branch protection — a hard merge block.

## Evidence
- Hit live on **PR #2939 / #2937** (config-only: `config/timeout-policy.json`). All 30 other required checks were green; only HAMR-bypass detection sat unreported. Merged via REST admin endpoint with an Epic #2517 `merge-bypass:admin-exception` (recorded), as a workaround — not a fix.
- Tier-1 incident: `pattern_id=hamr-bypass-detection-required-skips-config-docs-paths`.

## Root cause = #2812 defect class
This is the **same** failure mode #2812 fixed for `validate-config-schemas` (required check that skips on non-matching paths → never reports → blocks all merges). #2812 is CLOSED; the fix pattern was not generalized to sibling path-filtered required checks.

## Proposed fix (apply #2812's always-report treatment)
Make the required check **always report a status**, regardless of changed paths. Options:
1. Remove the `paths:` filter; inside the job, early-exit **pass** when no relevant files changed (cheap no-op, but the context still reports success). Preferred — single source of the context name.
2. Add a companion no-op job posting the same `HAMR-bypass detection` context as success when paths don't match.

## Scope / relationship
- Umbrella friction: **#2810** (trivial/it-ops fixes cannot traverse the merge gate suite). This is a concrete, reproducible instance.
- Audit **all** path-filtered **required** checks for the same trap while here (not just HAMR-bypass): any required job with a `paths:` filter is suspect.

## Goal-lens
G1 (governance: a required gate that can't be satisfied is a broken gate) + G6 (resilience: merges shouldn't depend on an admin-bypass workaround) + G10 (maintainability: generalize #2812's fix rather than re-patch per-check).
