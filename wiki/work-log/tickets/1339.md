# #1339 — Optimize harness and HAMR logging for goal-aligned observability

## Body

## Problem

The harness has accreted multiple logging surfaces over time — `~/.megingjord/incidents.jsonl`, `~/.megingjord/cache-stats.jsonl`, `dashboard/events.jsonl`, HAMR Cloudflare Worker logs, HAMR KV state (`/quota`, `/mcp doctor:probe`, `substrate-health:latest`, `cache-stats:hit-rate-7d`), `generated/anneal-sensor.json`, governance-audit output (`/tmp/governance-audit.json`), recurring-patterns evidence, and the new Tier-1/2/3 event schema from Epic #1308.

What's unclear:

1. **Goal alignment** — does each logging surface map to one or more `G1..G9` goals? Are we collecting data that actually serves governance / quality / cost / portability / resilience / throughput / observability / interoperability outcomes?
2. **Gaps** — for each goal, is the evidence signal sufficient to detect drift/regression? `harness-goal-controls` wiki page enumerates enforcement points; this Epic audits whether the *evidence-signal* side is equally complete.
3. **Excess / dead logs** — fields and event types that no consumer reads (KV keys with stale schema, jsonl fields never queried, dashboard panels that never render certain rows). These add maintenance burden + privacy surface without value.
4. **Schema consistency** — Epic #1308 shipped `incidents.jsonl` schema v2 with backward-compat shim. Are other surfaces (`cache-stats.jsonl`, `events.jsonl`) similarly versioned? Are field names consistent across surfaces (`timestamp` vs `ts`, `severity` vs `level`)?
5. **HAMR-side parity** — the Worker emits its own logs (Cloudflare console + KV writes). Are operator-facing local logs (`hamr:cache-emit`, `hamr:health-push`, `hamr:cache-push`) capturing the right signals at the right cadence?
6. **Retention + rotation** — `incidents.jsonl` is append-only with `WINDOW_MS = 7 days` for the recurring-failure-rate sensor. Is 7d right for every consumer? Are we rotating old logs or letting them grow unbounded?

## Goal

Audit current logging coverage against the harness goal-lens (`G1..G9` per `instructions/global-standards.instructions.md`), identify gaps and excess, define a logging standard, and ship a small set of implementation children based on Phase-0 R&D findings.

Per goal-lens for *this* Epic: **G8 Observability > G1 Governance > G3 Zero Cost > G6 Resilience > G5 Portability**.

## Client direction (operator input — 2026-05-11)

Three priorities surfaced by the client to be incorporated into Phase-0 R&D and standard codification:

1. **LLM-consumable log format.** Logs are not just operator-facing — LLM agents read them during anneal pickup (#1310 router), governance audit (`governance-audit.js`), drift analysis (`anneal-goal-sensor.js`), and Consultant critique. The format must be **token-efficient**, **semantically clear**, and **prompt-friendly**. Research vectors: LangSmith / Helicone / Langfuse trace conventions; OpenTelemetry GenAI semantic conventions (2025 spec); structured-log fields LLMs reliably parse; whether inline summary fields beat full-content fields on cost-per-token.

2. **Real-time dashboard display from logs.** Live log tails (via SSE or equivalent) should feed dashboard panels for live activity viewing — not just periodic-poll snapshots. The harness already has `dashboard/sse-handler.js` plumbing. The missing piece is a low-latency pipeline: `*.jsonl` append → tail process → SSE channel → panel re-render. Latency target to be set by R&D (industry norm: <500ms perceived-live).

3. **Dashboard visual animations driven by log streams.** Existing panels — explicitly `Context Flow`, and likely others (the anneal queue from #1316, baton flow, fleet routing, governance audit results) — would benefit from live animation as new log events arrive. Static-snapshot rendering is the current pattern; animation that updates as logs flow is the upgrade.

These three priorities reinforce **G8 Observability** and create a virtuous loop: better-shaped logs → better LLM-driven anneal/critique → better evidence → better governance. They also justify a stronger case for **cross-team Phase-0 R&D** (LLM-format expertise — CC team strength; SSE / web streaming infrastructure — CP team strength; dashboard animation — CP team strength).

## Constraints

- **G3 Zero Cost** — do not propose logging that just sits there. Every recommended addition must have a named consumer + a goal-lens justification.
- **G5 Portability** — log format must be substrate-agnostic. No team-specific structures; substrate-derived identity per `inventory/team-model-signatures.json`.
- **G6 Resilience** — bounded retention; no unbounded growth. Rotation mechanism defined for every append-only surface.
- **G8 Observability** — this Epic is meta-G8. The standard codified must be enforceable and verifiable (lint rules, dashboard signals, audit gates).
- **G4 Privacy** — explicit PII / secret-redaction policy in the standard. Cite `instructions/secret-exposure-prevention` skill conventions.
- **No anneal channel bypass** — per scope-correction (session 2026-05-10), all code changes flow through standard baton.
