# #2444 — Hook gap: roles.admin never auto-set after admin baton complete

> **Source**: github:issue/2444 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P1, area:hooks, lane:code-change, governance:close-without-merge
> Mirror of `gh issue view 2444` (derived; edit the GitHub item, not this page).

## Body

## Goal

Close the hook gap where roles.admin is never auto-set in session state despite successful admin baton completion (post then push then PR then CI green then merge then close).

## Evidence

stop_checks.py check_admin_ops blocks on:

    if flags.get("code_touched") and not roles.get("admin"):
        return ("Stop blocked: Admin role not complete for code session.", ...)

tool_activity.py lines 29-70 only auto-set roles.collaborator. No observed code path sets roles.admin.

This caused operators to repeatedly hit "Stop blocked: Admin role not complete for code session." even after legitimately completing the full admin baton, forcing manual state-file patches (recurring pattern across sessions; cited in memory anchor feedback_state_store_dual_variants).

Discovered 2026-05-30 in Epic #1112 session closeout after PR #2443 merge.

## Acceptance Criteria

- AC1: Identify the legitimate signal point that should flip roles.admin to True
- AC2: Implement the emitter so admin completion auto-marks the state
- AC3: Add regression test that simulates the post-merge sequence
- AC4: Document the signal source in instructions/role-baton-routing
- AC5: Remove documented manual state-patch workaround references from operator memory
