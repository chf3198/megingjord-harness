# #2789 — Prevent concurrent cross-runtime baton collisions on the same ticket

> **Source**: github:issue/2789 | **state: OPEN** | **Labels**: type:epic, status:backlog, priority:P1, role:manager, area:governance, phase-gate:research-first
> Mirror of `gh issue view 2789` (derived; edit the GitHub item, not this page).

## Body

## Problem

Two AI runtimes operating in the same workspace (Claude Code + Copilot, observed 2026-06-09)
can both pick up and execute the **same ticket** concurrently, producing baton collisions:
duplicate baton artifacts, single-status-invariant label thrash, and shared-worktree file
contention. This session reproduced it **three times**:

- **#2136** — Claude resumed a ticket whose entire baton (Manager/Collaborator/Admin) was
  Copilot-signed; both posted a CONSULTANT_CLOSEOUT; labels thrashed `review` vs `done`.
- **#2283** — Claude (Manager) re-scoped the ticket and flipped it to `status:in-progress` +
  `role:collaborator`; the Copilot agent treated that flip as a dispatch signal, picked up
  Collaborator, and implemented the same ACs in the same worktree as Claude.
- Recurrence captured in `~/.megingjord/incidents.jsonl`
  (`pattern_id: concurrent-agent-collision-*`).

This is a **self-anneal Epic** (Tier-3 class): a recurring governance-integrity defect surfaced
by live operation, not a feature request.

## Root cause (pre-research hypothesis — to be validated in Phase-0)

The harness's concurrency guardrails target the wrong threat model for this failure:

1. **Single-operator assumption.** `operator-identity-context` models ONE AI agent running all
   four baton roles single-threaded. There is no first-class model for two *different runtimes*
   contending for the *same* ticket.
2. **The claim is advisory, not atomic.** At Tier-1 (one workspace) the "claim" on a ticket is
   the `role:*` GitHub label. Nothing **atomically rejects** a second runtime that sets/holds a
   `role:*` already held by another runtime. Existing atomic primitives cover *adjacent* events,
   not baton-role acquisition:
   - **merge-claim** (#2458 / #2479) guards the **merge** event only.
   - **cross-team-lease-registry / multi-team leases** (#2366 / #1111 / #2377) guard **resource**
     leases (fleet, subagents), not issue/baton ownership.
3. **Worktree isolation prevents the wrong thing.** The dedicated-worktree rule +
   `canonical_main_enforcer.py` stop `main`-clobbering, but do not stop two runtimes choosing the
   same ticket or ending up in the same worktree.

The gap: **no atomic, cross-runtime claim at the baton-role-acquisition event.**

## Why this is critical (goal-lens)

- **G1 Governance** — concurrent execution breaks the single-active-role invariant and
  baton-artifact integrity.
- **G9 Interoperability** — the harness explicitly supports Claude/Copilot/Codex/Antigravity;
  multi-runtime is a first-class scenario, so multi-runtime safety must be too.
- **G6 Resilience** — collisions must degrade to "one runtime waits", never to corruption.
- **OWASP-agentic OA6 (Rogue Agents) / OA5 (Cascading Failures)** — maps directly; see
  `instructions/owasp-agentic-mapping.instructions.md`.

## Success criteria (validated in Phase-0, refined to AC list ONLY after the gate)

The Phase-0 research child must produce a design in which:

1. A second runtime attempting to acquire a baton `role:*` already actively held by a *different*
   runtime is **detected and prevented** (or made to wait) — deterministically, not by voluntary
   compliance.
2. The mechanism **reuses/extends** existing primitives (merge-claim, cross-team-lease-registry)
   rather than introducing a parallel one — no redundancy (the #2283 lesson, codified).
3. It degrades gracefully (G6): GitHub unreachable / single-runtime ⇒ no added friction; the
   claim falls back to local-state, never hard-fails.
4. It is **runtime-neutral** (G5/G9): identity resolved from `Team&Model`, not hard-coded per
   runtime; no Claude-only coupling.
5. Detection is **observable** (G8): a contended-claim event is emitted to a known surface.
