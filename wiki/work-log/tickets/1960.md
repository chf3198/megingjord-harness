# #1960 — Fix Stop hook + PostToolUse misfire on read-only and remote-only sessions

> **Source**: github:issue/1960 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P2, area:hooks, resolution:completed, lane:code-change

## Body

Parent Epic: #1956

## Problem

The harness Stop hook + PostToolUse "code was touched" signal misfire on sessions that produce only remote state changes (GitHub issue/PR operations, no local file edits) and on read-only diagnostic Bash invocations. The Stop hook appears to sum cumulative PostToolUse signals rather than evaluating actual working-tree state.

## Observed recurrences (single session, 2026-05-19)

1. After 6 read-only diagnostic Bash calls (git status, git log, ls, stat, cat, tail), `.claude/settings.json` was harness-auto-modified with new permission allowlist entries. Stop hook blocked with "uncommitted changes; Admin baton incomplete." Required user-authorized `git restore`.
2. After `git restore` cleaned the tree and the session continued with gh CLI + comments + label edits + fetch (zero local file edits), the Stop hook fired again with "Admin role not complete for code session." Working tree was clean per `git status`. Tier-1 incident event recorded at `~/.megingjord/incidents.jsonl` with `pattern_id=stop-hook-misfires-on-readonly-session`.

## Scope

Fix the Stop hook (`~/.copilot/hooks/scripts/stop_reminder.py` or equivalent) and the PostToolUse code-touched detector so they evaluate actual file mutation, not Bash tool invocation alone.

## Acceptance criteria

- [ ] Stop hook reads `git status --porcelain` at evaluation time; reports clean tree as clean even if PostToolUse fired during the session
- [ ] PostToolUse "code touched" alert distinguishes Bash invocations that mutated files from those that did not (read-only ones like `ls`, `cat`, `git log`, `gh view` should not fire the alert)
- [ ] Read-only gh CLI subcommands (issue view, pr view, label list) explicitly excluded from the heuristic
- [ ] Harness-auto-recorded settings entries either: (a) suppressed unless they correspond to a permission granted this session, or (b) tagged so the Stop hook can distinguish them from genuine user edits
- [ ] Regression test fixture: read-only session of 10+ Bash + gh calls produces no Stop block on a clean tree
- [ ] Companion fixture: a session that actually edits a tracked file DOES produce a Stop block (no false negative introduced)

## Validation evidence

- Replay both fixtures in CI
- Soak: confirm no incident events with `pattern_id=stop-hook-misfires-on-readonly-session` appear for the period after the fix lands
