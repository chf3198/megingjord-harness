# #1905 — feat: ticket-presenter npm script for token-efficient Epic/ticket listing

## Body

## Problem

Operator routinely requests "present all non-complete Epics and independent tickets, sort by priority" (observed 4+ times across the 2026-05-18 session alone). Each invocation currently consumes thousands of tokens:

- GraphQL response for 100 OPEN issues with full bodies (typically 30-50k tokens of body text that gets discarded)
- Python parsing script (multiple turns of script-write + script-output in context)
- Repeated `gh issue view` calls for each prose-Refs parent Epic to determine open/closed state
- Manual markdown-table formatting

Token cost per invocation: ~40-60k tokens of context churn. Across a session this compounds.

## Goal

A single `npm run` invocation that emits the canonical Epic + independent-ticket landscape in markdown, with all the filtering (child-ticket exclusion, parent-state lookup) handled in-script. Caller pastes the script output rather than re-deriving the report from raw API responses.

## Acceptance Criteria

- [ ] AC1: `scripts/global/ticket-presenter.js` accepts CLI flags: `--json` (raw output), `--md` (markdown, default), `--limit N` (issue cap, default 200), `--filter epics-only|independents-only|all` (default all).
- [ ] AC2: Filters out child tickets via native GitHub Sub-issues primitive (GraphQL `parent { number }`) AND legacy prose `Refs Epic #N` (with parent-Epic state check — only filter if parent is OPEN).
- [ ] AC3: Sorts Epics by priority (P0 > P1 > P2 > P3) then status (in-progress > review > triage > backlog > dormant > deferred). Independents by priority then status then number.
- [ ] AC4: Output includes: section headers, markdown tables (`# | Pri | Status | Title` for Epics; `# | Status | Type | Title` for Independents grouped by priority), total counts.
- [ ] AC5: Notable observations surfaced: orphan-children (parent Epic CLOSED), no-status tickets, conflicting labels (e.g. multi-status, role+terminal-status pairs).
- [ ] AC6: npm script `governance:list-tickets` invokes the presenter.
- [ ] AC7: Tests in `tests/ticket-presenter.spec.js`: filter logic, sort logic, child-detection, markdown output snapshot.
- [ ] AC8: Stress test (adversarial corpus): 1000 synthetic issues; perf budget p99 < 2s.
- [ ] AC9: Wired into harness:self-test registry as regression check.

## Why P1 (highest priority)

Token-cost saving on a recurring operator request. Per Epic #1792 (G3 Zero-Cost): minimize provider tokens. This script converts a 40-60k-token-per-invocation pattern into a ~5k-token-per-invocation pattern. ROI is immediate and durable.

## Test strategy

`tdd-pyramid+stress-test` per Epic #1875 (adversarial-input parser + perf budget surface).

## Composition

- `gh api graphql` for issue list with labels + body + parent (single query, paginated)
- Memory-only filtering (no per-issue `gh issue view` round-trips except for prose-Refs parent-state cache)
- Epic #1631 Sub-issues primitive (preferred over prose-Refs)
- Composes with `governance:role-baton-audit` style npm script convention
