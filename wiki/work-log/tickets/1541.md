# #1541 — Race: post-merge-automation overrides label-lint auto-transition

## Body

## Problem

Continuation of #1515 — that fix handled label-lint reopening merge-closed issues, but a SECOND race remains:

1. PR merges → issue auto-closes with `status:testing` + `role:admin`.
2. `label-lint.yml` fires on issue.closed → my #1515 `decide()` returns `auto-transition` → removes status:testing + role:consultant, adds status:done + resolution:completed. ✓
3. `post-merge-automation.yml` fires on pull_request.merged → reads issue labels (still has status:testing from race), removes status:* and role:admin, adds **status:review + role:consultant**. ✗ OVERWRITES label-lint's auto-transition.

Net result: issue closes with `status:review` + `role:consultant` remaining, requiring manual cleanup.

## Reproduction (verified on #1279)

```
22:46:42  PR #1538 merge
22:46:50  issue auto-closes
22:47:xx  label-lint runs, applies status:done + resolution:completed
22:47:xx  post-merge-automation runs, overwrites with status:review + role:consultant
22:48:xx  human (me) manually re-applies status:done + resolution:completed
```

Same pattern observed on every recent merge: #1521, #1374, #1536, #1279.

## Hypothesis

`post-merge-automation.yml` `consultant-activation` job uses `pull_request.closed` payload labels, which is a snapshot from BEFORE my #1515 label-lint mutations. The two workflows don't coordinate.

## Proposed fixes (Path B = recommended)

**Path A:** Have post-merge-automation re-fetch labels before deciding.
**Path B:** Skip the consultant-activation flip when CONSULTANT_CLOSEOUT exists AND status is already terminal.
**Path C:** Move the auto-transition logic into post-merge-automation (single workflow handles both).
