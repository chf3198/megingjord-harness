## Research Questions

### R1 — Exact MCP config file paths (filesystem-verified, all 4 runtimes)

- Claude Code: is it `~/.claude/mcp_config.json`, `~/.claude/settings.json` `mcpServers` key, or something else? Current `settings.json` has no `mcpServers` key.
- VS Code Copilot: `~/.config/Code/User/mcp.json` exists and contains `{}` — confirmed. What is the required JSON key (`servers` vs `mcpServers`)?
- Codex: `~/.codex/config.toml` uses `[mcp_servers.name]` blocks — confirmed. Does it also support a separate `~/.codex/mcp_config.json`?
- Antigravity: `~/.config/Antigravity/User/` exists (separate from VS Code). No `mcp.json` yet. What path does Antigravity expect for MCP configs?

### R2 — Exact schema format per runtime

For each runtime, what is the full required structure of one registered MCP server entry? (key names, command/args vs commandString, env object, stdio/sse transport indicator)

### R3 — Idempotent merge strategy

How should `mcp-register.js` merge the xteam entry into existing configs without:
- Duplicating existing entries (incl. Codex's openaiDeveloperDocs + tavily)
- Corrupting format (JSON beautification vs compact, TOML block ordering)
- Overwriting unrelated keys in JSON (e.g. Claude Code's hooks/permissions)

### R4 — Deploy script factoring

`deploy.sh` is at exactly 100 lines (lint hard limit). Options:
- Option A: `scripts/global/xteam-mcp-register.js` — called once with `--target all`
- Option B: `scripts/global/mcp-register.js` — generic MCP registration, not xteam-specific (better for future MCP servers)
- Option C: Add 4 one-liners per target section in `deploy.sh` calling a node helper inline

Which option best satisfies G2 Quality + G10 Maintainability + 100-line constraint?

### R5 — Testing approach

How should the registration be tested?
- Unit: mock filesystem; verify idempotent write
- Integration: actual config write + read-back verification
- What does success look like in CI where `~/.claude/` etc. don't exist?
