## Origin

Conversation thread (2026-05-27) during Phase-0 work on Epic #2295. The Client (chf3198) surfaced two role-taxonomy clarifications and one architectural direction:

1. **Consolidation**: "Guest-Collaborator is effectively the same as the Red-Team. We can probably reduce them to just the Red-Team."
2. **IT role contract**: "The IT role is for jobs that do not require work in the GitHub repo — no tickets, branching, commits, commenting — as the IT role works on setup and configuration of the user unique resources (fleet hardware and services)."
3. **Architectural direction**: "Ideally, you alone would not be the singular source for Agentic work in any of these roles. In fact it would be best if the harness eventually enforces fleet resources to do the work of every role where possible."

This Epic captures the conversation, ratifies the decisions, and tracks the refactor work to align the harness with the agreed taxonomy.

## Current defined role taxonomy (pre-refactor, 8 roles)

| Role | Scope | Authority |
|---|---|---|
| Manager | scope/plan/gates (Agile workflow) | sets ACs, files tickets, baton routing |
| Collaborator | implement/validate | produces deliverables per Manager scope |
| Admin | git/release/runtime ops | merge, publish, sync verification |
| Consultant | independent critique | rubric + closeout authority |
| IT | setup + config of fleet hardware/services | non-GitHub work (no tickets/branches/commits) |
| Guest-Collaborator | external/fleet model attribution | one-shot collaboration via dispatch |
| Red-Team | adversarial review (cross-family) | structured critique with hallucination-risk + score-honesty |
| Client | design direction + UAT | overrides Consultant atomic-close authority where carved out |

Additionally, `instructions/operator-identity-context.instructions.md` defines **Operator** as a meta-term for the AI agent that wears the four baton roles in single-thread sequence. Operator is NOT a distinct role; it is the entity-identity that takes on Manager/Collaborator/Admin/Consultant in turn.

## Ratified decisions (Client-set; this Epic implements them)

### Decision D1 — Consolidate Guest-Collaborator into Red-Team

Guest-Collaborator was historically used for "external/fleet model dispatch attribution" (e.g., "qwen2.5-coder:32b as guest-collaborator"). Red-Team was used for "adversarial review specifically". In practice, every Guest-Collaborator dispatch in the harness has been a Red-Team dispatch. Maintaining both creates taxonomy ambiguity.

**Resolution**: collapse Guest-Collaborator into Red-Team. Going forward, fleet-model dispatches for adversarial review are attributed as **Red-Team** (with model + host metadata). The historical "guest-collaborator" label/term is retired.

Edits required:
- `instructions/role-baton-routing.instructions.md` — remove Guest-Collaborator references
- `instructions/team-model-signing.instructions.md` — update signer-attribution conventions for Red-Team
- Any baton-validator scripts that recognize Guest-Collaborator strings — recognize Red-Team instead
- Update operator-local memory + Claude Code Team agent memory references

### Decision D2 — IT role contract clarified

The IT role is for setup + configuration of **fleet hardware and services** — non-GitHub work that does NOT require tickets, branches, commits, or comments. Current harness mentions IT via the IT-ops bypass markers (`MEGINGJORD_IT_OPS=1` env, `[it-ops]` subject literal, `chore(it-ops):` Conventional-Commits prefix). The bypass exists for commits that touch tracked files but don't warrant Agile baton workflow.

**Resolution**: formalize the IT role contract:
- IT scope: fleet hardware (the `36gbwinresource` host, Tailscale mesh, MCP server provisioning, Ollama model management, devbox environments)
- IT scope: services (HAMR activation, dashboard pid, cron schedules, hook installation)
- IT does NOT: create tickets, push branches, commit code, comment on issues
- IT bypass markers remain valid for the rare case where tracked-file edits are unavoidable (e.g., updating a fleet-host IP in a config schema)

Edits required:
- `instructions/global-standards.instructions.md` — clarify IT-ops bypass is the IT role's documented escape hatch, not a general bypass
- New `instructions/role-it-execution.instructions.md` — formal IT role contract (scope + boundaries + dispatch protocol)
- Update `skills/role-*` skill listing to include the IT role explicitly

### Decision D3 — Prefer fleet-resource execution for every role where possible

**Architectural direction**: the harness should eventually enforce that fleet resources (local Ollama, fleet-host GPU, OpenRouter free tier, HAMR routing) execute role work wherever the role's deliverable is producible without Claude/GPT-class reasoning.

Implication: a single AI agent (me, Claude Code) should NOT be the singular source for all four baton roles plus Red-Team plus IT. The right pattern is:
- **Manager**: scope work currently requires high-reasoning (deferred from fleet for now); migrate to fleet where ACs and gates are templatable
- **Collaborator**: many implementation tasks (boilerplate, codegen, refactor) are fleet-executable today
- **Admin**: git/release ops are scriptable; fleet-executable today via fleet-coding-local lane
- **Consultant**: rubric scoring + critique is fleet-executable (Red-Team pattern already proves this)
- **Red-Team**: already fleet-executed by contract
- **IT**: already non-AI / scripted; not applicable

Resolution: this Epic doesn't itself migrate all roles to fleet — that's substantial work. It DOES file Phase-1 children to:
- Document the per-role fleet-executability assessment
- Identify the migration order by impact × feasibility
- Land enforcement mechanisms (HAMR routing + lane policy) that prefer fleet wherever possible
