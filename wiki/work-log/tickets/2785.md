# #2785 — Provision canonical label taxonomy on onboarded repos (lint enforces labels onboarding never seeds)

> **Source**: github:issue/2785 | **state: CLOSED** | **Labels**: type:task, status:done, priority:P2, area:governance, resolution:released, lane:code-change, governance:close-without-merge
> Mirror of `gh issue view 2785` (derived; edit the GitHub item, not this page).

## Body

## Problem

The harness **enforces** a role/status/area label taxonomy via label-lint, but **never provisions** that taxonomy onto the repos it onboards. Result: a fresh onboarded repo has only an ad-hoc subset of labels, and any agent following the baton lifecycle must hand-create the missing labels mid-ticket (or the work silently violates lint).

### Evidence

- `scripts/global/label-rules.js` — the single-source-of-truth lint evaluator — declares `STATUS_ROLE_REQUIRED = { 'status:in-progress': 'role:collaborator', 'status:testing': 'role:admin', 'status:review': 'role:consultant' }`. These role labels are **required** for those statuses.
- No provisioner exists: `grep -rn "gh label create" scripts/` returns **zero** matches; there is no canonical label-seed manifest; and `skills/repo-onboarding-standards/SKILL.md` never mentions labels.
- Real-world hit (2026-06-08): in `chf3198/receipts-api-mcp`, only `role:manager` existed. A Manager-role ticket transition (`status:ready` + `role:collaborator`) **failed** with `'role:collaborator' not found`; the agent had to manually `gh label create role:collaborator|role:admin|role:consultant` before it could hand off. See receipts-api-mcp#33.

### Why this matters (G1 Governance)

The baton lifecycle (Manager→Collaborator→Admin→Consultant) is non-negotiable, yet its substrate (the labels) is not guaranteed to exist where the lifecycle runs. Every newly onboarded repo silently inherits this gap until an agent trips over it.

## Proposed solution

1. Define a **canonical label manifest** (single source of truth) covering `type:*`, `status:*`, `priority:*`, `area:*`, `role:*`, and `lane:*` — with name, color, description. This same manifest should feed `label-rules.js` so enforcement and provisioning can never drift.
2. Add an **idempotent provisioner** (e.g. `scripts/global/label-provision.js`, `gh label create --force` / GraphQL) that reconciles a target repo to the manifest (create missing, optionally update color/description; do not delete unknown labels by default).
3. Wire provisioning into **repo onboarding** (`repo-onboarding-standards` skill + first-session bootstrap) so any onboarded repo is reconciled automatically.

## Acceptance Criteria

- [ ] AC1: A single canonical label manifest exists and is the source consumed by both `label-rules.js` enforcement and the provisioner (no duplicated label lists).
- [ ] AC2: Idempotent provisioner reconciles a target repo to the manifest; re-running is a no-op.
- [ ] AC3: Onboarding (skill + bootstrap path) invokes the provisioner so a fresh repo has the full taxonomy before any baton transition.
- [ ] AC4: Regression check — a repo with only `role:manager` (the receipts-api-mcp starting state) ends up with the complete `role:*`/`status:*`/`area:*` set after provisioning.
- [ ] AC5: Docs updated (onboarding skill + governance instructions) to reference the manifest as the source of truth.

## Out of Scope

- Backfilling labels on every existing repo in one shot (provisioner enables it; bulk rollout is a follow-up).
- Destructive cleanup of non-canonical labels (separate concern; `coordinator-label-cleanup.js` already exists).
