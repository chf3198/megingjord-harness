## Goal

Identify which of the original #1472 hypotheses (A/B/C) explains why the auto-transition silently fails on CONSULTANT_CLOSEOUT-triggered close events.

## What to do

After observing a real close event WITH the new diagnostic log from #1472 active, look at the workflow output for the `#<N>: close-protection decision=<action> reason=<reason> (#1472)` line:

- If decision.action === 'auto-transition' but labels did NOT change → API call failure swallowed by `.catch(() => {})`. Likely token permission scope issue. Fix: remove the .catch swallow OR check return values.
- If decision.action === 'noop' reason='issue-open' → workflow runs before close transition propagates. Fix: poll for state=closed before deciding, OR use the issue payload directly instead of fetching.
- If decision.action === 'noop' reason='already-terminal' → another workflow ran first and added status:done. Fix: not a bug — this is the right behavior. Investigate which workflow.
- If decision.action === 'reopen' → closeout regex didn't match. Fix: update CLOSEOUT_HEADER_RE.
