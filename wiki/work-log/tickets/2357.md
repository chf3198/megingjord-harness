## Key Finding

**The harness governance model is effectively opt-in for lower-capability models.** All enforcement that the Auto model missed was post-hoc or asynchronous:
- Lint catches line-count violations but not missing GitHub artifacts
- CI gates catch test failures but have no issue-state awareness
- Pre-commit hooks catch path/branch violations but not baton sequence gaps
- The canonical-main enforcer is a write-protection guard, not a compliance gate

A sufficiently capable model follows the rules because it "knows" the instructions. A lower-capability model completes the task as scoped without following the meta-process. The harness has no layer between these.

## Research Scope

This ticket must answer:

1. **Taxonomy of enforceable vs advisory controls** — for each required governance step (baton artifact, changelog, cross-link, test_strategy), classify: (a) currently blocked by a mechanical gate, (b) currently advisory only, (c) entirely absent from enforcement.

2. **Root cause analysis of the enforcement gap** — why does the current hook architecture not catch missing baton artifacts? What would be required to add that capability?

3. **Capability threshold mapping** — at what model capability level does voluntary compliance with the harness begin to break down? What evidence exists from other sessions?

4. **Prior art survey** — do other AI governance frameworks (e.g., Claude Code system prompts, Codex AGENTS.md validation, OpenAI function calling guards) have mechanisms that the harness could adapt?

5. **Risk register** — what governance violations are possible in a low-capability session that could produce irreversible side effects (e.g., a false-compliant merge that breaks a downstream system)?
