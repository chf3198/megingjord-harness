# #2995 — Prevent tracked main edits during IT workflows

> **Source**: github:issue/2995 | **state: CLOSED** | **Labels**: type:task, status:done, priority:P1, area:hooks, governance:close-without-merge
> Mirror of `gh issue view 2995` (derived; edit the GitHub item, not this page).

## Body

## Problem
A recent optimization was edited directly in the canonical main checkout on tracked files instead of an isolated ticket branch/worktree. This created avoidable coordination risk across teams and violated the intended IT-vs-delivery boundary.

## Objective
Add self-annealing guardrails so no team can repeat this mistake during IT-type work.

## Acceptance Criteria
- Add a hard pre-tool guard that blocks tracked file edits in canonical main unless the target path is gitignored.
- Add an explicit, machine-readable remediation message that instructs routing to a ticket branch/worktree.
- Add a policy check that fails if IT markers are used while editing tracked files.
- Add tests that cover:
  - tracked path edit in canonical main is denied,
  - gitignored path edit in canonical main is allowed,
  - IT bypass marker does not authorize tracked-file edits.
- Add documentation for teams describing the correct path:
  - IT local config in gitignored files only on main,
  - all tracked work requires issue + branch/worktree.

## Self-Annealing Context
This is a process-hardening follow-up to prevent recurrence of the same workflow failure pattern across Copilot, Claude Code, Codex, and Antigravity teams.

## Verification Plan
- Focused unit tests for pretool guard behavior.
- Demonstrate denial/allow decision traces for canonical-main path cases.
- Document rollout and expected operator behavior.
