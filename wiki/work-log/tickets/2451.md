# #2451 — Collapse local state cache; GitHub-as-event-source for baton

> **Source**: github:issue/2451 | **state: CLOSED** | **Labels**: type:epic, status:done, priority:P1, area:governance, resolution:released, lane:docs-research, phase-gate:research-first
> Mirror of `gh issue view 2451` (derived; edit the GitHub item, not this page).

## Body

## Goal

Collapse the local `roles` dict and `admin_ops` cache in `~/.copilot/hooks/state/` in favor of **GitHub-as-event-source** for the baton state. Where local persistence is genuinely needed (cost telemetry, observability), move to an **append-only event log** rather than mutable state files.

## Problem

Megingjord's baton state currently exists in two places:

1. **Canonical**: GitHub issue labels + 4 named handoff artifacts (`MANAGER_HANDOFF`, `COLLABORATOR_HANDOFF`, `ADMIN_HANDOFF`, `CONSULTANT_CLOSEOUT`)
2. **Local cache**: `~/.copilot/hooks/state/repo-<hash>[-session].json` with `roles{}`, `flags{}`, `admin_ops{}`

These two stores drift. The drift produces:

- "Stop blocked: Admin role not complete for code session" false-positives (root cause of #2444, closed today)
- "Admin baton incomplete" false-positives from dual-file pattern (memory anchor feedback_state_store_dual_variants, recurring class)
- Manual state-file patching that the auto-mode safety classifier (rightly) blocks
- Cross-team residue when state files are shared across sessions (the original driver of #2091)

The local cache duplicates information GitHub already holds authoritatively. The cache is the source of brittleness, not the mechanism that prevents it.

## Industry context (2026)

Web research grounded this recommendation (full sources cited in Phase-0 child):

- **ESAA paper** (arXiv 2602.23193, Feb 2026): "Event Sourcing for Autonomous Agents in LLM-Based Software Engineering" — separates cognitive intention from project state mutation
- **Orchestrator-worker pattern** (~70% of 2026 production deployments): single coordinator, no parallel state. Megingjord today is "orchestrator-worker with a stale local cache" — gets neither benefit
- **State management is the primary technical challenge across every production multi-agent account** (Augment Code 2026 survey)

## Proposed architecture (3 moves)

### Move 1 — Make local state file an ephemeral read-cache (60s TTL)

`stop_reminder.py` and `check_admin_ops` derive `roles{}` and `admin_ops{}` from a fresh `gh issue view N --json labels,comments` query, with a 60s TTL cache for performance. No writes to the local state from baton transitions.

**Closes:** entire dual-file/`roles.admin`/stale-state failure class (the #2444 problem class).

### Move 2 — Event-source the baton for audit and replay

Append every baton transition to immutable `~/.megingjord/baton-events.jsonl` (schema v3 per `instructions/observability.instructions.md`). Use for replay/audit/eval-harness, never for live gate decisions.

**Closes:** ad-hoc state-file forensics; matches OWASP OA9 (Human-Agent Trust Exploitation) audit requirement.

### Move 3 — Add a thin cross-team control plane in HAMR Worker

When 4 teams (Claude Code + Codex + Copilot + Antigravity) all push PRs targeting main, race conditions surface as "Base branch was modified" GraphQL errors (hit during this session). Add a merge-claim primitive to HAMR (Cloudflare KV with TTL) so Admin roles across teams serialize on intent rather than discovering races after the fact.

**Closes:** silent merge-race retries; G7 throughput cost class.

## Pattern-specific design tradeoffs (added iter-1)

| Pattern | Considered? | Adopted? | Rationale |
|---|---|---|---|
| LangGraph time-travel checkpointer | Yes | No | Excellent for in-flight agent state, but overkill for baton transitions which are already linearized at the GitHub-issue layer. Adopting it would add a second persistence store (the failure class we're closing). |
| MAF 1.0 workflow events + checkpoints | Yes | Partial | Move 2 (event-source baton-events.jsonl) mirrors MAF's "events baked in" pattern at a lighter weight (JSONL vs. graph-native workflow runtime). Full MAF adoption would replace HAMR — out of scope. |
| A2A v1.2 signed agent cards | Yes | Yes (Move 3) | Move 3's merge-claim acquire/release MUST verify team identity. A2A signed cards (Ed25519 over DPoP) match HAMR's existing crypto stack. Implementation child #2458 will adopt A2A card format. |
| MCP 2026 stateless + explicit handles | Yes | Yes (Move 1) | Move 1's `derive_roles_from_github(ticket_n)` mirrors the MCP "explicit handles" pattern — ticket number IS the handle; cache holds no hidden session state. |
| Cloudflare KV with strongly-consistent reads | Yes | Yes (Move 3) | HAMR already on Cloudflare; KV's strong-consistency mode (2026 GA) gives lock-equivalent semantics without Paxos/Raft complexity. |
