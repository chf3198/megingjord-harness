# #2812 — validate-config-schemas required check skips on non-schema PRs, blocking all merges

## Body

## Problem (self-blocking required check — GitHub anti-pattern)
`validate-config-schemas` is a **required** status check, but its workflow uses a workflow-level `paths:` filter (`.claude/**`,`config/**`,…). Any PR that does NOT touch those paths SKIPS the workflow → the required check never reports SUCCESS → **GitHub branch protection blocks the merge** (mergeStateStatus BLOCKED). This blocks EVERY PR not touching config schemas (e.g. PR #2807, a hooks fix). Per GitHub docs: do not path-filter a required workflow.

## Fix (GitHub-recommended: always-run + step-level skip-pass)
Remove the workflow-level `paths:` filter; always run the job on PRs to main; a first step detects whether schema-relevant files changed (git diff) and the validation steps run only when they did — otherwise the job passes trivially. The check thus ALWAYS reports SUCCESS, validating for real only when relevant.

## Acceptance criteria
- AC1: `validate-config-schemas` runs on every PR to main and reports SUCCESS (no skipped-required state).
- AC2: real schema validation still runs when `.claude/**`/`config/**`/etc. change (step-level gate).
- AC3: no new third-party action dependency (pure git-diff step) — keeps the pin-to-SHA posture.
- AC4: this PR (editing the .yml, which is in the path list) triggers the check → green → mergeable normally.
