# #2067 — Annotate plan() report with registryError when lease registry is unavailable

> **Source**: github:issue/2067 | **state: CLOSED** | **Labels**: type:task, status:done, priority:P3, area:scripts, resolution:completed, lane:code-change

## Body

## Problem

`branch-cleanup-plan.js`'s `plan()` returns `orphanedLeases: []` in two
semantically distinct situations:

1. **Registry read succeeded and found no orphaned leases** — safe signal
2. **Registry threw (file missing, malformed, locked)** — unknown state; the
   caller cannot distinguish this from situation 1

A caller that treats `orphanedLeases.length === 0` as a safe-to-proceed signal
will silently proceed with incomplete information when the registry is unavailable.

## Expected Outcome

When `orphanedLeases()` catches a registry error, `plan()` should include a
`registryError` field in its return object so callers can detect the degraded
state:

```js
// registry OK → no registryError field
{ mode: 'dry-run', orphanedLeases: [], branches: [...] }

// registry failed → registryError field present
{ mode: 'dry-run', orphanedLeases: [], branches: [...], registryError: 'ENOENT: ...' }
```

The `run()` console output should also surface a visible warning when
`registryError` is present.

## Acceptance Criteria

- AC1: `plan()` report includes `registryError: <message>` when registry throws
- AC2: `plan()` report has NO `registryError` field when registry succeeds
- AC3: `run()` console output prints a warning line when `registryError` present
- AC4: Test updated/added to verify AC1, AC2, AC3
- AC5: All existing tests still pass; line limits maintained (≤100 lines per file)
