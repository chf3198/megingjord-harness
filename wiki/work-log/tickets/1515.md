# #1515 — Race: label-lint auto-reopens tickets closed via merge Closes #N trailer when status:testing

> **Source**: github:issue/1515 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P1, area:governance, resolution:completed, lane:code-change
> Mirror of `gh issue view 1515` (derived; edit the GitHub item, not this page).

## Body

## Problem

`.github/workflows/label-lint.yml` auto-reopens issues that GitHub closes via PR-merge `Closes #N` trailer. Pattern confirmed across #1506 / #1508 / #1512 (all my recently-merged Epic #1486 children).

### Reproduction (from #1506 event timeline)

```
21:20:56  chf3198             closed (via Closes #1506 in PR #1507 merge)
21:21:05  github-actions[bot] REOPENED (9 seconds later)
21:21:06  github-actions[bot] unlabeled  role:admin
21:21:10  github-actions[bot] unlabeled  status:testing
21:21:11  github-actions[bot] labeled    role:consultant
21:21:11  github-actions[bot] labeled    status:review
```

### Root cause

`label-lint.yml` fires on `issues.closed`. Its close-protection branch checks:

```js
if (issue.state === 'closed' && !hasTerminal) {
  // ...has CONSULTANT_CLOSEOUT comment + status:review label?
  if (hasCloseout && inReview) {
    // auto-transition status:review → status:done — OK
  } else {
    // REOPEN + post "Close blocked" comment
    await github.rest.issues.update({ state: 'open' });
  }
}
```

The auto-transition path requires **both** `CONSULTANT_CLOSEOUT` AND `status:review`. But the auto-close-via-merge path fires while the issue is at `status:testing` (since `post-merge-automation` hasn't flipped it yet — different workflow trigger, race condition). So `inReview === false` → reopen branch fires.

`post-merge-automation` triggers on `pull_request.closed/merged=true`. `label-lint` triggers on `issues.closed`. The issue close event arrives at label-lint **before** post-merge-automation has finished transitioning status:testing → status:review.

## Fix

Accept `status:testing` as a valid pre-close state in the auto-transition path. When CONSULTANT_CLOSEOUT exists AND the issue closed from either `status:review` OR `status:testing`, transition to `status:done` + `resolution:completed`.

This widens the auto-transition's accepting set; nothing currently in the harness depends on rejecting `status:testing` closures.
