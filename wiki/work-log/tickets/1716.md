# #1716 — Enforce no-self-review via model rotation across baton roles

> **Source**: github:issue/1716 | **state: CLOSED** | **Labels**: type:epic, status:done, priority:P2, area:governance, resolution:released

## Body

## Epic Summary

Establish a hard contract that no LLM-driven artifact in the harness baton flow is reviewed by the same model that produced it. Extends **model-diversity-advisory** (#1572 / Epic #1568) from per-critical-path to per-step rotation, and from advisory to required.

## Problem

The current harness runs an advisory gate on critical-path diversity (Admin ≠ Collaborator team, Consultant ≠ Admin team). Two gaps remain:

1. **Within-role review is unconstrained.** A Collaborator that pauses mid-implementation to "review what I just wrote" runs that review on the same model. Self-preference bias is documented in 2026 literature: linear correlation between self-recognition capability and self-preference strength; root cause is lower perplexity for own-style text.
2. **Critical-path diversity is advisory.** Operators may waive via `model-diversity:waived` label without a rotation alternative.

## Rotation contract (research-confirmed direction)

Three rules, applied across baton transitions:

- **Rule 1 — Collaborator self-review**: `COLLABORATOR_SELF_CHECK` artifact must use a different model than the implementation model. **Within-family-different-version is NOT acceptable** (research-confirmed: same-family stylistic overlap defeats independence).
- **Rule 2 — Admin diversity**: Admin uses a model not appearing in any Collaborator or Manager step on the ticket. **Strict cross-family** required.
- **Rule 3 — Consultant fully independent**: Consultant uses a model not appearing in any prior step on the ticket (Manager + Collaborator + Admin).

## Validated design direction (from dual-model research)

Dual-model research with Claude Opus 4.7 (live WebSearch) + Gemma3:1b (cross-family validation) confirmed:

1. **Same-family-different-version is NOT acceptable substitute** — Phase 2 design requires true cross-family rotation.
2. **Crypto-signing moves from optional to default** within 2 phases — existing `Crypto-Signature` field promoted.
3. **Cold-start cost is manageable** (~250ms FlashBoot-class infra; semantic caching absorbs most rotation cost). NOT a prohibitive blocker.
4. **HAMR is the natural enforcement point** — it already tracks Team&Model per call; the validator lives HAMR-side, not pre-push.

Full dual-model comparison artifact: see comment [#issuecomment-4466325376](https://github.com/chf3198/megingjord-harness/issues/1716#issuecomment-4466325376).

## Risk register

| Risk | Mitigation |
|---|---|
| Forced rotation pushes every Consultant to Premium tier (cost blowup) | HAMR cost-routing stays canonical; Phase 2 design includes rotation-aware lane policy |
| Same-family-different-version slipping through as "good enough" | Phase 2 contract explicitly defines "family" via `team` field of `Team&Model` (e.g., anthropic vs openai vs google) |
| Spoofed Team&Model signatures defeat the contract | Crypto-signing default-on (Child 3.3) makes spoofing detectable; Phase 1.2 also investigates fingerprinting |
| Air-gapped + single-model operators silently bypass | Operator declares `single-model-fleet` classification at activation; opt-out is explicit not implicit |
| Rotation invalidates prompt-cache too aggressively | Phase 3 implementation uses semantic-cache cushion per Anthropic/OpenAI cache adapters (#926 family) |

## Out of scope

- Building a NEW provider — uses existing HAMR-routed providers.
- Mandating crypto-signing as the ONLY enforcement path — fingerprinting (Phase 1.2) is the secondary path for non-crypto operators.
- Retroactive enforcement on closed tickets — forward-looking only.
- Replacing #1572 advisory entirely until the v2 required-mode lands — both run in parallel during Phase 3-4 soak.
- Cross-runtime intra-team model rotation (Claude Code claude-code:opus vs Claude Code claude-code:sonnet) — research-confirmed within-family is not acceptable, but the harness's existing `team:model@substrate` signature lets the rotation rules treat these as the same `team`; Phase 2.1 contract spec resolves this explicitly.
