# #2707 — Enforce mandatory parallel technical writing as a hard Collaborator gate

> **Source**: github:issue/2707 | **state: OPEN** | **Labels**: type:epic, priority:P1, role:manager, area:knowledge, area:governance, status:dormant
> Mirror of `gh issue view 2707` (derived; edit the GitHub item, not this page).

## Body

## Problem

Every code change that ships through the baton workflow should be accompanied by
parallel technical writing. The harness has the plumbing (`doc-coverage-matrix.yml`,
`doc-coverage.js`) but it is **advisory-only** — no PR is blocked for missing
documentation, `N/A` passes with any freeform reason, and there is no diff-based
verification that declared surfaces were actually updated.

The result: significant documentation debt has accumulated across README,
architecture docs, developer guides, AGENTS.md, wiki pages, and howto docs. New
exports and capabilities ship without any developer-facing documentation.

## Root cause

1. `DOC_COVERAGE_GATE_ADVISORY=1` is the default — the gate warns but never fails.
2. The `doc-coverage-matrix.yml` matrix is incomplete: `area:scripts` requires only
   `README.md`; new public exports, new modules, and new CLI flags have no surface
   requirements. No requirement for design docs, architecture docs, or developer
   guides on new features.
3. `N/A` passes with any non-empty string — "no user-visible CLI change" is accepted
   even when a new exported API is introduced.
4. The validator checks that a surface is *mentioned* in the handoff block but does
   not verify the file was actually modified in the diff.
5. No approved enumeration of valid N/A reasons forces surface acknowledgment to be
   deliberate rather than perfunctory.

## Objective

Design and implement a **mandatory parallel technical writing contract** enforced
at the Collaborator gate for every `lane:code-change` PR. Zero PRs merge without
either documented updates or a governed N/A with a reason from an approved set.

## Expected outcome

- `doc-coverage` is a **hard CI gate** (not advisory) for all `lane:code-change` tickets.
- `doc-coverage-matrix.yml` comprehensively maps every `area:*` label to all applicable
  documentation surfaces (README, architecture docs, howto guides, wiki pages, AGENTS.md,
  CONTRIBUTING, developer guides for new APIs).
- `N/A` reasons are validated against an approved enumeration defined in the matrix or
  a companion config file. Freeform reasons fail the gate.
- Diff-based verification: if a surface is declared `UPDATED`, CI can optionally verify
  the file appears in the PR diff (stretch goal / Phase-1 child).
- The Collaborator role description and MANAGER_HANDOFF schema explicitly require
  doc-coverage as a baton gate prerequisite, not an afterthought.
- This Epic's own implementation children demonstrate compliance: every child PR carries
  a complete `doc-coverage:` block and updates all applicable surfaces.

## Acceptance criteria

### Phase-0 — Research (must close before any Phase-1 children open)

- **AC-R1**: Inventory every documentation surface the harness currently maintains
  (README, CHANGELOG, AGENTS.md, CLAUDE.md, .codex/AGENTS.md, CONTRIBUTING, SECURITY,
  ARCHITECTURE, docs/howto/*, docs/workflow/*, wiki/wisdom/*, wiki/code/*, skills/*/SKILL.md,
  instructions/*). Produce a structured manifest with current freshness, owner, and
  last-meaningful-update evidence.
- **AC-R2**: Audit `doc-coverage-matrix.yml` against the manifest from AC-R1. For each
  `area:*` label, identify: (a) surfaces currently in `required` that are correctly scoped,
  (b) surfaces missing from `required` that should be required, (c) surfaces that should
  move from `suggested` to `required`.
- **AC-R3**: Design the governed N/A reason enumeration. Define the complete approved set
  (e.g., `no-public-api-change`, `no-user-facing-behavior`, `docs-only-no-functional-change`,
  `covered-by-sibling-pr:<N>`). Document when each is legitimate and what evidence it
  requires.
- **AC-R4**: Research and spec the diff-based surface-update verification approach: how to
  confirm that a surface declared `UPDATED` appears in the PR diff, handling symlinks,
  indirect changes, and generated files. Produce a design doc with pass/fail semantics.
- **AC-R5** (this AC): Synthesize Phase-0 into a MANAGER_HANDOFF-ready rescope comment
  on this Epic with Phase-1 child AC definitions and priority order.

### Phase-1 — Implementation (gated on all Phase-0 ACs closed)

- **AC1**: Promote `doc-coverage` from advisory to hard gate: remove
  `DOC_COVERAGE_GATE_ADVISORY` escape hatch from `.github/workflows/baton-gates.yml`
  and from `doc-coverage.js` default path. Replace with an explicit `LEGACY_DOC_SKIP`
  env var that requires a `BLOCKER_NOTE` on the linked issue to activate.
- **AC2**: Expand `doc-coverage-matrix.yml` with all surfaces identified in AC-R2,
  covering at minimum: new exported functions → `docs/howto/<feature>.md` required;
  new CLI flags → README.md required with actual flag documentation; new config keys →
  AGENTS.md + CLAUDE.md required; architecture changes → ARCHITECTURE.md required.
- **AC3**: Implement N/A reason enum validation in `doc-coverage.js`: approved reasons
  load from a `config/doc-coverage-na-reasons.json` file; unrecognized reasons fail
  with a structured error naming the closest valid reason.
- **AC4** (stretch): Implement diff-based surface update verification in a companion
  `doc-coverage-diff.js` validator. Runs as an advisory initially with promotion gate.
- **AC5**: Update `instructions/role-baton-routing.instructions.md` §Tech-Writer
  sub-phase to require the complete `doc-coverage:` block format with N/A reasons
  from the approved set. Promote from advisory to required in the instructions prose.
- **AC6**: 8+ unit tests in `tests/doc-coverage.spec.js` covering: hard gate blocks on
  missing block, approved N/A reasons pass, unapproved N/A reasons fail, UPDATED
  surfaces pass, diff-check advisory fires correctly.

## Non-goals

- Retroactive documentation of historical code (that is Epic #CATCH-UP).
- Documentation for research/config-only lanes (those are exempt by design).
- Auto-generating documentation content (the gate enforces existence, not quality).
