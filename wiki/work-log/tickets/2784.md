# #2784 — Research and plan AC check-off enforcement at Consultant closeout

## Body

## Problem

During a Consultant-role session on **chf3198/receipts-api-mcp** (2026-06-08), a post-hoc audit discovered **55 unchecked acceptance criteria** across **9 closed tickets**. Every ticket had been closed with a CONSULTANT_CLOSEOUT, yet zero ACs had been checked off in the issue body at the time of closure.

This is not a project-specific gap — it is a harness process gap. The baton workflow (Manager → Collaborator → Admin → Consultant) has no step that enforces, or even reminds, that AC checkboxes in the issue body must be marked complete before the ticket can close.

**Tier-2 signal**: this pattern has been observed across multiple sessions and projects. It qualifies as a recurring medium-severity governance gap (G1 violation: AC evidence completeness is non-negotiable per ticket lifecycle contract).

---

## Root Cause (Observed)

The `CONSULTANT_CLOSEOUT` schema and the `consultant-closeout.js` validator check for:
- `verdict:` field
- `rubric_rating:` field
- `anneal_tickets_filed:` field
- `mid_flight_flaws:` field
- Signer-alias fidelity

**They do not check** whether the issue body's `- [ ] AC*` checkboxes are all in `- [x]` state at close time.

The Collaborator phase verifies ACs pass functionally (COLLABORATOR_HANDOFF per-AC block) but does not update the GitHub issue body checkboxes. No subsequent role is assigned this responsibility, so it falls through every phase.

---

## Scope

Research the minimal-friction solution to guarantee AC checkboxes are in `[x]` state before a ticket reaches `status:done` + close. Produce a plan with implementation path and phased rollout recommendation.

**Non-goals**: do not redesign the baton workflow; do not change COLLABORATOR_HANDOFF per-AC verification format.

---

## Acceptance Criteria

- [ ] AC1: Document all current baton phases and identify the correct ownership phase for AC check-off (Collaborator, Admin, or Consultant)
- [ ] AC2: Evaluate whether `consultant-closeout.js` is the right enforcement point vs. a dedicated pre-close validator
- [ ] AC3: Research GitHub API/CLI approach to programmatically flip `- [ ]` → `- [x]` in issue body as part of the closeout script
- [ ] AC4: Define the validator rule: "issue body must have 0 unchecked `- [ ]` items at closeout" with acceptable escape hatches (e.g., intentional deferred ACs with explicit `DEFERRED:` annotation)
- [ ] AC5: Produce an implementation plan (script path, validator change, MANAGER_HANDOFF schema extension if needed) with phased rollout: advisory first, then blocking gate after one soak cycle
- [ ] AC6: Document the escape hatch pattern for genuinely deferred ACs so the validator does not produce false positives
