# #2972 — Fix hamr:activate cyclic symlink that ELOOP-bricks all tool calls

> **Source**: github:issue/2972 | **state: OPEN** | **Labels**: type:bug, status:backlog, priority:P1, area:hooks, area:infra, lane:code-change
> Mirror of `gh issue view 2972` (derived; edit the GitHub item, not this page).

## Body

## Severity: HIGH (operator lockout)

## Symptom
After running `npm run hamr:activate` a second time (to refresh the fleet with a completed .env),
`~/.copilot/hooks/scripts/pretool_guard.py` became a **cyclic symlink**. Every subsequent tool call
(Bash, Read, Edit, Write) failed at the PreToolUse hook with:
`python3: can't open file '.../pretool_guard.py': [Errno 40] Too many levels of symbolic links`
This blocks the PreToolUse hook for ALL tools → the operator is fully locked out and cannot
self-recover (the repair needs the very tools that are blocked).

## Likely cause
`install-hooks.sh` / the claude deploy uses `ln -sf <src> <dst>`. Re-running against an already
materialized tree appears to point `pretool_guard.py` at itself (non-idempotent `ln -sf`).

## Impact
- G6/Resilience: catastrophic, non-recoverable from inside the session.
- Only `pretool_guard.py` looped (sibling hook scripts stayed readable), so it is file-specific.

## Fix direction
- Make hook materialization idempotent: detect when `src` resolves to `dst` and skip; or copy the
  file instead of symlinking; or `rm -f` the dst before `ln`.
- Add a `hamr:doctor` check that fails fast if any deployed hook path is an unreadable/cyclic symlink.
- Consider a guard so a broken PreToolUse hook degrades to allow-with-warning rather than hard-block.

## Repro
Run `npm run hamr:activate` twice; inspect `readlink -f ~/.copilot/hooks/scripts/pretool_guard.py`.
