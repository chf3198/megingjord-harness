# #2901 — Wire fleet dispatch timeouts to timeout-policy.json + heartbeat

## Body

## Objective
Both fleet dispatch entry points still hardcode their request timeouts while `config/timeout-policy.json`
already carries the empirically-derived classes — wire them up so the values are single-source-tunable, and
add a dispatch heartbeat so slow Tailscale generations don't read as a hung terminal.

## Background
This change was authored during an IT-role fleet-maintenance phase and committed directly to canonical main
with an `[it-ops]` bypass — which waives ticket/baton ceremony but NOT the canonical-main-read-only policy
(#2107) or the worktree+PR flow. It was preserved on `chore/it-ops-fleet-timeout-wiring` and main was reset
(Tier-1 incident `it-ops-commit-stranded-on-canonical-main`). This ticket lands it correctly because it
modifies fleet EXECUTION code and warrants review, not an it-ops rubber-stamp.

## Acceptance criteria
- AC1: `fleet-red-team-dispatch.js` REQUEST_TIMEOUT_MS reads `classes['fleet-red-team-rate'].ms` (1200s),
  falling back to the prior 600s on a missing/malformed policy.
- AC2: `ollama-direct.js` TIMEOUT_MS reads `classes['fleet-dispatch-basic'].ms` (300s), falling back to the
  prior 120s.
- AC3: a 30s stderr heartbeat during a fleet generation (cleared on completion), bounded + non-throwing.
- AC4: the timeout loaders are exported + path-injectable; tdd-pyramid tests cover class-read + fallback.
