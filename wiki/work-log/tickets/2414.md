# #2414 — Enforce doc-coverage Tech-Writer sub-phase as hard-blocking governance gate

> **Source**: github:issue/2414 | **state: CLOSED** | **Labels**: type:epic, status:done, priority:P1, area:governance, resolution:released, lane:code-change, phase-gate:research-first

## Body

## Objective

The current doc-coverage/Tech-Writer sub-phase (Epic #2148) emits an advisory when a COLLABORATOR_HANDOFF is missing a `doc-coverage:` block, but does NOT block merge. The result: documentation surfaces drift silently whenever contributors omit the block.

This Epic enforces doc-coverage as a hard-blocking gate — a mandatory precondition for the Collaborator → Admin baton transition — with the same rigor as existing baton gates (admin-gate, consultant-gate, test-evidence).

## Why P1

Every shipped ticket without a doc-coverage block is a documentation debt accumulation event. At current velocity (several PRs per day), this compounds into significant doc drift that erodes harness maintainability (G10), observability (G8), and quality (G2).

## Acceptance Criteria (Epic-level)

- AC-R1: Research ticket #TBD completes — existing programmatic doc guardrail patterns surveyed and synthesized
- AC-R2: ACs and implementation child tickets created from research findings
- AC1: `doc-coverage-required` CI gate fails when COLLABORATOR_HANDOFF is missing a `doc-coverage:` block
- AC2: `doc-coverage-required` gate verifies each required surface (per area-label matrix) is listed in the block as DONE or N/A+reason
- AC3: Gate is fleet-compatible (advisory mode via env var for operators without CI access)
- AC4: Existing COLLABORATOR_HANDOFF artifact template updated to require the block
- AC5: Validator is promoted from advisory to blocking in `scripts/global/megalint/collaborator-handoff.js`
- AC6: `docs/workflow/learnings.md` and `.changes/unreleased/` updated

## Phase-0 Research Gate

Implementation child tickets (AC1–AC6) may NOT be authored until AC-R1 is closed and this Epic receives an EPIC_RESCOPE comment from Manager.
