## Pattern

PR #1550 included `git rm --cached node_modules` to un-track a self-referential symlink. The intent was repo-level only: stop tracking the symlink so future clones don't restore it. The actual side-effect on collaborator checkouts:

```
On every collaborator's working tree, when they `git pull`:
  1. Index drops the entry (intended).
  2. Working tree also drops the path — REGARDLESS of whether it
     is the same symlink or a different content type (real dir, etc.)
     that the operator placed there as a manual fix or local install.
  3. If a real `node_modules` dir was sitting there (e.g., from
     `npm install` or operator-applied filesystem repair), it gets
     force-removed.
```

Codex Team's pull of origin/main (to do unrelated #1551 work) faithfully applied #1550's untracking step on their main checkout. The real `node_modules` dir I had moved into main as part of the incident-1539 repair was force-removed. Cross-team work didn't cause harm directly — it propagated the latent side-effect of my PR exactly as git is designed to do.

## Why this is a governance gap (not just user error)

- The PR diff showed `delete mode 120000 node_modules` (symlink deletion mode). No reviewer or megalint rule flagged this as potentially destructive to collaborator working trees.
- The harness has 13 megalint rules for governance signals; **none of them detect cross-checkout destructive changes**.
- The PR body's "post-merge filesystem step" note buried the operator instruction inside changelog prose; it didn't surface as a structured warning anywhere a collaborator would see during pull/merge.

## Possible interventions (order of cost)

1. **Cheap**: PR template checklist line: "Does this PR un-track (`git rm --cached`) a previously-tracked path? If yes, document the working-tree impact on collaborator checkouts."
2. **Medium**: Megalint rule that parses PR-file-diffs for `delete mode \d+ <path>` entries on previously-tracked paths AND flags the PR body when no corresponding warning section exists.
3. **Medium**: A `governance:cross-checkout-destructive` label that the megalint rule auto-applies, plus a workflow comment summarizing the affected paths.
4. **Stronger**: A pre-merge gate that requires explicit Manager sign-off via a `cross-checkout-destructive:approved` label OR PR-body acknowledgement marker, similar to the `merge-evidence-override:approved` pattern from Epic #1486.

## Acceptance Criteria

- [x] AC1: A megalint or workflow check detects PRs that include `delete mode \d+ <path>` for a path that was previously tracked.
- [x] AC2: The check emits an advisory comment listing affected paths and the cross-checkout removal risk.
- [x] AC3: After a 1-week soak with low false-positive rate, the check promotes to a required gate that needs a `cross-checkout-destructive:approved` label or PR-body marker to merge.
- [x] AC4: PR template has a checkbox line covering this pattern.
- [x] AC5: `instructions/team-model-signing.instructions.md` OR a new `instructions/cross-checkout-safety.instructions.md` documents the rule + override path.
