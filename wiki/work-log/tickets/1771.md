# #1771 — Replace calendar soaks with replay-based eval gates

> **Source**: github:issue/1771 | **state: CLOSED** | **Labels**: type:epic, status:done, priority:P2, area:governance, resolution:released
> Mirror of `gh issue view 1771` (derived; edit the GitHub item, not this page).

## Body

## Epic Summary

Replace the harness's calendar-bound soak pattern (14-day advisory wait → promote-or-back-off) with **replay-based eval gates** that produce promotion-decision evidence in **hours, not weeks**. Per 2026 frontier research (eval-driven development; shadow testing on historical traffic; synthetic adversarial fixtures), calendar waiting is the lowest-signal validation layer; agentic infrastructure can compress it 99%.

## Problem

The harness currently uses 14-day soaks for new governance gates (#1727 rotation contract; #1756 pre-merge review). The 14-day wait is:

1. **Path-dependent** from human-throughput-limited SRE practice (canary releases waiting for traffic accumulation).
2. **Wasteful in an agentic context** — the reviewer is itself an agent; it doesn't need calendar time to accumulate signal.
3. **Lower-signal than replay** — calendar-only soaks sample organic traffic; replay covers historical PRs + synthetic adversarial fixtures + edge cases organic traffic won't surface.

Cited research (in the closeout summary on this issue): Braintrust EDD; arxiv 2411.13768 EDD-for-LLM-agents; TianPan 2026 shadow testing; arxiv 2605.08083 offline replay environment; Speedscale traffic replay; LangChain LLM evals; AppScale AI-Native CI/CD.

## Proposed contract

Replace single calendar-bound soak with a 5-layer eval stack:

| Layer | What | Wall-time |
|---|---|---|
| 1. Golden dataset | Curated input/expected-output fixtures | seconds-minutes |
| 2. Historical replay | Run candidate against last N PRs | minutes-hours |
| 3. Synthetic adversarial | Generate edge-case inputs deterministically | hours |
| 4. Shadow mode | Run alongside real traffic, no user impact | hours-days |
| 5. Canary | Small-% real exposure | days only when 1-4 inconclusive |

Promotion decision = eval-score-driven, not calendar-driven.

## Phase structure with concrete children

### Phase 1 — Light research (~1 child)

The literature is cited above; this phase captures the harness-specific application.

- **Child 1.1**: Research artifact codifying the replay-based eval pattern for the harness; survey existing test/eval surfaces.

### Phase 2 — Design (~2 children)

- **Child 2.1**: Design replay-runner helper spec — input format (closed-PR baton trail), output format (compliance metrics JSON), severity gate decision logic.
- **Child 2.2**: Design synthetic adversarial fixture catalog — naming convention, severity expectation, per-rule coverage.

### Phase 3 — Implementation (~2 children)

- **Child 3.1**: Implement replay-runner — `scripts/global/soak-replay-runner.js` consumes closed PRs and the rotation v2 helper; outputs compliance metrics. Tests.
- **Child 3.2**: Implement adversarial fixture generator + ship 20+ fixtures covering each rotation rule + pre-merge review trigger.

### Phase 4 — Apply (~2 children)

- **Child 4.1**: Apply to #1727 rotation soak — produce replay evidence; decide promote/extend/back-off; close #1727 with decision rationale.
- **Child 4.2**: Apply to #1756 pre-merge review soak — produce replay evidence; decide.

## Out of scope

- Replacing the harness's monitoring/observability infrastructure (HAMR + dashboards remain canonical for live observation).
- Building a NEW LLM provider — uses existing HAMR-routed providers and local execution.
- Mandating replay-only validation — calendar canary remains available for cases where replay is genuinely inconclusive (high-novelty operator-environments).

## Risk register

| Risk | Mitigation |
|---|---|
| Replay evidence over-fits to historical PR distribution | Adversarial fixtures compensate; synthetic edge cases force coverage |
| Replay-runner false-promotes a gate that fails on novel PR shapes | Eval-score thresholds conservative (compliance ≥85%, FP ≤20%, latency p95 budget); shadow-mode still available as Layer 4 |
| Operators expect 14-day soaks per current docs | Phase 4.1/4.2 explicitly close existing soaks with replay-based decisions; instruction update lands as part of Phase 2 |
| Per-Gemma-cross-review pattern doesn't compose with hour-scale promotion | Gemma reviews are cheap (~30s each); fits within hour budget |
