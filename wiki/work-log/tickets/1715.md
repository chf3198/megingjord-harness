## Body

## Symptom

In the 2026-05-16 session, the `closeout-preflight` hardcoded-lane bug (#1639) was discovered DURING the work and triggered `SKIP_CLOSEOUT_PREFLIGHT=1` bypasses on every push for 6+ tickets before #1639 itself shipped. Codex Team (gpt-5.5) correctly noted:

> `SKIP_CLOSEOUT_PREFLIGHT=1` was used repeatedly before the fix landed, which means early closure evidence relied on bypassing a governance gate during a governance Epic.

The bypass itself is sanctioned (env var is the documented escape hatch). The issue is **session ordering** — infrastructure bugs surfaced during pre-flight should be the FIRST work, not parallel work.

## Recurrence evidence

The pattern recurred on the first 6 PRs of the 2026-05-16 session before #1639 landed via PR #1689. **Affected PRs** (every one used `SKIP_CLOSEOUT_PREFLIGHT=1` to push past the hardcoded-lane bug): PR #1638 (#1628), PR #1645 (#1629), PR #1653 (#1630), PR #1661 (#1631), PR #1667 (#1632), PR #1669 (#1633). 6 PRs total. After #1689 landed the fix, no further bypass was needed.

## Pattern class

**Session-ordering anti-pattern**: when pre-flight surfaces a bug in a governance gate, the natural impulse is to set the bypass env var and continue. The correct move is to elevate the bug to first-work, then continue. Two memories pair here: `[[feedback-self-anneal-scope]]` (anneal is docs/process, not code — except infrastructure bugs ARE the exception) and `[[feedback-anneal-emission-during-implementation]]` (file during implementation, don't wait).
