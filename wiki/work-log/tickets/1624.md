Addresses Epic AC1 (websearch-backed research) with a narrower technology-focused scope: which GitHub-native CLI + API primitives can solve cross-team live coordination, planning, and past-work review for human + AI-agent collaborators.

## Problem framing

Current pain points across Claude Code / Copilot / Codex teams (per #1604 problem statement + observed friction this session):

1. Pre-edit conflict prediction is absent — teams don't see what other teams are about to touch
2. Real-time visibility is missing — no live "who is working on what" cross-team view
3. Live coordination is indirect — we use issue comments after work begins, not pre-edit signals
4. Hierarchy fragility — Epic-child relationships rely on prose "Refs Epic #N" matching, not native parent/child
5. Live-vs-archival conflation — actionable work and conversational decision-making live in the same Issues stream

## Findings: GitHub primitives inventory (current state + Feb 2026 previews)

### F1 — GitHub MCP server (official)
The Model Context Protocol server published by GitHub (https://github.com/github/github-mcp-server) exposes Issues, PRs, Discussions, Projects v2, Sub-issues, Actions, Repositories as MCP tools. Claude Code, Copilot CLI, and Codex CLI all support MCP natively. This is the **single most leveraged primitive**: any agent gets read+write access to every relevant GitHub surface through a standardized contract.

Implication: our cross-team protocols can be implemented as MCP-callable tools that every team consumes uniformly without per-team adapter code.

### F2 — GitHub Projects v2 + GraphQL API
Projects v2 is the central live tracker. Full GraphQL automation for: read items, write status, set custom-field values, list dependencies, drag across columns. The GraphQL schema is introspectable, which means AI agents can dynamically discover available fields without hardcoded payload shapes.

Implication: a Projects v2 board with custom fields like `claimed-by`, `locked-paths`, `in-flight-since`, `expected-completion` becomes the live state plane every team queries before starting work.

### F3 — Sub-issues + Issue Types (GA mid-2025)
Native parent/child issue relationships: up to 100 children per parent, 8 levels deep, 25 issue types per org. Tracked progress field on parent rolls up child completion automatically. REST + GraphQL API for both directions.

Implication: replace prose-level "Refs Epic #N" with native parent/child — eliminates the trap class my #1614 / role:NAME / Team&Model prose-collision memories document.

### F4 — Webhooks + repository_dispatch + workflow_dispatch
Sub-second event delivery on issue events (created, edited, labeled, commented, closed, reopened) and PR events. repository_dispatch event lets one team programmatically trigger workflows in another team's namespace. workflow_dispatch enables on-demand or scheduled invocation with arbitrary JSON payload.

Implication: real-time cross-team coordination becomes event-driven, not poll-driven. A "team A claimed path X" event fires within seconds to team B's preflight check.

### F5 — GitHub Discussions (Q&A + threaded)
Distinct from Issues: Discussions are conversational + persistent + community-pattern. Categories: Q&A (with answer marking), Ideas, Announcements, Polls, General. Full REST + GraphQL API. Discussion-to-Issue conversion is supported. Native @-mentions deliver notifications without spamming the Issue stream.

Implication: separate the *decisional* layer (Discussions, "should we adopt X?") from the *actionable* layer (Issues, "ship X by Y"). Today we conflate them.

### F6 — GitHub Agentic Workflows (technical preview, Feb 2026)
GitHub's gh-aw (https://github.com/github/gh-aw) is a CLI extension + Markdown DSL for workflows that execute via Copilot CLI / Claude Code / Codex. Workflows are authored in plain Markdown describing intent, not YAML describing steps. The agent figures out steps. Critical security primitive: the agent cannot write to GitHub directly — it produces a structured artifact, and a gated job applies only the explicitly-permitted subset.

Implication: our advisory workflow family (#1554, #1555, #1572, #1571, #1573, #1334-AC1, #1589, #1590, #1472 diagnostic) could migrate to gh-aw format for cross-tool consumption + a stronger security boundary than our current actions/github-script + .catch(() => {}) pattern.

### F7 — Codespaces + Live Share
Real-time pair-programming: up to 30 participants, shared terminals, shared debuggers, shared host machine. Free tier. Copilot integration works inside Codespaces. Doesn't natively coordinate multiple Codespaces talking to each other, but does enable a human + multiple agents on the SAME Codespace.

Implication: for high-bandwidth multi-actor work (incident response, complex refactor), spin up one Codespace and have all parties join Live Share. For our async multi-agent flow, less directly applicable.

### F8 — GraphQL introspection for AI-agent self-discovery
GraphQL's strongly-typed schema + built-in introspection gives AI agents a complete machine-readable map of the domain at runtime. No need for hardcoded REST payload shapes. Apollo / others have published patterns for AI agents that introspect and execute against arbitrary GraphQL schemas.

Implication: agents can reflect on available Projects v2 fields, Issue Types, Discussion categories etc. at session start, eliminating the documentation-drift class of bugs.

### F9 — Built-in Claude Code agent teams (mid-2025+)
Claude Code's agent-teams feature has shared task lists with statuses (pending, in_progress, completed, blocked), dependency tracking, file locking, and peer-to-peer messaging between teammates. Built-in coordination — agents claim tasks from a shared list, merge changes continuously, resolve conflicts automatically.

Implication: parts of what we're hand-building in scripts/global/cross-team-* may be obsoleted by this built-in capability. Worth a focused experiment vs. our hand-rolled cross-team-queue.js.

### F10 — Squad "drop-box decisions.md" pattern (2026)
A coordinated AI agent pattern documented by GitHub: every architectural choice gets appended as a structured block to a versioned decisions.md file in the repo. Not using Issues/Discussions/Projects — just markdown files. Trade-off: lower latency for read-side; higher latency for write-side conflicts.

Implication: our planning/ folder usage and feedback memory pattern is conceptually similar. Worth comparing rigor.

## Recommended adoption path (Epic #1604 implementation children)

**Phase 1 (lowest effort, highest leverage):**
1. Adopt **F1 GitHub MCP server** as the standard tool surface for all three teams. Every team's skill should consume MCP rather than direct REST/CLI. Single ticket: `mcp-cross-team-baseline-adoption`.

**Phase 2 (medium effort):**

5. Split **F5 Discussions for decisional vs Issues for actionable**. Process ticket + one-time migration of existing decisional content out of issue comments.
6. Pilot **F6 Agentic Workflows** for one existing advisory (start with `model-diversity-advisory` #1572). Compare gh-aw markdown to current github-script YAML. Single ticket: `agentic-workflow-migration-pilot`.

**Phase 3 (experimental):**
7. Compare **F9 Claude Code agent teams** to our hand-rolled cross-team-queue.js. Decide: replace vs augment.
8. Decide on **F10 decisions.md pattern** vs full Projects v2 reliance. Likely both: human-facing surface (Projects) + agent-facing surface (decisions.md or similar persistent markdown).
9. Test **F7 Codespaces + Live Share** for an incident-response scenario.
