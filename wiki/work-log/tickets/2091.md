## Body

## Self-anneal trigger

Tier-2 mid-flight pivot. Pattern observed throughout 2026-05-21/22 session: harness hooks fired ~20 false-positive "Admin baton incomplete — missing: merge" alarms while my work was actively shipping through dedicated worktrees with their own ticket lifecycles. The alarms were technically correct ABOUT the data they read — but the data itself was a CONFLATED MIX of three different teams' sessions sharing one state file. The harness alerted faithfully on polluted data.

This Epic addresses **the data pollution**, not the alarms.

## Goal (corrected from v1)

**v1 framing (rejected by operator review)**: "ownership-aware alert filter that suppresses noise about other teams' work" — this was symptom-suppression, the wrong fix.

**v2 framing (this Epic)**: ensure each session's state is **isolated by construction** so that no cross-team data exists for alarms to fire on inappropriately. When state is properly scoped, alarms become inherently relevant — there IS no other team's residue to filter against. The alert filter goes away entirely.

Industry consensus 2026 (cited in Phase-0 synthesis): "symptom suppression should not be used as a substitute for better monitoring or quick engineering fixes. Too many low-priority alerts train teams to ignore notifications, causing real issues to slip through." — Datadog RCA 2026.

## Root-cause fixes (no alert filtering)

1. **Per-session state file isolation** — state file name includes session ID (not just checkout path). When Claude Code starts a session, it gets a fresh state file; other teams' prior state files are not read.
2. **Per-worktree git hooks** — `core.hooksPath` + `extensions.worktreeConfig` (git ≥2.36 native). Hooks scope to the worktree they fire FROM, eliminating cwd-confusion when shells inherit cwd from a different team's worktree.
3. **Canonical-main read-only enforcement** — main checkout's TRACKED state is the canonical reference; no team edits tracked files there during a session. Pre-tool-guard policy is `.gitignore`-allowlist-driven: writes to **ignored paths** (`.env`, `.env.local`, `.envrc`, `.npmrc`, `node_modules/`, `dist/`, `*.cache/`, etc.) are PERMITTED — these are per-operator config or build artifacts; writes to **tracked paths** are REJECTED with redirect-to-worktree. Branch switches off `main` are REJECTED. Caveat: 2026 industry direction is moving secrets out of `.env` files toward workload-identity-fetched credentials ([Bitwarden Secrets Manager](https://bitwarden.com/blog/secure-ai-agent-access-with-secrets-manager/), [Infisical](https://infisical.com/blog/your-ai-coding-agent-is-reading-your-env-file), [Zylos Secretless AI Agents](https://zylos.ai/research/2026-05-09-secretless-ai-agents-workload-identity)); the allowlist accommodates today's `.env` reality but should narrow as secrets migration progresses.
4. **Ephemeral session boundary** — session-end archives the state file with session ID; session-start creates fresh. No residue accumulates.

The "ownership-aware alert filter" + "audit-emit suppressed signals" from v1 are **removed** — they're unnecessary once the 4 root-cause fixes are in place.

## Branch vs worktree vs sandbox (Phase-0 deliverable: full comparison)

Operator question from 2026-05-22: "I need to know the difference between each team having its own branch vs. worktree vs. sandbox."

Phase-0 synthesis will produce a clear comparison table. Preview (full detail in Phase-0 child):

| Layer | What's isolated | What's NOT isolated | Cost |
|---|---|---|---|
| **Branch** | Git history line | Files on disk (same checkout), state, runtime, ports | Free (git native) |
| **Worktree** | Files on disk + git history | Runtime, ports, env vars, package caches, test state, OS processes | Cheap (shared object store; ~no disk cost) |
| **Sandbox** (container/microVM) | Files + runtime + ports + env + processes + network | Truly isolated (modulo shared host kernel for containers) | Expensive (per-sandbox memory/disk; longer startup) |

Per 2026 industry sources (Towards Data Science, agyn.io, Augment Code): worktrees are sufficient for sequential lightweight tasks on a controlled host. For concurrent multi-agent work that touches real services (ports, databases, deploys), sandboxes become necessary.

Megingjord today is sequential per-team (one session per runtime at a time) and lightweight (no deploy collisions). Worktrees + per-session state are sufficient. Sandbox is overkill for now but documented as a future option.
