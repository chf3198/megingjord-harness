## Problem

Repeated pattern across multiple teams in the same 7-day window — Tier-2 trigger per `instructions/workflow-resilience.instructions.md` (recurrence ≥ 2, severity = medium):

- 2026-05-11 / 2026-05-12: Claude Code Team used \"Cole Mason\" instead of registry-derived \"Orla Mason\" for `claude-code:opus@anthropic` role=manager (logged in [feedback_signer_alias_derivation.md](/home/curtisfranks/.claude/projects/-home-curtisfranks-devenv-ops/memory/feedback_signer_alias_derivation.md)).

Today's enforcement is **post-hoc** — megalint scans comments and posts an advisory after the violation lands. By then the artifact is already in the baton trail and downstream gates may have already consumed it.

## Goal

Catch signer drift at write-time, not after-the-fact. Either:
(a) a comment-template tool every team uses, or
(b) a pre-comment hook / GitHub Action that validates `Signed-by:` against `inventory/team-model-signatures.json` and either auto-corrects or fails fast.

## Acceptance Criteria

- [ ] AC1: Single source-of-truth helper (`scripts/global/baton-comment-build.js`?) that emits a fully-signed baton comment from `--ticket N --role <r>`, deriving signer from registry + active `Team&Model` env/config
- [ ] AC2: Existing megalint `signer-fidelity` check stays as belt-and-braces backstop
- [ ] AC3: Add a check in `closeout-schema` workflow that fails when `Signed-by:` line in any baton artifact doesn't equal the registry-derived alias — promote from advisory to required
- [ ] AC4: Update `instructions/team-model-signing.instructions.md` to reference the helper as the recommended path
- [ ] AC5: Test coverage: unit test the helper for all 3 teams × 4 roles = 12 combinations
