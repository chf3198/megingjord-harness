# #1736 — Substantive cross-family pre-merge code review baton step

> **Source**: github:issue/1736 | **state: CLOSED** | **Labels**: type:epic, status:done, priority:P2, area:governance, resolution:released

## Body

## Epic Summary

Add a substantive **cross-family code-review baton step** between `COLLABORATOR_HANDOFF` and `ADMIN_HANDOFF`. The reviewer is a different-`Team&Model`-family agent that reads the actual diff (not just the baton text), runs specialized multi-agent checks (bug detection, security, test coverage, architectural drift), and emits structured findings. Composes with Epic #1716 (rotation contract) and supersedes the current operational-only Admin role for code-change lanes.

## Problem (the gap)

Tracing the current baton:

1. **Manager**: scopes the work; emits `MANAGER_HANDOFF`.
2. **Collaborator**: implements + writes tests; emits `COLLABORATOR_HANDOFF`.
3. **Admin**: per `feature-completion-governance.instructions.md`, role is *operational* — version collision check, push, CI green verification, **merge**, release, sync verification. **No substantive code review of the diff before merge.**
4. **Consultant**: per `role-baton-orchestrator`, "post-execution critique and risk scoring." Under the race-aware-sequencing pattern (all 4 baton artifacts before PR opens), the Consultant scores baton-text descriptions, not the actual diff.

**Result**: the diff goes from Collaborator → CI structural gates → Admin merge → Consultant post-merge rubric. **No role reads the diff and gates merge on substantive findings.**

Cited 2026 evidence (link in research children below):

- Agent-generated code introduces 1.7× more issues than human code (Jan 2026 study).
- Multi-agent specialized review architecture wins benchmarks (Qodo 2.0: 60.1% F1 vs single-pass).
- Reviewers feel better approving agent code despite worse quality — trust gap.

## Composition with #1716

Epic #1716 (rotation contract, soak active 2026-05-16 → 2026-05-30) enforces *who* can review — different family per role. This Epic adds *what* gets reviewed — the actual diff, by specialized sub-agents. Both together = WHO + WHAT + WHEN (pre-merge).

## Proposed contract (subject to research)

A new baton step `COLLABORATOR_REVIEW` (or `PRE_MERGE_REVIEW`; name TBD in design phase) between `COLLABORATOR_HANDOFF` and `ADMIN_HANDOFF`:

- Reviewer's `Team&Model` family ≠ Collaborator family (Rule 2 via #1716 contract; this Epic adds the substantive review action).
- Reviewer runs 4 specialized sub-agents in parallel: **bug detection**, **security**, **test coverage**, **architectural drift**.
- Sub-agents emit **structured findings** (severity, category, file:line, suggestion).
- Severity gates:
  - `low` → annotate PR; no block
  - `medium` → block but waivable with rationale
  - `high` → block, require approving signature from a 5th-family agent OR human escalation
- Auto-escalate triggers (mandatory `high` severity regardless of agent confidence):
  - Touching auth/authn/authz code
  - Database schema migrations
  - New external dependencies (package.json, requirements.txt, go.mod)
  - Secret/credential paths
  - Workflow YAML changes (security surface)

## Research questions (Phase 1 children)

1. **Multi-agent architecture**: 4 parallel specialized sub-agents (Qodo 2.0 pattern) vs 1 broad agent with structured-output prompts. Trade-offs in cost, latency, F1.
2. **Auto-escalate trigger matrix**: which file patterns, diff patterns, and AC labels deterministically raise severity to `high`? How to integrate with existing `area:*` labels?
3. **Structured-finding format**: SARIF (industry standard) vs custom JSON Lines vs GitHub Review API. Which is most queryable and cross-runtime compatible?
4. **Integration with #1716 rotation**: the reviewer's `Team&Model` family is enforced different-family by Rule 2 of the rotation contract. Does the rotation validator need extension for the new step, or does it work out-of-the-box?

## Risk register

| Risk | Mitigation |
|---|---|
| False positive rate burns operator trust | Path D advisory-first 14d; structured-finding format with confidence scores; waiver mechanism for medium severity |
| Multi-agent cost blowup (4 sub-agents per PR) | HAMR cost-routing; sub-agents share context window; cache per-PR base |
| Reviewer-agent could itself need review (turtles all the way down) | Reviewer findings are structured + auditable; rotation contract (#1716) ensures different family from Collaborator |
| Existing CI pipelines (#1572 model-diversity, baton-gates, evidence-completeness) collision | Phase 2.1 contract explicitly compatible; new step adds, doesn't replace |
| Auto-escalate triggers over-fire on trivial diffs | Trigger matrix tuned with whitelist patterns; severity-medium-waivable pathway |
