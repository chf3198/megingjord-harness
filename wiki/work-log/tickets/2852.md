# #2852 — label-lint reopens children closed via Closes #N while status:in-progress

> **Source**: github:issue/2852 | **state: OPEN** | **Labels**: type:bug, status:backlog, priority:P2, area:infra, lane:code-change
> Mirror of `gh issue view 2852` (derived; edit the GitHub item, not this page).

## Body

## Problem

A child PR that closes its ticket via `Closes #N` auto-closes the issue on merge **while the issue
still carries `status:in-progress`** (the merge path does not flip the status label). The `label-lint`
workflow then runs on the close event, sees a closed issue at `status:in-progress`, posts
"❌ Close blocked: add `status:done` or `status:cancelled` before closing", and **reopens the issue**
~9 seconds later. A stale `governance:close-without-merge` advisory also attaches even though the PR
merged. This forces manual rework on **every** child ship: relabel → re-close → remove stale advisory.

## Evidence (recurrence ≥2 in one session — Tier-2)

- #2844 (PR #2845 merged `84233872`): closed by chf3198 `2026-06-10T03:07:39Z`, **reopened by
  github-actions[bot] `2026-06-10T03:07:48Z`** (+9s).
- #2847 (PR #2849 merged `b5fa5637`): closed `2026-06-10T03:42:01Z`, **reopened by
  github-actions[bot] `2026-06-10T03:42:10Z`** (+9s).

Both required: flip `status:in-progress`→`status:done`, remove `role:*`, re-close, remove stale
`governance:close-without-merge`.

## Root cause (hypothesis)

The merge-driven auto-close (`Closes #N`) and the baton status-transition are not coupled: nothing
flips `status:in-progress`→`status:done` on the Closes-merge path, and `label-lint`'s close-guard
fires on the resulting close, reopening before any terminal-state advancement occurs. ADR-010 correctly
requires a terminal `status:*` on a closed issue — the gap is that no automation supplies it on this
path. Classification: **missing guardrail / ordering gap**. Recurrence risk: **high** (hits every
`Closes #N` child PR merged while in-progress).

## Proposed fix (route through normal baton; options for scoping)

1. **Auto-transition on Closes-merge** (preferred): on PR merge, resolve each `Closes #N`, flip
   `status:in-progress`→`status:done` and strip `role:*` atomically so `label-lint` sees a valid
   terminal state (no reopen). Also suppress `governance:close-without-merge` when a merged PR closed it.
2. **label-lint grace**: if the closed issue has a linked merged PR, auto-apply `status:done` instead
   of reopening.
3. **Process-only fallback** (weakest): codify "flip `status:done` at Admin terminal-finalize before
   merge" — already captured in operator memory `feedback-flip-status-done-before-closes-merge`; does
   not structurally prevent recurrence.

## Acceptance criteria

- AC1: a child PR merged with `Closes #N` while `#N` is `status:in-progress` ends CLOSED + `status:done`
  with **no** `label-lint` reopen.
- AC2: no stale `governance:close-without-merge` attaches to a genuinely-merged close.
- AC3: golden-file/workflow test asserting the close→terminal transition (no reopen) for the Closes-merge path.
