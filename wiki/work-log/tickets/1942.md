## Body

## Epic intent (research-first per Epic #1399 phase-gate rule)

Operationalize three distinct **Karpathy LLM Wiki types** as a unified knowledge substrate that increases harness efficiency toward **G3 Zero-Cost** (reduce expensive CLI/code-search per model query) and **G7 Throughput** (faster answers to model prompts about project state, history, and architecture).

This Epic composes existing closed wiki infrastructure (Epic #142, #403, #864, #866, #1017, #1083, #1485, #1625, #1626, #1673, #1857) and existing open research (#1858) into a coherent three-Wiki typology with a programmatic auto-update pipeline.

## The three Wiki types (per operator vision 2026-05-19)

### Wiki Type A: Code-Base Wiki (per-project)

**Purpose**: holds all details of the code base so models answer questions about features, dependencies, known issues, architecture without expensive `grep`/`rg`/`find` CLI research.

**Audience**: models (primarily), humans (secondarily).

**Analogue**: developer's guide + set of historical release docs.

**Scope per project**: one Code-Base Wiki per repo (Megingjord-harness, future repos).

**Failure mode addressed**: "models often need to search the code base via lengthy CLI research to find dispersed references" — this directly costs token spend (G3 violation) and response latency (G7 violation).

### Wiki Type B: Project Work-Log Wiki (per-project)

**Purpose**: local store/log of all work done; mirrors GitHub ticket state so models have a canonical record independent of GitHub API availability or rate-limit cost.

**Audience**: models reasoning about historical decisions, anneal lineage, completed/abandoned features.

**Scope per project**: one Work-Log Wiki per repo.

**Failure mode addressed**: `gh issue view` / `gh pr view` round-trips on every query consume rate budget and tokens; outage-time queries fail entirely; cross-Epic synthesis requires N round-trips per Epic.

### Wiki Type C: Research-Based-Wisdom Wiki (dual-scope)

**Purpose**: research synthesis (the original Karpathy LLM Wiki intent — "local collection of research-based wisdom").

**Dual scope**:
- **Global Wisdom Wiki**: general knowledge helpful to ALL projects (cross-project applicable). Existing `wiki/concepts/`, `wiki/entities/`, `wiki/syntheses/` are this scope.
- **Project-specific Wisdom Wiki**: knowledge deemed specific to a particular project (lives in repo, not shared cross-project).

**Audience**: models + humans, both consumers and contributors.

**Failure mode addressed**: research findings buried in `research/*.md` files are not retrievable via the Karpathy 3-layer pattern; cross-project knowledge fragmentation.

## Operator vision quote (2026-05-19, for posterity)

> "The original Karpathy LLM Wiki of the harness was an attempt to build a local collection of research-based wisdom. The third consideration for another Wiki was to provide a local store (log) of all work done — mirroring that found in the GitHub tickets. Lastly, while the code-base Wiki and the project work-log Wiki are obviously specific to individual projects, the research-based-wisdom Wiki would need both a globally available Wiki with general knowledge helpful to all projects, and a project specific research-based-wisdom Wiki when knowledge is deemed specific to a particular project."
