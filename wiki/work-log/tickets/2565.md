# #2565 — Verify prompt activation after file-based MCP registration

> **Source**: github:issue/2565 | **state: OPEN** | **Labels**: type:task, status:backlog, priority:P2, area:scripts, phase-gate:phase-1

## Body

## Problem
Phase-0 research and the first red-team pass identified a gap between config-file registration and live prompt activation. A runtime can contain the correct MCP entry yet still fail to expose `/xteam`-family prompts.

## Objective
Verify that prompt-type MCP servers activated through file-based registration behave the same as expected runtime-side prompt registration, and document any runtime-specific gaps.

## Refs
Refs #2553 (Epic), Refs #2554 (Phase-0 research), Refs #2559 (smoke validation)
phase_0_sources: [#2554]

## Acceptance Criteria
- [ ] Define the expected prompt-activation behavior for `megingjord-xteam` in each active runtime
- [ ] Verify whether file-based MCP registration is sufficient to expose `/xteam`, `/xteam-create`, and `/xteam-status` in each runtime
- [ ] Distinguish config-presence checks from prompt-visibility checks in the validation notes
- [ ] Document any runtime where prompt activation requires an extra reload, command, or unsupported capability
- [ ] If a runtime registers cleanly but does not surface prompts, file or link a defect with reproduction steps
- [ ] Feed the result back into C5 #2559 so smoke validation reflects the proven behavior rather than assumption

## Notes
This is a research/validation follow-on ticket. It does not expand server behavior; it closes the remaining planning uncertainty before we treat prompt activation as solved.
