# #2564 — baton-artifact-governance entries() over-matches artifact tokens in prose

> **Source**: github:issue/2564 | **state: CLOSED** | **Labels**: type:bug, status:done, priority:P2, area:scripts, lane:code-change, governance:close-without-merge
> Mirror of `gh issue view 2564` (derived; edit the GitHub item, not this page).

## Body

severity: MED — false `artifact-role-mismatch` failures whenever a baton artifact mentions another artifact's name in prose.

## Root cause
`scripts/global/baton-artifact-governance.js` `entries()` classifies a comment as a given artifact via `body.includes(artifact)` — a bare substring match anywhere in the body. So a MANAGER_HANDOFF or CONSULTANT_CLOSEOUT that merely *mentions* `COLLABORATOR_HANDOFF` in prose is misclassified as a COLLABORATOR_HANDOFF artifact, and its Role (manager/consultant) then fails the collaborator role check → `artifact-role-mismatch`.

This is the same naive-substring over-match class as #2562 (collaborator-gate). It is a documented operator trap (memory `feedback_prose_collision_non_baton_comments`, `feedback_baton_governance_iterates_all_comments`) that currently forces operators to avoid artifact tokens in prose.

## Reproduction
Hit live on #2562: MANAGER_HANDOFF + CONSULTANT_CLOSEOUT mentioned `COLLABORATOR_HANDOFF` in prose → admin-gate failed `artifact-role-mismatch ×2`. Worked around by editing the prose (source-edit-first).

## Fix
Detect artifacts by HEADER (line-anchored `^\s*(?:\*\*|##\s+)?<ARTIFACT>\b`, mirroring the per-validator finders) rather than `body.includes()`. Add regression tests: an artifact that mentions a sibling token in prose must NOT be classified as that sibling.
