# #2573 — Add strict wait-for-green loop before merge attempts

> **Source**: github:issue/2573 | **state: CLOSED** | **Labels**: type:task, status:done, priority:P1, area:governance, resolution:released, lane:code-change, governance:close-without-merge, anneal:tier-2
> Mirror of `gh issue view 2573` (derived; edit the GitHub item, not this page).

## Body

## Problem
A recurring execution-pattern flaw occurred while finishing #2560: checks were still pending, but additional merge/CI interventions were attempted instead of staying in a stable wait-and-merge loop.

## Evidence
- PR #2572 had periods of pending required checks while additional actions were taken.
- User explicitly flagged recurrence: "You should have patiently waited for all checks to finish... This deserves a self-anneal ticket."
- Final state eventually succeeded, but with avoidable churn and extra CI turns.

## Root Cause Hypothesis
The operator flow lacks a strict "all-required-checks-pending -> wait-only" guard, so intervention opportunities are taken too early when no actionable failure exists.

## Parent Epic
- #2574

## Acceptance Criteria
- [x] AC1: Add a deterministic wait policy for PR completion flow:
  - If required checks are pending and none are failing, do **not** modify branch/PR state.
  - Only re-evaluate status at bounded intervals using status-check read operations.
- [x] AC2: Add a merge-attempt gate:
  - Merge is attempted only when required checks are green.
  - If green + merge blocked by policy, use one explicit escalation path with reason capture.
- [x] AC3: Add regression coverage (script/unit/integration) for:
  - pending-only state,
  - failing-check state,
  - green-but-policy-blocked state.
- [x] AC4: Update operator/governance guidance to codify this wait discipline.
- [x] AC5: Capture mid-flight flaw accounting in closeout (`mid_flight_flaws`) when a wait-discipline breach occurs.

## Proposed Fix Plan
1. Introduce a helper in admin/merge flow to classify check states as `pending-only`, `failing`, `green`.
2. Enforce state-machine transitions so `pending-only` can only lead to `wait` actions.
3. Add tests and docs updates for the behavior.

## Scope
Workflow/governance behavior only. No product/runtime feature change.
