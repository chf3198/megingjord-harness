# #2251 — Add stale worktree cleanup planner

## Body

Parent Epic: #2248
Research basis: #2249
Prerequisite: **#2552** (squash-merge-aware merge detection — see AC5; implementation
can proceed but `remove` accuracy requires #2552)

## Scope

Update `scripts/global/worktree-cleanup-plan.js` (currently 78 lines) to
implement the safety-first five-state taxonomy from #2249 research. The existing
`classify()` function uses internal state names (`merged-clean`, `preserve-dirty`,
`stale-open-pr`, etc.); this ticket replaces those with five external taxonomy
states (`remove`, `prune-metadata`, `quarantine`, `preserve`, `needs-review`) and
adds `reason` fields to every plan entry. The script already exists — this is a
taxonomy upgrade, safety-hardening, and test addition.

## Acceptance Criteria

- [ ] **AC1 — Taxonomy**: `classify(entry, lease)` emits exactly five external
  states: `remove`, `prune-metadata`, `quarantine`, `preserve`, `needs-review`.
  All five states appear in both code paths and test cases.

- [ ] **AC2 — Safety invariant**: `remove` is emitted ONLY when ALL of the
  following hold (using enriched entry fields from `worktree-inventory.js`):
  `dirtyCount === 0`, `untrackedCount === 0`, `mainAhead === 0`,
  `mergedToMain === true`, `locked === false`, `lease === null` (no active lease
  from `leaseFor()`), and branch does not start with `sandbox/`. Every entry
  failing any criterion MUST produce `quarantine`, `preserve`, or `needs-review`.

- [ ] **AC3 — Quarantine/preserve cases**: All of the following conditions MUST
  produce `quarantine` or `preserve` (never `remove`): `dirtyCount > 0`,
  `untrackedCount > 0`, `mainAhead > 0`, detached HEAD, `locked === true`,
  `sandbox/*` branch, ticket-less non-sandbox branch, unknown state.

- [ ] **AC4 — Output schema with reason**: `plan()` returns
  `{ generatedAt: ISO8601, mode: "plan-only", worktrees: [{ ...entry,
  cleanupState, reason: string, commands: string[] }] }`. The `reason` field is
  non-empty for every entry (e.g., `"dirty: 2 modified files"`,
  `"merged: confirmed ancestor of origin/main"`,
  `"unverified-merge-status: pending #2552"`).

- [ ] **AC5 — Ambiguous merge state**: Entries where `mergedToMain === false`
  AND `mainAhead > 0` cannot be confirmed merged (squash-merge blind spot —
  see #2552). These MUST emit `cleanupState: "needs-review"` with
  `reason: "unverified-merge-status: may be squash-merged (pending #2552)"`.
  A code comment in `classify()` documents the gap.

- [ ] **AC6 — Dry-run flag**: `run()` CLI adds `--dry-run` as an explicit alias
  for the existing default plan-only mode; output always includes
  `mode: "plan-only"` regardless.

- [ ] **AC7 — Backward compatibility**: `--json` and `--workspace` flags work
  unchanged. `npm run governance:worktrees` produces valid JSON output. Exported
  API (`classify`, `commands`, `plan`, `ticketFrom`, `workspace`) is preserved.

- [ ] **AC8 — No duplicate logic**: `classify()` consumes the enriched entry
  shape from `worktree-inventory.js`. No new git command invocations in
  `worktree-cleanup-plan.js`.

- [ ] **AC9 — Safety tests**: Tests call `classify(entry, lease)` directly with
  synthetic plain-object entries (no git I/O, no import of worktree-inventory).
  Each risky combination (dirty / unpushed / detached / locked / sandbox /
  ticket-less / ambiguous-merge) asserts `cleanupState !== "remove"`. All five
  taxonomy states appear in at least one test case.

- [ ] **AC10 — File size**: `worktree-cleanup-plan.js` stays ≤ 100 lines after
  changes. If needed, extract a `worktree-cleanup-commands.js` helper (also
  ≤ 100 lines each).

## Constraints

- Planner only; no destructive execution at any point.
- No new git command invocations inside the planner script.
- Branch: `feat/2251-stale-worktree-cleanup-planner`.

## Blocking Dependencies

**#2552 required for full `remove` accuracy.** Until #2552 ships, branches
squash-merged into main show `mergedToMain === false` AND `mainAhead > 0` — the
planner cannot distinguish these from genuinely unmerged branches. AC5 mandates
`needs-review` for this case, which is the safe default.
