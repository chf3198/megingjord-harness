### Added
- Fleet-dev capability packaged as a discrete, opt-in, tier-3-gated HAMR module (#2806, Epic #2791 P1-8 capstone; design D16). `config/fleet-dev-module.json` declares the module boundary (requiredTier 3, the opt-in `MEGINGJORD_FLEET_DEV_ENABLED` env, and the member-file list for #2794–#2800); `scripts/global/fleet-dev-module.js`'s `fleetDevAvailable()` offers the fleet-dev lane only when the module is installed AND enabled AND the operator has explicitly asserted `MEGINGJORD_MINIMUM_TIER >= 3` (an unset tier is treated as fleetless and never offered the lane). Fail-closed on any uncertainty; an empty/unset env can never enable-by-default; manifest reads are own-property-only. A fleetless (Tier 0–2) operator carries zero fleet-dev code and the baseline router (`model-routing-engine.js`, which has no fleet-dev imports) is byte-for-byte unchanged when the module is off. Every member script declares `// tier: 3`, enforced by a manifest↔tag contract test. Cross-family reviewed (gemini-2.5-pro, ACCEPT 10/10 after a 2-round loop that caught an enable-by-default and a prototype-pollution gap).
