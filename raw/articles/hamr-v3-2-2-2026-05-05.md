# HAMR v3.2.2 patch — R9.2 cwd-vs-branch hook enforcement

## 0. Document role

Patch document amending v3.2.1 §R9.2 (#907) with three sub-patterns
that turn cwd-vs-branch pre-flight from a deploy-script convention
into a harness-level guard. v3.2 (#890) + v3.2.1 (#907) + this
patch are the combined input contract for Wave 4.

## 1. Empirical motivation

The cwd-vs-branch hazard caught HAMR PR cycles three times across
Waves 1–3:

| Occurrence | PR | Symptom | Mitigation used |
|---|---|---|---|
| Wave 1 #895 | #899 → #900 | `gh pr create` resolved head to stale `feat/894` | Close + reopen; explicit `--head` after the fact |
| Wave 2 #910/#912 | #913–#916 | External branch-switching dropped commits onto wrong branch | Cherry-pick + force-with-lease; explicit `--head` |
| Wave 3 #918 | #920 → #921 | Commit landed on Copilot's `feat/919-worktree-staleness` | Cherry-pick onto clean main; force-push correct branch |

Each occurrence cost ~5–10 min of recovery and risked dropping
work onto another team's branch (a Copilot Team boundary
violation). The defense currently lives in `hamr-deploy.sh` only
(`R9.2 pre-flight: branch=<name>`), which does not protect every
`gh pr create` / `git push` invocation.

## 2. §R9.2.1 (NEW) — Bash-hook contract

**Pattern:** every `git`-bearing command run within Claude Code,
Copilot, or Codex sessions runs through a Bash hook that asserts:

1. `git branch --show-current` is non-empty (HEAD not detached).
2. No in-progress rebase, merge, cherry-pick, or bisect (checked
   via presence of `.git/rebase-*`, `.git/MERGE_HEAD`,
   `.git/CHERRY_PICK_HEAD`, `.git/BISECT_LOG`).
3. If the caller passes `EXPECTED_BRANCH=<ref>` env var, current
   branch matches that ref exactly.

**Implementation home (deferred to development child ticket):**
either a Claude Code `PreToolUse(Bash)` hook in
`~/.claude/hooks/scripts/branch-preflight.py` OR a global Git hook
under `.git/hooks/pre-receive`. The contract specifies behavior;
the implementation is operator-choice.

**Failure mode:** hook prints `R9.2 pre-flight FAILED` + the
specific assertion that failed + an actionable remediation
("checkout a non-detached branch", "abort in-progress rebase",
"branch=<actual> ≠ EXPECTED_BRANCH=<expected>"). Exit code 2.

## 3. §R9.2.2 (NEW) — `gh pr create` MUST use explicit `--head`

**Pattern:** any `gh pr create` invocation in scripts, deploys, or
operator-CLI flows MUST pass an explicit `--head <ref>` flag.
Cwd-resolved head is BANNED.

**Why:** the runtime cannot rely on local cwd → branch resolution
when external tooling (IDE auto-checkout, parallel-team rebase,
agent worktree switches) can reposition HEAD between Bash calls.

**Enforcement:** lint check on shell scripts + workflow files
that grep for `gh pr create` and assert the `--head` flag is
present in the same logical command. Defer implementation to
development child ticket.

**Note:** the same principle applies to `gh pr merge` (always pass
the PR number explicitly; never rely on cwd-inferred PR) and
`git push` (always pass the remote + branch ref explicitly when
the script is one-shot).

## 4. §R9.2.3 (NEW) — branch-ops audit log

**Pattern:** every cross-branch operation (checkout, branch
create/delete, rebase, merge, cherry-pick, push, pull) emits one
line to `~/.megingjord/branch-ops.log` in this format:

```
<iso-8601-utc> <pid> <op> <from-ref> -> <to-ref> [extras]
```

**Why:** post-incident reconstruction. When a commit lands on the
wrong branch, the audit log shows the last 10 cross-branch
operations and which agent/session triggered them.

**Implementation home (deferred):** a `git` wrapper or post-checkout
hook. Append-only; rotation handled by a daily script.

**Storage cost:** ~1 KB per operation; with ~50 ops/day, log
reaches 10 MB after a year. Rotate weekly.

## 5. Wave 4 prerequisites (post-patch)

After this patch merges, Wave 4 kickoff has the same five
prerequisites as Wave 3 plus:

5. ✅ R9.2.1–R9.2.3 patterns documented as required architectural

Wave 4 children (3, 7, 9) do not directly implement R9.2.1–
R9.2.3 — those are tracked as a separate development child
ticket (e.g., "implement R9.2 hooks") spawned post-Wave-4 if
needed. The patch ensures the *requirement* is recorded so future
work can plan against it.
