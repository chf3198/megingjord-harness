# Epic #1271 Codex R&D Plan: Epic-State Truthfulness

## Summary

Codex Team recommends treating Epic truthfulness as an evidence-reconciliation
problem, not a wording problem. An Epic should close only when its checkbox
state, child graph, time-window obligations, dependency graph, and independent
Consultant verification all agree.

## Web Evidence Digest

- GitHub tasklists auto-track referenced issue completion, proving checkbox
  state can be machine-observable instead of decorative.
- GitHub issue dependencies and API endpoints provide native blocked/blocking
  relationships that can seed cross-Epic dependency detection.
- Linear models blocked/blocking issue and project relationships as first-class
  planning properties with violated-dependency filters.
- Jira workflow validators and post functions show that transition gates and
  state-mutating actions should be separate mechanisms.
- Google SRE guidance recommends documented time windows, reviewers, approvers,
  next review dates, and dashboards for objective state decisions.
- DORA 2025 frames AI adoption as a systems problem and warns that local
  productivity must translate into measured flow, not downstream chaos.
- 2025 issue-tracking research stresses traceability across linked artifacts,
  workflows, stakeholders, and context-rich issue ecosystems.
- 2026 oversight research supports a solve-verify split: AI can prepare
  evidence bundles, while humans or independent agents verify against external
  criteria.
- 2026 agentic code-review research finds human reviewers still contribute
  testing, understanding, and contextual feedback that AI agents lack.
- 2025 AI-augmented CI/CD research supports policy-as-code guardrails and trust
  tiers for autonomous delivery decisions.
- 2026 software artifact analytics research supports traceability graphs for
  detecting issue-tracking process smells.
- 2025 GitHub dependency graph accuracy research warns that dependency graph
  tooling should be reconciled and validated before governance decisions rely
  on it.
- Microsoft Research's 2026 intent-formalization paper argues that informal
  intent must become checkable specifications to keep AI work reliable.

Relevant existing pages: [[epic-governance]], [[baton-protocol]],

state, dependency APIs, and independent verification.

## Recommended Mechanisms

1. `epic-ac-reconcile.js`: parse Epic AC checkboxes, evidence references, child
   states, and closeout comments; emit a deterministic truth table.
2. `epic-close-readiness.yml` v2: block closure when narrative says complete
   but unchecked ACs remain without `EPIC_RESCOPE`.
3. `EPIC_RESCOPE` template and gate: every unchecked or changed AC needs old
   text, new text, rationale, approver, and date.
4. `status:measuring`: explicit state for time-windowed ACs with
   `Recheck-after: YYYY-MM-DD` and sensor reminders.
5. Cross-Epic dependency registry: parse text plus native GitHub dependency API,
   detect cycles, and publish blocking chains.
6. Consultant-only Epic verification: Manager may scope and report progress;
   Consultant signs final close readiness.
7. GHS sensor: add `declared_complete_unmet_ac` and `measuring_overdue`
   signals to the self-annealing governance health layer.
